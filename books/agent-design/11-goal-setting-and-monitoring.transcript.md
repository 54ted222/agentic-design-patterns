這一章我們要談的,是怎麼讓 AI 代理不只是「會做事」,而是「知道自己要去哪裡,而且知道自己到了沒有」。這個主題叫做目標設定與監控,英文是 Goal Setting and Monitoring。

我們先把問題講清楚。一個 AI 代理,英文叫 AI agent,意思就是一個能自己感知環境、做決定、採取行動的人工智慧程式。要讓這樣的代理真正有用、真正帶著目的性去工作,光會處理資訊、光會使用工具是不夠的。它還需要兩樣東西:第一,是清楚的方向感,也就是它知道自己到底要達成什麼;第二,是一種能知道自己有沒有在前進的方法,也就是它能追蹤進度、能判斷目標到底達成了沒有。這一整套做法,就是目標設定與監控模式的核心。簡單講,它做兩件事:一是給代理一個具體要努力達成的目標,二是給代理一套能追蹤進度、判定成敗的手段。

那這一章我會怎麼帶你走呢?我會先用一個生活化的比喻,帶出「規劃」這個底層概念;接著講在 AI 代理的世界裡,規劃跟監控具體是什麼樣子;然後我會列出六個真實世界的應用案例,讓你感受這個模式的用途;再來會帶你走過一個完整的程式範例,我會用講的方式描述它在做什麼,你閉著眼睛也能在腦中想像它的運作;最後我會把重點整理收束起來。

我們先從規劃說起。你想像一下規劃一趟旅行。你不會憑空就出現在目的地,對吧?你會先決定你想去哪裡,這就是「目標狀態」;然後你會弄清楚你現在人在哪裡,這是「初始狀態」;接著你會考量手上有哪些選項,像是交通方式、路線、預算;最後你才會排出一連串步驟:訂票、打包行李、前往機場或車站、上車或上飛機、抵達、找住宿等等。這種一步一步來、而且還要考慮先後順序跟各種限制的過程,本質上就是我們在代理系統裡所說的「規劃」,英文是 planning。

那把這個概念搬到 AI 代理身上,是什麼樣子呢?在 AI 代理的脈絡裡,規劃通常是這樣的:代理接收到一個高層次的目標,然後它會自己、或半自動地,把這個大目標拆解成一連串的中間步驟,或者說是一個個小目標,英文叫 sub-goal,子目標。這些步驟接下來可以一個接一個照順序執行,也可以用更複雜的流程來跑,過程中還可能搭配其他模式一起用。比方說工具使用,英文是 tool use,就是代理去呼叫外部工具;比方說路由,英文是 routing,就是把任務分派到不同的處理路徑;又比方說多代理協作,英文是 multi-agent collaboration,就是好幾個代理分工合作。至於規劃這件事本身怎麼做到的呢?它可能用到精密的搜尋演算法,可能用到邏輯推理,但現在愈來愈常見的,是直接運用大型語言模型的能力。大型語言模型,英文縮寫是 LLM,全名是 Large Language Model,它可以根據自己的訓練資料跟對任務的理解,直接生成一份看起來合理、而且行得通的計畫。

那為什麼規劃這麼重要?因為良好的規劃能力,能讓代理處理那些不是「問一句、答一句」就能解決的問題。它讓代理能應付多面向、很多環節的請求;它讓代理能在情況改變時,透過「重新規劃」來適應,重新規劃英文叫 replanning,意思就是計畫趕不上變化時,代理重新生成一份新計畫;它還讓代理能協調複雜的工作流程。所以說,規劃是一種很基礎的模式,它撐起了很多更進階的代理行為。它最關鍵的價值,是把一個單純的「反應式」系統,也就是英文 reactive、你戳它一下它才動一下的系統,轉變成一個能主動朝著既定目標前進的系統。這個「從被動反應到主動追求目標」的轉變,你先記在心裡,因為它是整章一再回來的主軸。

接下來,我們看看這個模式在真實世界裡到底用在哪些地方。我會講六個案例,你會發現它們有一個共同的骨架:都是先設一個目標,然後持續監控,再根據監控的結果去調整。

第一個是客戶支援自動化。代理的目標可能是「解決客戶的帳單疑問」。它會一邊監控對話,一邊去查資料庫裡的紀錄,然後動用工具去調整帳單。那它怎麼知道自己成功了?它會去確認帳單真的改好了,而且收到客戶的正面回饋。如果問題一直沒解決,它就會把這個案子升級處理,英文叫 escalate,也就是交給更高層級或人工來接手。

第二個是個人化學習系統。學習代理的目標可能是「提升某位學生對代數的理解」。它會監控這位學生在練習題上的進度,適時調整教材,同時追蹤一些表現指標,像是答對的準確率、完成題目花的時間。一旦發現學生卡住了,它就調整自己的教學方式。

第三個是專案管理助理。你可以指派給它一個任務:「確保專案的某個里程碑,在某個日期之前完成」。它會去監控各項任務的狀態、團隊之間的溝通、還有資源夠不夠用。當這個目標看起來有風險、可能要延遲的時候,它會把延遲標示出來,並且主動提出修正措施的建議。

第四個是自動化交易機器人。交易代理的目標可能是「在不超出可承受風險的前提下,把投資組合的收益最大化」。它會持續盯著市場資料、目前投資組合的價值、還有各種風險指標。當條件符合它的目標時,它就執行交易;當風險超過了設定的門檻,它就調整策略。你注意到沒有,這裡的目標是帶條件、帶限制的,不是無腦衝收益。

第五個是機器人與自駕車。自駕車最首要的目標是「安全地把乘客從一個地方載到另一個地方」。它會不停地監控周遭環境,像是其他車輛、行人、交通號誌;也監控自己的狀態,像是車速、油量;同時還監控自己沿著規劃路線走到哪了。然後它會持續調整駕駛行為,務求又安全又有效率地達成目標。

第六個是內容審核。代理的目標可以是「找出並移除某個平台上的有害內容」。它會監控傳進來的內容,套用分類模型去判斷,並且追蹤一些指標,特別是「誤判」這件事——也就是把好內容錯殺,或者把壞內容放過。然後它會根據這些指標去調整過濾的標準,或者把那些模稜兩可、不好判斷的案例,升級交給人工審核者。

這六個案例講完,你應該能感受到一個共通點:凡是需要可靠運作、需要達成特定成果、又需要適應動態變化條件的代理,這個模式都是根本的基礎。它為「智慧化的自我管理」提供了一套必要的框架。

好,概念講夠了,我們來動手看一個實際的範例。這個範例用的是 LangChain 跟 OpenAI 的應用程式介面,也就是 OpenAI API。LangChain 是一套幫你串接大型語言模型、打造代理應用的開發框架。這支範例是一段 Python 腳本,它描繪了一個自主 AI 代理,專門用來「生成並精煉 Python 程式碼」。它的核心功能,是針對你指定的問題寫出解決方案,同時確保這個方案符合你自己定義的品質基準。

那它怎麼體現「目標設定與監控」呢?關鍵在於:它不是寫一次程式碼就交差,而是進入一個迭代循環,這個循環由三個動作組成——創作、自我評估、然後改善。最有意思的地方在於,這個代理「成功了沒有」,是由它自己用 AI 來判斷的;它會自己評判,剛剛生成的程式碼有沒有真的滿足一開始設定的目標。最後它會輸出一個潤飾過、附上註解、可以立刻拿來用的 Python 檔案,作為這整個精煉過程的成果。

在跑這支腳本之前,你要先安裝幾個套件,主要是 LangChain 對 OpenAI 的整合套件、OpenAI 官方套件,還有一個用來讀取環境設定檔的小工具。另外你要準備一個設定檔,裡面放你的 OpenAI 金鑰,程式啟動時會去讀這把金鑰,如果沒設定,它就會直接報錯停下來,提醒你要先設好金鑰。

理解這支腳本最好的方式,是把它想像成一位被指派到專案上的自主 AI 程式設計師。整件事的起點,是你交給這個 AI 一份詳細的「專案說明書」,英文叫 project brief,也就是它要解決的那個具體程式設計問題。除了這份說明書,你還會給它一份嚴格的「品質檢查清單」,英文叫 quality checklist,這份清單就代表了最終程式碼必須滿足的目標。舉幾個例子:解決方案必須夠簡單、它在功能上必須正確、它要能處理那些意料之外的邊界情況等等。請注意,這份品質清單,就是這個模式裡的「目標」;後面的監控,全都是拿成果來對照這份清單。

接下來我用講的,帶你走過這支程式內部的幾個關鍵環節,我不會去唸什麼括號引號,我只講每一塊在做什麼、輸入什麼、輸出什麼、為什麼這樣設計。

首先,程式一開始會初始化一個大型語言模型,這裡用的是 GPT-4o 這個模型,並且把它的「溫度」參數設得比較低。溫度這個參數,你可以理解成「創意的隨機程度」,設低一點,代表模型回答會比較穩定、比較可預測,這對寫程式碼這種需要正確性的任務來說是合理的選擇。

再來是一個負責「建構提示詞」的環節。它的輸入有四樣:要解決的使用案例、目標清單、上一輪生成的程式碼、還有上一輪得到的回饋。它的輸出,是一段給模型看的完整指令。這段指令大致在說:「你是一個 AI 程式設計代理,請根據這個使用案例寫 Python 程式碼,而你的目標有以下這幾項。」如果這不是第一輪,它還會聰明地把上一版的程式碼跟針對那一版的回饋,一併附進指令裡,讓模型知道「上次你寫成這樣,別人這樣批評,請你改進」。最後它會特別交代模型:只要回傳修訂後的程式碼就好,不要在程式碼之外多寫說明。這一塊的設計意圖,就是讓每一輪的生成都能「站在上一輪的肩膀上」。

第二塊,是負責「評估程式碼、給出回饋」的環節。它的輸入是一段程式碼跟目標清單,輸出是一段文字回饋。它送給模型的指令是:「你現在是一位 Python 程式碼審查員,這裡有一段程式碼,請根據以下目標來評論它,指出這些目標達成了沒有,並說明在清晰度、簡潔性、正確性、邊界情況處理、測試覆蓋率這些方面要不要改進。」這一塊扮演的,就是「監控」裡頭那個「評估」的角色。

第三塊,我覺得是整個模式最畫龍點睛的地方,它負責「判定目標達成了沒有」。它的輸入是剛剛那段文字回饋跟目標清單,輸出只有一個東西:真或假,也就是 True 或 False。它怎麼做的呢?它再一次去問大型語言模型,但這次的問題很單純:「這是目標,這是針對程式碼的回饋,根據這份回饋,目標到底達成了沒有?請你只回答一個詞,True 或 False。」為什麼要逼模型只回一個詞?因為這樣程式就能很乾脆地判斷:回的是 True,就停止循環;不是 True,就再跑一輪。這個「用一個布林值當作停止訊號」的設計,就是監控回饋迴路能不能收斂的關鍵開關。

剩下還有幾個比較輔助性的環節,我簡單帶過。有一塊負責「清理程式碼區塊」,因為模型有時候會習慣在程式碼前後包上標記,這塊就把那些多餘的標記去掉,只留下乾淨的程式碼。有一塊負責「加上註解標頭」,在程式碼最前面補一行註解,說明這支程式對應的是哪個使用案例。有一塊負責「把文字轉成適合當檔名的格式」,把空白換成底線、去掉奇怪的符號。還有一塊負責「把程式碼存成檔案」,它甚至會再請模型幫忙,把使用案例濃縮成一個十個字元以內的小寫短名,當作檔名的一部分,後面再接一組隨機數字避免檔名撞名,最後寫進磁碟。

最核心的,是那個「主代理函式」,也就是把上面所有環節串起來的主迴圈。它的輸入是使用案例、用逗號分隔的目標字串、還有一個最多迭代次數,這裡預設是五次。它的運作流程是這樣的:它會跑一個最多五輪的迴圈,每一輪先建構提示詞、生成程式碼,然後把生成的程式碼送去評估、拿到回饋,接著用剛剛那個「判定」環節去問模型「目標達成了嗎」。如果回答是達成了,它就印出「確認目標已達成,停止迭代」然後跳出迴圈;如果還沒達成,它就把這一輪的程式碼存起來當作「上一版」,帶進下一輪繼續改。等迴圈結束之後——不管是達標而提早結束,還是跑滿五輪——它都會把最終的程式碼加上註解標頭,然後存成檔案。腳本最底下還準備了幾個範例輸入,預設啟用的那個,是請它寫一支計算正整數「二進位間隙」的程式,目標清單包含「容易理解、功能正確、完整處理邊界情況、只接受正整數輸入、印出幾個範例結果」這幾項。

走到這裡,你應該已經能在腦中想像這位 AI 程式設計師的工作節奏了。它拿著任務指派開始動工,寫出第一版草稿。但它不會馬上交件,而是停下來做一件關鍵的事:嚴謹的自我審查,英文叫 self-review。它會一絲不苟地把自己寫的東西,對照品質清單上的每一個項目,扮演起自己的品質保證檢查員,品質保證英文是 quality assurance。檢查完,它對自己的進度給出一個不偏不倚的裁決:全部符合就回答 True,還有不足就回答 False。

如果裁決是 False,這個 AI 並不會放棄。它會進入一個審慎的修訂階段,用剛剛自我批判得到的洞見,精準找出弱點,然後聰明地重寫程式碼。這個「起草、自我審查、精煉」的循環會一直跑下去,每一輪都力求更靠近目標一點。它會持續重複,直到它真的滿足了每一項要求、達成 True 的狀態,或者直到它撞到事先定好的嘗試次數上限為止——就像一個趕著截止期限的開發者,時間到了也得交。一旦程式碼通過了最後的檢查,腳本就把這份潤飾好的解決方案打包,加上有用的註解,存成一個乾淨、全新、隨時能用的 Python 檔案。

不過,這裡我要很誠實地提醒你一些注意事項與考量,英文叫 caveats and considerations。第一,必須講清楚,這只是一個示範性的說明,不是可以直接丟到正式生產環境用的程式碼,正式生產環境英文叫 production-ready。在真實世界的應用裡,有好幾個因素你必須納入考量。第二,大型語言模型可能根本沒完全搞懂某個目標真正的意思,結果它就錯誤地把自己的表現評為成功。第三,就算目標被正確理解了,模型還是可能產生「幻覺」,英文叫 hallucinate,意思是它可能一本正經地生出看似合理、實際上錯誤或虛構的內容。第四,這點特別微妙:當同一個大型語言模型,同時負責「寫程式碼」又負責「評判程式碼好不好」的時候,它會更難察覺自己正走錯方向——因為球員兼裁判,本來就容易自我感覺良好。

說到底,大型語言模型不會像變魔術一樣,憑空生出毫無瑕疵的程式碼;你終究還是得實際把產出的程式碼跑起來、測試過。而且,這個簡單範例裡的「監控」其實相當基礎,它還帶來一個潛在風險:這個過程有可能會一直跑、停不下來。所以「最多五次迭代」這個上限,不只是隨便設的,它其實是一道防止無窮迴圈的保險。

那有沒有更好的做法?有的。書裡給了一個方向,而且還附了一段可以直接拿來用的審查員提示詞。這段提示詞的精神是這樣的:它要求模型扮演一位資深的程式碼審查員,核心使命是消除程式碼的幻覺,確保每一項建議都立基於現實跟最佳實務。具體來說,它要求審查員做四件事:第一,找出並修正錯誤,包括邏輯瑕疵、臭蟲、還有可能在執行時爆掉的問題;第二,簡化跟重構,在不犧牲正確性的前提下,讓程式碼更好讀、更有效率、更好維護;第三,提供清楚的說明,每一項建議都要講清楚它為什麼是改進,並且援引乾淨程式碼、效能或安全性的原則;第四,提供修正後的程式碼,把「改之前」跟「改之後」都秀出來,讓改進一目瞭然。整體基調要直接、有建設性,始終以提升品質為目標。

但比起調整提示詞,更穩健的解法是「把職責分開」。與其讓一個代理球員兼裁判,不如組一支代理團隊,英文叫 crew,讓每個代理各司其職。書裡的作者就用 Gemini 建了一支這樣的團隊,裡頭有五種角色。第一種是同儕程式設計師,負責協助寫程式碼、一起腦力激盪。第二種是程式碼審查員,專門抓錯誤、提改進建議。第三種是文件撰寫者,負責生成清楚又精簡的文件。第四種是測試撰寫者,負責建立全面的單元測試。第五種是提示精煉者,負責最佳化跟 AI 的互動。

這樣分工的妙處在哪裡?關鍵在於,程式碼審查員現在是一個「獨立於程式設計師之外」的實體。雖然它用的提示詞,跟前面那個球員兼裁判的範例裡那位評判者很類似,但因為它是另一個獨立的代理,客觀評估的品質就大幅提升了——它沒有「護短自己作品」的包袱。而且這種結構自然就帶來更好的實務做法,比方說那位測試撰寫者代理,天生就能滿足「為同儕程式設計師寫的程式碼補上單元測試」這個需求。至於怎麼把這些更精密的控制機制加進去、讓程式碼更接近能上正式生產環境的水準,書裡把這個任務留給有興趣的讀者自己去完成。

我們把整章的重點收束一下。先講「這是什麼」。AI 代理常常缺乏明確的方向,這讓它們除了簡單的反應式任務以外,就沒辦法帶著目的去行動。沒有既定目標,它們既無法獨立處理複雜的多步驟問題,也無法協調精密的工作流程。更重要的是,它們本身沒有內建的機制,去判斷自己的行動到底有沒有導向成功。這就限制了它們的自主性,也讓它們在那些「光把任務做完還不夠」的動態真實情境裡,難以真正派上用場。

再講「為什麼」要用這個模式。目標設定與監控提供了一套標準化的解法,把「目的感」跟「自我評估能力」嵌進代理系統裡。它做的事,一是為代理明確定義出清楚、可衡量的目標;二是建立一套監控機制,持續對照這些目標,去追蹤代理的進度跟它所處環境的狀態。這就形成了一個關鍵的回饋迴路,英文叫 feedback loop。有了這個迴路,代理就能評估自己的表現、修正方向,在它偏離成功路徑的時候,即時調整計畫。透過實作這個模式,開發者就能把一個單純的反應式代理,改造成一個能自主、可靠運作,而且以目標為導向的主動式系統。

那什麼時候該用它?給你一句經驗法則:當一個 AI 代理必須自主執行多步驟任務、必須適應動態變化的條件,而且要在沒有人持續盯著、持續介入的情況下,可靠地達成某個特定的高層次目標時,就該用這個模式。

最後我把幾個最該記住的要點再串一次。第一,目標設定與監控,為代理配備了目的感,以及追蹤進度的機制。第二,目標最好符合所謂的 SMART 原則。SMART 是五個英文字的首字母縮寫:目標要具體,英文是 Specific;要可衡量,Measurable;要可達成,Achievable;要相關,Relevant;而且要有時限,Time-bound。把這五個字記起來,你設目標就不會空泛。第三,清楚地定義指標跟成功標準,對有效的監控至關重要——因為你沒辦法監控一個你說不清楚的東西。第四,監控的內容,涵蓋觀察代理的行動、環境的狀態、還有工具的輸出。第五,來自監控的回饋迴路,讓代理能夠適應、能夠修訂計畫,或者在搞不定的時候把問題升級處理。第六,如果你用的是 Google 的 ADK,也就是 Agent Development Kit、代理開發套件,那麼在它裡頭,目標通常是透過「代理指令」來傳達,英文是 agent instructions;而監控則是靠「狀態管理」跟工具互動來達成,狀態管理英文是 state management。

總結來說,這一章聚焦在目標設定與監控這個關鍵範式。我們一再強調的,是它怎麼把 AI 代理從一個單純的反應式系統,轉變成一個主動的、目標驅動的實體。我們特別著重兩件事:一是定義清楚、可衡量目標的重要性,二是建立嚴謹監控程序來追蹤進度的重要性。那一連串的實務應用,從客戶服務到自駕車,展示了這個範式怎麼在各種領域裡支撐起可靠的自主運作;而那個會自我精煉的程式碼代理範例,則具體說明了怎麼在一個結構化的框架裡,把這些原則落地。歸根究柢,賦予代理「自己制定目標、又自己監督目標」的能力,是我們朝著打造真正智慧、而且可究責的 AI 系統,所邁出的根本一步。可究責,英文是 accountable,意思是它的行為跟結果是可以被追蹤、被檢驗、被負責的——而這,正是目標設定與監控帶給代理最珍貴的東西。
