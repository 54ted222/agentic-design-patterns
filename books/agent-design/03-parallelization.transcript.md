這一章我們要談的,是一個能讓你的代理系統「跑得更快」的關鍵設計模式,叫做平行化,英文是 Parallelization。

在開始之前,我先幫你把這一章的路線圖講清楚。我們會先弄懂平行化到底是什麼、它跟前面學過的兩個模式有什麼不一樣;接著用一個研究主題的例子,讓你直觀感受循序做事跟平行做事的差別;然後我會帶你看七種真實世界裡用得上平行化的場景;再來是兩段動手實作的範例,一段用 LangChain,一段用 Google 的代理開發套件;最後我們把整章的重點收斂起來。整段都是用聽的就能懂,程式碼我不會逐行唸,而是講它在做什麼、為什麼這樣設計。

我們先從「平行化到底要解決什麼問題」說起。在前面的章節裡,我們談過兩個模式。第一個叫提示鏈,英文是 Prompt Chaining,它處理的是一步接著一步、循序進行的工作流程,前一步的輸出餵給後一步。第二個叫路由,英文是 Routing,它處理的是動態決策,也就是根據情況,決定要走哪一條路徑。這兩個模式都很重要,可是它們有一個共同的前提,就是事情大致是一件做完才做下一件。

問題來了。很多複雜的代理任務,英文我們把這種具自主行為的系統叫 agentic,它裡面其實包含好幾個子任務,而這些子任務彼此之間根本不需要互相等待,完全可以同時進行。一旦遇到這種情況,平行化就變得非常關鍵。

那平行化的做法是什麼?簡單講,就是同時執行多個元件。這些元件可以是對大型語言模型的呼叫,大型語言模型英文叫 Large Language Model,簡稱 LLM,後面我都直接說 LLM。元件也可以是工具的使用,甚至可以是一整個子代理,英文叫 sub-agent。重點在於,與其等某一步做完才開始下一步,我們讓那些彼此獨立的任務同時進行。對於那些可以被拆成好幾個獨立部分的任務來說,這能大幅縮短整體的執行時間。

我用一個具體的例子讓你感受這個差別。想像你有一個代理,它的任務是研究某個主題,然後把研究發現摘要出來。

如果用循序的做法,它會這樣走:第一,去搜尋來源 A;第二,把來源 A 摘要出來;第三,去搜尋來源 B;第四,把來源 B 摘要出來;第五,綜合摘要 A 跟摘要 B,整理出最終答案。你注意到了嗎?它每一步都在等前一步。搜尋 B 明明跟搜尋 A 一點關係都沒有,卻硬要排在 A 後面才做。

那平行的做法會怎麼走呢?第一步,同時去搜尋來源 A 跟來源 B,兩個一起發出去。第二步,等兩個搜尋都回來之後,同時摘要來源 A 跟來源 B。第三步,綜合兩份摘要,整理出最終答案。最後這一步通常還是得循序,因為它必須等前面兩條平行的線都跑完,才有東西可以綜合。

你抓到核心理念了嗎?平行化的精神,就是去工作流程裡面找出那些「不依賴別人輸出」的環節,然後把它們同時執行。哪些步驟彼此不互相需要,就讓它們一起跑。

這個做法在一種情境下特別有威力,就是當你需要去跟有延遲的外部服務打交道的時候。延遲,英文叫 latency,意思就是你發出請求之後,得等一段時間才會有回應。像是呼叫外部的應用程式介面,也就是 API,或是查詢資料庫,這些動作的特色就是你大部分時間都在「等」。既然在等,那何不一次把好幾個請求都發出去,讓它們一起等?這就是平行化省時間的關鍵。

要實際做出平行化,通常需要框架支援一種叫「非同步執行」的能力,非同步英文叫 asynchronous;或者支援多執行緒、多行程,英文是 multi-threading 跟 multi-processing。好消息是,現代的代理框架在設計的時候就把非同步操作考慮進去了,所以你可以很輕鬆地定義出哪些步驟要平行跑。

說到框架,目前主流的幾個都提供了平行執行的機制,我幫你說明它們各自的做法。第一個是 LangChain,它有一套叫 LangChain 表達式語言的東西,英文是 LangChain Expression Language,簡稱 LCEL。在這套語言裡,你可以用運算子去組合那些「可執行物件」,英文叫 runnable。比方說有一個直立線的符號,代表的是循序串接;而你也可以把鏈或圖組成「有好幾條分支可以同時跑」的結構,這樣就達成了平行。第二個是 LangGraph,它靠的是「圖」這種結構,英文是 graph。它讓你可以定義出,從同一個狀態轉移點,一次觸發好幾個節點,這樣就在工作流程裡開出了平行的分支。第三個是 Google 的代理開發套件,英文全名是 Agent Developer Kit,簡稱 ADK,後面我都說 ADK。它提供了一套穩健又原生的機制,專門用來促成並管理代理的平行執行,讓複雜的多代理系統效率更高、更容易擴展。因為這個能力是內建的,開發者可以直接設計出讓多個代理同時運作、而不是排隊運作的解決方案。

總結這一段:平行化對於提升代理系統的效率跟回應速度非常重要,尤其當任務牽涉到很多獨立的查詢、運算,或是要跟外部服務互動的時候。它是優化複雜代理工作流程效能的一項關鍵技術。

講完了概念,我們來看看平行化在實務上到底可以用在哪些地方。我會給你七個使用案例,每一個我都用一句話的場景加上「同時做哪些事」來說明,你聽的時候可以想像那個畫面。

第一個場景是資訊蒐集與研究。這是平行化最經典的用法。假設你有一個代理,要研究某一家公司。它可以同時做四件事:去搜尋這家公司的新聞文章、去拉取它的股價資料、去檢查社群媒體上有沒有人提到它、再去查公司的內部資料庫。這四件事彼此沒有先後關係,一起做,你就能比一個一個查快很多,而且更快拿到一個全面的視角。

第二個場景是資料處理與分析。這裡的概念是,你可以同時套用好幾種不同的分析技術,或者同時處理不同的資料區段。舉個例子,一個分析顧客回饋的代理,拿到一批回饋之後,它可以同時做情緒分析、擷取關鍵字、把回饋分類,還有找出哪些是緊急問題。同一批資料,四種分析一起跑,馬上就給你一個多面向的結果。

第三個場景是多 API 或多工具的互動。也就是同時呼叫好幾個彼此獨立的 API 或工具,去蒐集不同類型的資訊,或執行不同的動作。最好懂的例子是旅遊規劃代理。它可以同時去查機票價格、搜尋飯店空房、找當地有什麼活動、再找餐廳推薦。這四個查詢來自完全不同的服務,一起發出去,你就能更快地拿到一份完整的旅遊計畫。

第四個場景是「由多個元件組成的內容生成」。意思是,一份複雜的內容,它的不同部分可以平行生成。比如一個撰寫行銷電子郵件的代理,它可以同時生成郵件的主旨列、草擬郵件的內文、找一張相關的圖片,還有寫好那個「行動呼籲」按鈕上的文字。行動呼籲英文叫 call-to-action,就是那種「立即購買」「馬上報名」的引導文字。這些部件各自獨立,平行生成完再組裝起來,效率高很多。

第五個場景是驗證與查核。也就是同時跑好幾項彼此獨立的檢查。舉例來說,一個負責驗證使用者輸入的代理,它可以同時檢查電子郵件的格式對不對、驗證電話號碼合不合法、去資料庫比對地址、再檢查內容裡有沒有不雅字眼。這幾項檢查互不相干,一起做,使用者就能更快得到「你填的資料有沒有問題」的回饋。

第六個場景是多模態處理。模態這個詞英文叫 modality,指的是資訊的不同形式,像是文字、影像、音訊都是不同的模態。概念是,同一份輸入裡面如果同時有好幾種模態,你可以平行去處理它們。例子是一個分析社群媒體貼文的代理,而這篇貼文同時有文字也有圖片。它可以一邊針對文字去分析情緒跟關鍵字,同時另一邊針對圖片去分析裡面有什麼物件、場景是什麼。兩種模態同時處理,就能更快地把不同來源的洞見整合起來。

第七個場景是 A/B 測試,或者說多選項生成。A/B 測試的意思,就是同時做出好幾個版本,再從裡面挑一個最好的。所以這裡的做法是,平行生成回應或輸出的好幾個變體。例子是一個生成創意文案的代理,它可以用稍微不一樣的提示詞,或甚至用不同的模型,同時幫一篇文章生出三個不同的標題。這樣你就能很快地把它們擺在一起比較,挑出最好的那一個。

這七個場景講完,你應該能體會到一件事:平行化是代理設計裡一項很基礎的優化技術。只要懂得對那些獨立的任務施加「同時執行」,你就能打造出效能更高、回應更靈敏的應用程式。

好,概念跟場景都鋪陳完了,接下來我們進入動手實作的部分。我會帶你看兩個範例,先看 LangChain,再看 Google ADK。我提醒你一次,程式碼我不會唸符號,我會講它在做什麼、輸入是什麼、輸出是什麼、為什麼這樣設計。

先講 LangChain 這個範例的整體想法。在 LangChain 裡,平行執行是透過剛剛說的那套 LCEL 表達式語言來達成的。它的核心招數是這樣:你把好幾個可執行的元件,組織成一個字典或一個清單的結構。字典你可以想成是一組「名字配對應內容」的對照表。當這一整包東西被當成輸入,傳給鏈裡面後續的元件時,LCEL 的執行階段就會聰明地把這一包裡面的東西「同時」執行掉。這就是它平行的祕密。

順帶一提 LangGraph 是怎麼做的,原理其實一樣,只是換成套用在「圖的拓樸」上。拓樸英文叫 topology,你可以理解成「節點之間怎麼連接的結構」。你把圖設計成,讓好幾個彼此沒有直接前後依賴的節點,都從同一個共同節點被觸發出去。這些平行的路徑各自獨立跑,一直跑到後面一個「匯流點」才把結果彙整起來。匯流點英文叫 convergence point,你就想成是好幾條河最後匯流在一起的那個交會處。

現在我們看這個 LangChain 範例具體在做什麼。它要解決的事情是:當使用者丟進來一個主題,系統要同時做兩件以上彼此獨立的事,然後把這些結果彙整成一個統一的答案。在跑之前,你需要先準備好幾個東西:安裝必要的 Python 套件,包括 LangChain 本體、它的社群套件,還有一個模型供應商的函式庫,例如連接 OpenAI 的那個;另外,你選用的那個語言模型的 API 金鑰,也就是用來驗證身分的那把鑰匙,必須先設定在你本機的環境裡。

程式一開始,先初始化一個語言模型的實例,這個範例用的是一個叫做 gpt-4o-mini 的模型,並且設了一個叫 temperature 的參數,這個參數是用來控制模型回答的「創意程度」,值越高回答越天馬行空。初始化的時候,程式用了一個防錯的結構把它包起來,意思是萬一模型初始化失敗,程式不會直接崩潰,而是會印出錯誤訊息,讓系統更穩健。

接著,程式定義了三條彼此獨立的鏈,每一條負責對同一個輸入主題做一件不同的事。第一條鏈,負責把主題精簡地摘要出來。第二條鏈,負責針對這個主題,生成三個有趣的問題。第三條鏈,負責從主題裡面辨識出五到十個關鍵詞,並且用逗號分隔。這三條鏈長得很像,每一條都是由三個部分串起來的:一個為它特定任務量身打造的提示模板、接上那個語言模型、最後再接上一個把輸出整理成純文字字串的解析器。你可以把每一條鏈想成是一條小生產線,原料是主題,成品分別是摘要、問題跟關鍵詞。

關鍵的一步來了。程式建立了一個平行的區塊,這個結構在 LCEL 裡叫 RunnableParallel,你聽名字就知道,它就是「可平行執行的東西」。它把剛剛那三條鏈綁在一起,讓它們可以同時開跑。除了這三條鏈,這個平行區塊裡還多放了一個東西,叫 RunnablePassthrough,直譯是「直接通過」。它的作用是,把原始的那個主題原封不動地往下傳。為什麼要這樣?因為後面綜整的時候,除了要用到摘要、問題、關鍵詞,可能還會想參考最原始的主題本身,所以要確保它不會在中途搞丟。

接著程式定義了最後的綜整步驟,這也是一個提示模板。它把四樣東西當成輸入,分別是摘要、那三個問題、那些關鍵詞,還有原始主題,然後要求模型根據這些資訊,生出一個全面的答案。

最後,程式把整條端到端的完整鏈組起來,取名叫 full_parallel_chain。它的組裝順序是這樣:先是那個平行區塊,跑完三條鏈;接著把結果交給綜整提示;再接上語言模型;最後接上輸出解析器。整條鏈的故事就是:三件事同時做完,然後交給最後一次模型呼叫,把它們組合成一個答案。

程式還提供了一個非同步的執行函式,用來示範怎麼呼叫這條完整的鏈。它接收一個主題當輸入,用一個叫 invoke 的非同步呼叫方式去跑。最後,程式用一個標準的 Python 進入點,實際拿一個範例主題去執行,這個範例的主題是「太空探索的歷史」,並且用 asyncio 這個工具來管理整個非同步的執行。

這裡有一個非常重要、很容易搞混的觀念,我特別把它拉出來講。這段程式用的 asyncio,它提供的其實是「並行」,英文叫 concurrency,而不是真正的「平行」,英文叫 parallelism。這兩個詞中文聽起來很像,但意義不一樣,我用比喻幫你分清楚。並行,比較像一個廚師在廚房裡同時顧好幾道菜:他煮著湯的時候,湯在那邊滾,他不會乾等,他會轉去切菜、去煎魚。表面上好幾道菜都在推進,但其實從頭到尾只有「一個」廚師在動。asyncio 就是這樣運作的:它在單一執行緒上,靠一個叫「事件迴圈」的機制,英文是 event loop,當某個任務閒下來、在那邊等的時候,比方說在等網路回應,事件迴圈就聰明地切換去處理別的任務。這營造出好幾個任務同時前進的效果,可是底層仍然只有單一執行緒在跑。

那真正的平行又是什麼?那比較像廚房裡請了好幾個廚師,真的有好幾雙手同時在動。而 Python 之所以在這種情境下通常做的是並行而非真平行,是因為它有一個東西叫「全域直譯器鎖」,英文是 Global Interpreter Lock,簡稱 GIL。你可以把它想成廚房裡只有一把「主廚的刀」,同一時間只能有一個人拿著它做核心工作,所以即使你開了多執行緒,真正執行 Python 程式碼的核心動作還是被這把鎖限制住。不過你別擔心,對於我們這種「大部分時間在等外部服務回應」的代理任務來說,並行就已經非常夠用了,因為瓶頸本來就在「等」,不在「算」。

所以整段 LangChain 範例的本質就是:針對一個給定的主題,讓好幾次 LLM 呼叫——摘要、生成問題、抓關鍵詞——同時發生,然後再由最後一次 LLM 呼叫把它們的結果組合起來。這就完整展示了用 LangChain 在代理工作流程裡做平行化的核心理念。

講完 LangChain,我們把注意力轉向第二個範例,用的是 Google 的 ADK 框架。這個範例會讓你看到兩個基本元件怎麼搭配,一個叫 ParallelAgent,也就是「平行代理」,另一個叫 SequentialAgent,也就是「循序代理」。我們要建的,是一個善用同時執行來提升效率的代理流程。

這個範例的任務是:研究並綜整「永續科技的最新進展」。整個系統的設計是這樣分工的。

首先,程式設置了三個研究員代理,它們都是語言模型代理,英文叫 LlmAgent,而且它們各有專長。第一位研究員專精能源,負責去研究「再生能源來源」的最新進展。第二位研究員專精交通運輸,負責研究「電動車技術」的最新發展。第三位研究員專精氣候解決方案,負責調查「碳捕捉方法」的現況。

這三位研究員的設定模式幾乎一樣,我講共通點。每一位都被指定使用同一個 Gemini 模型,而且都配備了一個 Google 搜尋工具,讓它們能上網查資料。它們收到的指令都很明確:你是某某領域的研究助理,去研究指定的題目,用提供的搜尋工具,然後用一到兩句話精簡地摘要你的關鍵發現,而且只輸出摘要本身,不要囉嗦。最關鍵的設計是,每一位研究員都設了一個叫 output_key 的東西,中文是「輸出鍵」。它的作用是,把這位研究員產出的摘要,存進一個共享的地方,這個地方叫 session state,也就是「會話狀態」,你可以把它想成是一塊三位研究員跟後面的人都看得到的共用白板。再生能源的結果存在白板上一個格子、電動車存在另一個格子、碳捕捉存在第三個格子,各自有各自的名牌。

接下來是第二個部分,也是這個範例的重頭戲。程式建立了一個平行代理,取名叫 ParallelWebResearchAgent。它的職責,就是協調那三位研究員「同時」開工。把三位研究員當成它的子代理交給它,它就讓研究平行進行,這樣有機會省下大把時間。這個平行代理有一個明確的結束條件:一旦它底下所有的研究員都跑完、而且都把結果寫上了共用白板,它就結束自己的任務。

第三個部分,程式定義了一個合併代理,叫 MergerAgent,它本身也是一個語言模型代理。它的工作是把研究結果綜整起來。它的輸入,就是剛剛三位研究員寫在共用白板上的那三份摘要。這個代理的指令裡有一個極度重要的原則,我特別強調:它被要求,整份輸出「必須、而且只能」根據提供給它的那三份輸入摘要,嚴禁加入任何外部知識、任何沒有出現在這些摘要裡的事實或細節。這個原則在做檢索增強生成這類應用時很常見,目的是避免模型自己「腦補」,確保報告裡的每一句話都有來源、都站得住腳。這個合併代理被設計成,把組合後的發現整理成一份結構化的報告:每個主題各有一個標題,清楚標明這段發現是來自哪一位研究員,最後再附上一段一到兩句話的整體結論。另外有一個小細節值得一提,這個合併代理不需要設輸出鍵,因為它就是整個流程的最後一棒,它的回應本身就是最終輸出,不需要再存給誰。

第四個部分,程式建立了一個循序代理,叫 ResearchAndSynthesisPipeline,中文是「研究與綜整管線」。它扮演的是整個流程的總指揮。它底下放了兩個子代理,順序很重要:先是那個平行代理,然後才是合併代理。執行的時候,它會先啟動平行代理,讓三位研究員同時去查資料、把白板填滿;等平行代理整個結束之後,它再啟動合併代理,把白板上的資料綜整成最終報告。最後,程式把這個循序代理設定成 root_agent,也就是「根代理」,代表整個多代理系統的執行進入點,你從這裡按下啟動鍵,整套流程就跑起來。

我幫你把這個 ADK 範例的設計再串一次,你就完全清楚了:三位各有專長的研究員,被一個平行代理協調著同時上網查資料,各自把摘要寫上共用白板;接著一個合併代理,只根據白板上的內容,把三份發現整理成一份結構化、有出處的報告;而最外層,有一個循序代理當總指揮,先平行研究、再循序合併。整個設計的目的,就是有效率地從多個來源平行蒐集資訊,然後把它們組合成一份單一、結構清楚的報告。

兩個範例都看完了,我們來把這一章的重點收斂起來。我會用「是什麼、為什麼、什麼時候用」這個三段式,幫你把整章的精神再過一遍。

先講「是什麼」。很多代理工作流程裡,都包含好幾個必須完成才能達成最終目標的子任務。如果你純粹用循序的方式跑,每一項任務都得乾等前一項做完,那往往又慢又沒效率。尤其當任務依賴的是外部的輸入輸出操作,例如呼叫不同的 API、查詢好幾個資料庫,這種「等待」的延遲就會變成很顯著的瓶頸。因為在沒有同時執行的機制下,你的總處理時間,就等於把每一項任務的耗時通通加起來。這會嚴重拖累整個系統的效能跟回應速度。

再講「為什麼」。平行化模式提供了一套標準化的解法,讓那些彼此獨立的任務能夠同時執行。它的運作方式,就是去找出工作流程裡那些「不需要等別人即時輸出」的元件,像是某些工具使用或某些 LLM 呼叫,然後讓它們一起跑。LangChain 跟 Google ADK 這些框架,都提供了內建的結構來定義並管理這些同時進行的操作。舉個典型的例子,一個主行程可以一口氣觸發好幾個平行的子任務,然後等它們全部完成,才進到下一步。靠著讓這些獨立任務同時跑、而不是一個接一個排隊,這個模式大幅縮短了總執行時間。

接著講「經驗法則」,也就是你什麼時候該用它。判斷的標準很簡單:當一個工作流程裡,包含好幾個可以同時做的獨立操作時,就用平行化。比方說,要從好幾個 API 擷取資料、要處理不同的資料區塊、或者要先生成好幾份內容再拿去綜整,這些都是平行化的主場。

最後,我再幫你補幾個容易被忽略、但很重要的提醒,把整章的脈絡串完。

第一,平行化的核心一句話:它是一種同時執行獨立任務、以提升效率的模式。當任務牽涉到等待外部資源,例如等 API 回應的時候,它格外有用。

第二,這是一個很多人會忽略的代價:採用並行或平行的架構,並不是免費的午餐。它會引入相當可觀的複雜度跟成本。具體影響在哪?在設計、在除錯、還有在系統的日誌記錄這幾個關鍵的開發階段。你想想看,好幾件事同時在跑,出了問題你很難重現、很難追蹤是哪一條線出錯、日誌也會交錯在一起。所以平行化雖然好,但你要評估「省下的時間」值不值得「多出來的複雜度」。

第三,回顧一下框架層面的兩個關鍵字。在 LangChain 的 LCEL 表達式語言裡,讓多個可執行物件並排運行的關鍵結構,就是我們前面用過的 RunnableParallel。而在 Google ADK 裡,它有一個特別的促成方式,叫做「LLM 驅動的委派」,英文是 LLM-Driven Delegation。意思是,由一個叫「協調者」的代理,英文是 Coordinator,它的 LLM 負責去判斷出哪些子任務是彼此獨立的,然後把這些任務分派出去,觸發各有專長的子代理同時處理。換句話說,連「哪些事可以平行」這個判斷,都可以交給模型自己來做。

第四,平行化最終帶來的好處,就是降低整體的延遲,讓代理系統在面對複雜任務時,反應更快、更有回應力。

那我幫你做整章最後的總結。平行化模式,本質上是一種優化運算工作流程的方法,做法就是同時執行那些彼此獨立的子任務。它降低了整體的延遲,而這個好處,在那些需要做很多次模型推論、或要呼叫很多外部服務的複雜操作裡,會特別明顯。

不同的框架,提供了實作這個模式的不同機制。在 LangChain 裡,你用 RunnableParallel 這類結構,明確地定義並同時執行好幾條處理鏈。而像 Google ADK 這類框架,則是透過「多代理委派」來達成平行,也就是由一個主要的協調者模型,把不同的子任務指派給各有專長、能同時運作的代理。

最後,把這一章放回我們整個學習的脈絡裡:當你把平行處理,跟前面學過的循序也就是鏈式控制流程、以及條件也就是路由控制流程,這三者整合在一起的時候,你就有能力建構出真正精密、高效能的運算系統,從容地處理各式各樣又多又複雜的任務。這,就是平行化模式帶給你的力量。
