這一章我們要談的,是怎麼讓一個語言模型不再只是會「說話」,而是能真正「動手做事」。這個關鍵的橋樑,就叫做模型情境協定,英文是 Model Context Protocol,縮寫為 MCP。接下來我會帶你從它要解決什麼問題講起,一路談到它的架構、它跟傳統工具呼叫的差異、實務上你必須注意的眉角,還有兩個動手實作的範例。你只要用聽的,就能在腦中把整套運作流程想像出來。

我們先從「為什麼需要它」說起。大型語言模型,也就是我們常說的 LLM,Large Language Model,它本身很會生成文字、生成圖片、生成各種多模態的內容。但是,如果我們希望它扮演一個「代理」的角色,英文叫 agent,也就是一個能替你完成任務的小幫手,那光會生成內容是不夠的。它必須能跟外面的世界互動:去查最新的資料、去操作別人寫好的軟體、去執行某個具體的動作,比方說寄一封信、改一筆資料庫紀錄。問題是,外面的世界五花八門,每個系統、每個資料庫、每個工具,介面都長得不一樣。如果每接一個新對象,你就要為它量身打造一套客製化的整合,那這件事很快就會變成一場惡夢:做不完、也重複用不了。MCP 就是為了終結這種混亂而誕生的,它提供一個標準化的介面,讓語言模型能用一致、可預期的方式去對接各種外部資源。

那 MCP 到底是什麼?我給你一個最直觀的比喻。你可以把它想像成一個「萬用轉接頭」。我們出國旅行的時候,各國插座規格不一樣,但只要帶一個萬用轉接頭,任何電器都能插到任何插座上。MCP 就是語言模型世界裡的那個萬用轉接頭。它是一套開放標準,目的是統一各種 LLM,不管是 Google 的 Gemini、OpenAI 的 GPT 系列、Mixtral,還是 Anthropic 的 Claude,讓它們跟外部的應用程式、資料來源跟工具溝通的方式,都遵循同一套規矩。有了它,語言模型要取得情境、要執行動作、要跟各種系統互動,就變得簡單多了,而你不必為每一個對象重寫一套整合。

接著我們來看它的架構。MCP 採用的是「主從式架構」,英文叫 client-server。這裡有兩個角色要記住。一邊是 MCP 伺服器,英文是 server,它負責「對外揭露」三種東西。第一種叫資源,英文是 resource,指的是資料,比方說一份文件、一筆資料庫紀錄。第二種叫提示,英文是 prompt,它其實是一種互動的範本,引導模型該怎麼跟資源或工具打交道。第三種叫工具,英文是 tool,它是一個能執行動作的函式,比方說寄信、查 API。另一邊的角色是 MCP 用戶端,英文是 client,它負責去「消費」這些被揭露出來的東西。這個用戶端,可能是一個包著語言模型的宿主應用程式,也可能就是 AI 代理本身。透過這種標準化的分工,把語言模型整合進各種操作環境的複雜度,就被大幅降低了。

不過這裡有一個很重要、也很容易被忽略的觀念,我要特別強調。MCP 本質上是一份「代理介面的契約」,但契約歸契約,它好不好用,高度取決於它底下包的那個 API 設計得好不好。這裡藏著一個風險:有些開發者偷懶,直接把一個老舊的、既有的 API 原封不動地包起來就丟上去,而這種舊式 API,對代理來說往往很難用。我舉一個很具體的例子你就懂了。假設有一個工單系統,它的 API 設計成「一次只能取出一張工單的完整明細」。現在你叫代理去「幫我摘要所有高優先度的工單」,結果代理只能一張一張慢慢撈,資料一多,就會又慢又容易出錯。怎麼解決?應該在底層 API 上,補上像「過濾」跟「排序」這種功能。注意這兩個功能的特性:它們是「確定性」的,英文叫 deterministic,意思是同樣的輸入一定給你同樣的輸出,很可靠。而代理本身是「非確定性」的,它的判斷帶有彈性、也帶有不確定。所以這裡有一個很深刻的體會:代理不會神奇地取代那些確定性的工作流程,相反地,它往往需要更強的確定性支援,才能真正把事情做好。

跟這個觀念相關的,還有第二個容易踩的坑,就是「資料格式」的問題。MCP 能幫你把線接上,但它不會幫你保證「接過來的資料,代理看得懂」。我再給你一個例子。假設你為一個文件儲存庫做了一個 MCP 伺服器,結果它回傳檔案的方式,是丟一個 PDF 給代理。可是如果消費端這個代理根本不會解析 PDF 的內容,那這個伺服器基本上就是廢的。更聰明的做法,是讓 API 回傳文件的「文字版本」,比方說 Markdown 這種純文字格式,代理才能真正讀進去、處理它。所以你看,開發者不能只想著「有沒有接上」,還必須想清楚「交換的資料,本質上對方吃不吃得下」,這樣才有真正的相容性。

講完 MCP 是什麼,接下來我們做一個很重要的對照:MCP 跟「工具函式呼叫」到底差在哪?工具函式呼叫,英文是 tool function calling。這兩者都能讓語言模型超越單純的文字生成,去使用外部能力、去執行動作,但它們的做法跟抽象層次,其實完全不同。

我們先講工具函式呼叫。你可以把它想成:語言模型對某一個「特定的、事先定義好的」工具,直接發出一個請求。順帶一提,在這個脈絡裡,「工具」跟「函式」這兩個詞是交替使用的,指的是同一件事。它的特徵是一種「一對一」的溝通模式:模型理解了使用者的意圖,判斷這個意圖需要動用某個外部動作,於是它把請求格式化好;接著由應用程式的程式碼去執行這個請求,再把結果回傳給模型。這整個過程的關鍵特點是:它通常是「專有的」,英文叫 proprietary,而且不同的 LLM 供應商之間,做法各不相同。換句話說,你為 A 廠商寫的工具呼叫,搬到 B 廠商不一定能用。

那 MCP 不一樣。MCP 是一個「標準化的介面」,它讓語言模型能去「發現」、去「溝通」、去「運用」各種外部能力。它扮演的是一套開放協定的角色,促成跟各式各樣工具跟系統的互動,而它的目標很遠大:打造一個生態系,讓任何符合規範的工具,都能被任何符合規範的語言模型存取。這帶來三個很關鍵的特性,我幫你把英文都點出來:第一是互通性,interoperability,不同系統之間能互相搭配;第二是可組合性,composability,各種能力可以像積木一樣拼起來;第三是可重用性,reusability,做一次到處都能用。它採用的是一種「聯邦式」的模型,英文叫 federated。這個策略最漂亮的地方在於:你那些各自獨立、零散分布的舊系統,不需要砍掉重練,你只要把它們包進一層符合 MCP 規範的介面裡,它們就能被帶進現代化的生態系。這些服務還是各自獨立運作,但現在它們可以被組合進新的應用程式跟工作流程,由語言模型來居中協調它們的協作。這就在「不必花大錢重寫底層系統」的前提下,換來了敏捷性跟可重用性。

為了讓你聽得更清楚,我把原文那張對照表的五個面向,逐一講給你聽。第一個面向是「標準化」。工具函式呼叫是專有的、跟供應商綁定的,格式跟實作換個廠商就不一樣;而 MCP 是一套開放、標準化的協定,促進不同模型跟工具之間的互通。第二個面向是「範疇」。工具函式呼叫是一個比較窄的機制,就是讓模型請求執行某個特定、事先定義好的函式;MCP 則是一個更廣的框架,它規範的是「模型跟外部工具要怎麼彼此發現、彼此溝通」。第三個面向是「架構」。工具函式呼叫是模型跟應用程式工具邏輯之間的一對一互動;MCP 是主從式架構,讓由模型驅動的應用程式,也就是用戶端,去連接並運用各式各樣的 MCP 伺服器。第四個面向是「發現」。工具函式呼叫裡,模型是在特定對話中被「明確告知」有哪些工具可用;而 MCP 支援「動態發現」,用戶端可以主動去查詢伺服器,問它「你提供哪些能力」。第五個面向是「可重用性」。工具函式呼叫的整合,常常跟特定的應用程式跟特定的模型綁得死死的;MCP 則鼓勵你開發可重用、獨立的 MCP 伺服器,任何符合規範的應用程式都能來存取。

這裡再給你一個很傳神的比喻,把這兩者的差別釘進腦海。工具函式呼叫,就像是給 AI 一組「特定、客製打造的工具」,好比某一把特定的扳手跟某一把特定的螺絲起子。對於任務固定的工作坊來說,這很有效率,因為你就那幾樣活兒。而 MCP 呢,它比較像是建立一整套「萬用、標準化的電源插座系統」。它本身不提供工具,但它讓任何廠商生產、只要符合規範的任何工具,都能插上去運作,於是你打造出來的,是一個動態的、會不斷擴充的工作坊。所以一句話總結:函式呼叫給你的是對少數幾個特定函式的直接存取;MCP 給你的是一套標準化的溝通框架,讓模型能去發現並使用範圍極廣的外部資源。簡單的應用,特定工具就夠了;但如果你要做的是需要持續調適、複雜而且彼此互連的 AI 系統,那像 MCP 這種萬用標準,就變得不可或缺。

接下來,我們進入一個比較實務的環節,聊聊在採用 MCP 之前,你應該想清楚的幾個考量面向。這些都是決定它適不適合你案子的關鍵。

第一,要分清楚「工具、資源、提示」這三者的角色,它們不是同義詞。資源是「靜態資料」,比方說一份 PDF、一筆資料庫紀錄,它就是擺在那裡的資料。工具是「會執行動作的函式」,比方說寄一封信、去查一個 API,它會改變狀態或產生結果。提示則是一個「範本」,它引導語言模型該怎麼跟資源或工具互動,確保整個互動是有結構、有效的。

第二,是「可發現性」,英文叫 discoverability。這是 MCP 的一大優勢:用戶端可以動態地去查詢伺服器,問清楚它現在提供哪些工具跟資源。這種「即時」的發現機制,英文叫 just-in-time,對代理特別有價值,因為它意味著代理可以在「不重新部署」的情況下,就學會使用新的能力。

第三,是「安全性」。只要你透過任何協定,把工具跟資料對外揭露,就一定要有穩固的安全措施。一個像樣的 MCP 實作,必須納入兩件事:身分驗證,英文是 authentication,確認「你是誰」;以及授權,英文是 authorization,決定「你能做什麼」。你要能控制哪些用戶端可以連哪些伺服器,以及它們被允許執行哪些動作。

第四,是「實作」這件事本身。MCP 雖然是開放標準,但真要自己實作起來,可能相當複雜。好消息是,供應商已經開始幫你簡化了。比方說,像 Anthropic 這樣的模型供應商,或是 FastMCP 這樣的框架,都提供了 SDK,也就是軟體開發套件,它們把大量重複性的「樣板程式碼」,英文叫 boilerplate,給抽象掉了,讓你更輕鬆就能建立並連接 MCP 的用戶端跟伺服器。

第五,是「錯誤處理」。一套完備的錯誤處理策略至關重要。協定必須定義清楚:當出狀況的時候,比方說工具執行失敗、伺服器連不上、請求本身無效,這些錯誤要怎麼回傳給語言模型。為什麼?因為模型要能「理解這次為什麼失敗」,它才有機會去嘗試別的替代做法,而不是傻在那裡。

第六,是「本機跟遠端伺服器」的選擇。MCP 伺服器可以部署在跟代理同一台機器上,這叫本機;也可以部署在另一台遠端的機器上。怎麼選?如果你處理的是敏感資料,可能會為了速度跟安全,選擇本機伺服器,資料不出門。而遠端伺服器的架構,好處是可以讓整個組織共享、可擴展地去存取那些通用的工具。

第七,是「隨選跟批次」的取捨。MCP 兩種都支援。隨選,指的是即時、互動式的工作階段,比方說一個對話式代理,使用者問一句它馬上要用工具回一句。批次,指的是更大規模的成批處理,比方說一條資料分析管線,一次處理一大堆紀錄。要選哪一種,取決於你的應用情境。

第八,是「傳輸機制」,英文叫 transport。協定也定義了底層實際用來通訊的傳輸層。這裡有兩種場景。對於本機互動,它用的是「透過 STDIO 的 JSON-RPC」。我解釋一下:STDIO 就是標準輸入跟標準輸出,是程式之間最基本的溝通管道;JSON-RPC 則是一種用 JSON 格式來做遠端呼叫的方式。兩者搭起來,可以達成高效率的「行程間通訊」,也就是同一台機器上不同程式之間的對話。而對於遠端連線,它用的是對網路比較友善的協定,像是 Streamable HTTP,以及伺服器推送事件,英文是 Server-Sent Events,縮寫 SSE,這是一種讓伺服器能主動把資料推送給用戶端的技術。這些都是為了達成持久而且高效率的主從式通訊。

談完考量,我們把整個 MCP 的「角色」跟「流程」完整走一遍,這是掌握進階代理行為的核心。MCP 用主從式模型來把資訊流標準化,裡面有四個角色。第一個是語言模型,也就是 LLM,它是核心智慧,負責處理使用者請求、擬定計畫,並決定「什麼時候該去拿外部資訊或執行動作」。第二個是 MCP 用戶端,它是包在語言模型外層的一個應用程式或封裝層,扮演中介者:把模型的「意圖」翻譯成一個符合 MCP 標準的正式請求,並且負責去發現、連接、溝通那些伺服器。第三個是 MCP 伺服器,它是通往外部世界的閘道,向有授權的用戶端揭露一組工具、資源跟提示;每一個伺服器通常專門負責某個領域,比方說連到公司內部資料庫、連到一個電子郵件服務,或連到一個公開的 API。第四個角色是「選擇性的第三方服務」,英文常簡寫成 3P。它代表的是 MCP 伺服器在背後管理、實際存在的那個外部工具、應用程式或資料來源,是真正執行動作的最終端點,比方說去查一個專有資料庫、去跟某個 SaaS 平台互動,或去呼叫一個公開的天氣 API。

那這四個角色一次互動是怎麼跑的?我帶你走完五個步驟,你閉著眼睛想像一個「代理幫你寄信」的場景就行了。第一步叫「發現」。用戶端代表模型,去問伺服器:「你有哪些能力?」伺服器回給它一份清單,英文叫 manifest,上面列出可用的工具,比方說一個叫「send email」的寄信工具;可用的資源,比方說一個叫「customer database」的客戶資料庫;還有可用的提示。第二步叫「請求擬定」。模型判斷它需要用到剛剛發現的某個工具,比方說它決定要寄一封信。於是它擬出一個請求,指定要用哪個工具,也就是那個寄信工具,並且填好必要的參數:收件人是誰、主旨是什麼、內文是什麼。第三步叫「用戶端通訊」。用戶端接過模型擬好的這個請求,把它變成一個標準化的呼叫,傳給對應的那個 MCP 伺服器。第四步叫「伺服器執行」。伺服器收到請求後,先驗證用戶端的身分、再驗證請求是否合法,然後它去跟底層的軟體對接,實際執行動作,比方說呼叫某個郵件 API 的「send」函式,把信真的寄出去。第五步叫「回應與情境更新」。執行完之後,伺服器把一個標準化的回應傳回給用戶端,告訴它這次動作成功還是失敗,並附上相關的輸出,比方說一封已寄出郵件的確認編號。用戶端再把這個結果交回給模型,更新它的情境,於是模型就能接著推進任務的下一步。你看,這五步走下來,一個原本只會講話的模型,就真的把一封信寄出去了。

理解了流程,我們來看 MCP 在實務上能做什麼。原文整理了九個關鍵的使用案例,我一個一個講給你聽,你會發現它的應用面非常廣。

第一個是「資料庫整合」。MCP 讓模型跟代理能無縫地存取並操作資料庫裡的結構化資料。舉個例子,用一個叫「MCP Toolbox for Databases」的工具,代理可以去查 Google BigQuery 的資料集,拿即時資訊、生成報告,甚至更新紀錄,而且這一切都是用自然語言指令驅動的,你用講的它就去做了。

第二個是「生成式媒體協調」。MCP 讓代理能跟先進的生成式媒體服務整合。透過一個叫「MCP Tools for Genmedia Services」的工具,代理可以協調一整套工作流程,動用到 Google 的多個服務:Imagen 負責影像生成、Veo 負責影片製作、Chirp 3 HD 負責擬真的語音、Lyria 負責音樂創作。這讓 AI 應用能進行動態的內容創作。

第三個是「外部 API 互動」。MCP 提供一種標準化的方式,讓模型去呼叫任何外部 API 並接收回應。這意味著代理可以去抓即時天氣、拉股價、寄信,或跟 CRM 客戶關係管理系統互動,把能力擴展到遠超它核心語言模型的範圍。

第四個是「基於推理的資訊擷取」。這個很有意思,它借助語言模型強大的推理能力,讓擷取資訊的表現超越傳統的搜尋系統。傳統搜尋工具,會回你一整份文件,要你自己去翻;而代理不一樣,它會去分析文字,然後直接擷取出「那一個精確的條款、那一筆資料、那一句陳述」,正好回答你那個複雜的問題。

第五個是「客製化工具開發」。開發者可以自己打造客製工具,再透過 MCP 伺服器把它揭露出去,比方說用 FastMCP。這讓你那些專門的內部函式或專有系統,能用一種標準化、好消費的格式提供給模型跟其他代理使用,而且完全不必去動模型本身。

第六個是「標準化的模型對應用程式通訊」。MCP 確保模型跟它互動的應用程式之間,有一個一致的通訊層。這降低了整合的負擔、促進了不同供應商跟宿主應用之間的互通,也簡化了複雜代理系統的開發。

第七個是「複雜工作流程協調」。把各種由 MCP 揭露的工具跟資料來源組合起來,代理就能協調高度複雜的多步驟流程。舉個很生動的例子:代理可以先從資料庫撈出客戶資料,接著生成一張個人化的行銷影像,然後草擬一封量身打造的電子郵件,最後把它寄出去,而這一連串動作,全都是透過跟不同的 MCP 服務互動完成的。

第八個是「物聯網裝置控制」。物聯網的英文是 IoT。MCP 可以讓模型去跟 IoT 裝置互動,代理能對智慧家居設備、工業感測器或機器人發送指令,實現用自然語言去控制跟自動化實體系統。

第九個是「金融服務自動化」。在金融領域,MCP 可以讓模型跟各種金融資料來源、交易平台或合規系統互動。代理能分析市場資料、執行交易、生成個人化的理財建議,或把法規申報自動化,而這一切都在安全且標準化的通訊下進行。

把這九個案例濃縮成一句話:MCP 讓代理能從資料庫、API 跟網路資源去拿即時資訊,也讓代理能執行各種動作,像是寄信、更新紀錄、控制裝置,還能整合並處理來自各方來源的資料來完成複雜任務,甚至支援媒體生成的工具。

好,概念講到這裡,我們進入動手實作的環節。原文用的是 ADK,也就是 Agent Development Kit,代理開發套件,這是 Google 的一套開發框架。第一個範例,是讓一個 ADK 代理去連接一個「本機的檔案系統 MCP 伺服器」,好讓代理能讀寫你電腦上的檔案。我不會逐行唸程式碼,我會描述它在做什麼。

這個設定的核心,是建立一個叫 agent.py 的檔案,在裡面組裝出一個代理。程式做的第一件事,是算出一個「目標資料夾」的絕對路徑。它刻意把這個資料夾放在跟代理腳本同一個目錄底下,取名叫「mcp managed files」,這樣示範的時候開箱即用;它還會先確認這個資料夾存在,不存在就先建好。為什麼強調「絕對路徑」?因為這個資料夾,會成為代理做所有檔案操作的「根目錄」,代理只能在這個範圍裡活動,這也是一種安全邊界。接著它建立一個語言模型代理,用的模型是 Gemini 2.0 Flash,給它一個名字叫「filesystem assistant agent」,並寫了一段指令告訴它:你的工作是幫使用者管理檔案,你可以列出檔案、讀檔、寫檔,而且你正在哪個目錄裡操作。

這段程式最關鍵的地方,在於它怎麼把 MCP 工具掛上去。它在代理的工具清單裡,放進一個叫 MCPToolset 的東西,而這個 toolset 的連線參數,用的是 StdioServerParameters,也就是我們前面講過的、透過標準輸入輸出溝通的本機方式。它指定的執行指令是 npx,後面接的引數,是去執行一個社群做好的套件,叫「server filesystem」,也就是現成的檔案系統 MCP 伺服器,並把剛剛那個目標資料夾的絕對路徑交給它。程式裡還順帶提到一個可選的功能,叫 tool filter,工具過濾器:如果你想限制代理只能做某些事,比方說只准它「列目錄」跟「讀檔」,不准它寫檔,你就可以在這裡把允許的工具列出來。這個小功能,呼應了我們前面講的安全考量。

這裡補充一個你可能會卡住的名詞:npx。它的全名是 Node Package Execute,它隨附在 npm,也就是 Node 套件管理工具,5.2.0 版以後的版本裡。npx 能讓你「直接從套件登錄處執行一個 Node.js 套件」,而不必先把它全域安裝起來。它扮演的是套件執行器的角色,而很多社群開發的 MCP 伺服器,正好都是以 Node.js 套件的形式發布的,所以你會常常看到用 npx 來啟動它們。

另外,要讓 ADK 能把你的代理認出來,你還得在同一個目錄裡建一個叫 init,寫法是雙底線 init 雙底線、副檔名 .py 的檔案。它的內容很簡單,就是把同目錄的 agent 模組引入進來。它的作用,是確保這個資料夾被當成一個 ADK 能發現的 Python 套件。

原文也提醒你,啟動伺服器的指令不是只有 npx 一種。如果你的 MCP 伺服器是一支 Python 腳本,你可以把指令換成 python3,然後在引數裡指向你那支伺服器腳本的路徑,還可以透過一個 env 環境變數區塊,把像是服務帳號路徑、雲端硬碟資料夾編號這類設定傳進去。另外還有一個工具叫 UVX。在 Python 的脈絡下,UVX 指的是一個利用 uv 這個工具,在「暫時的、隔離的」Python 環境裡執行指令的命令列工具。它的好處跟 npx 很像:讓你能執行 Python 工具跟套件,而不必把它們全域安裝、也不必裝進你專案的環境裡。原文舉的例子,就是用 uvx 去跑一個叫「mcp google sheets」的套件,連到 Google 試算表。

伺服器建好、代理設好之後,怎麼把它跑起來看效果?你在終端機裡,切換到代理資料夾的「上層目錄」,然後執行一個指令叫 adk web。這會啟動一個網頁版的操作介面。介面在瀏覽器裡載入後,你從代理選單裡選那個「filesystem assistant agent」,接著就可以用自然語言去測它了,比方說你打「給我看這個資料夾的內容」,或「幫我讀 sample.txt 這個檔」,或「another file.md 裡面寫了什麼」,代理就會透過 MCP 去操作檔案系統,把答案回給你。

講完「使用現成的 MCP 伺服器」,我們來看第二個方向:「自己做一個 MCP 伺服器」。這裡用的工具叫 FastMCP。FastMCP 是一個高階的 Python 框架,專門用來簡化 MCP 伺服器的開發。它提供一層抽象,把協定本身那些複雜的細節都藏起來,讓你只要專注在自己的核心邏輯上。

FastMCP 最方便的一點,是它讓你用簡單的 Python「裝飾器」,英文叫 decorator,就能快速定義工具、資源跟提示。裝飾器你可以想成是一個貼在函式上方的標籤,貼上去,這個函式就被賦予了新身分。而 FastMCP 還有一個很聰明的能力,叫「自動生成綱要」,綱要的英文是 schema。它會自動去讀你那個 Python 函式的「簽名」、它的「型別提示」,還有它的「說明字串」,然後根據這些資訊,自動幫你建構出 AI 模型需要的那份介面規格。我解釋一下這幾個詞:函式簽名,指的是這個函式叫什麼名字、收哪些參數;型別提示,指的是你標註每個參數跟回傳值是什麼資料型別,比方說字串、數字;說明字串,指的是寫在函式裡那段描述它用途的文字。FastMCP 把這些統統讀進來,自動產生規格,於是你幾乎不用手動設定,人為的錯誤也跟著變少。除了基本的工具建立,FastMCP 還支援更進階的架構模式,像是「伺服器組合」跟「代理轉送」,讓你能模組化地開發複雜的多元件系統,也能把既有服務無縫整合進一個 AI 可存取的框架裡,而且它還針對高效率、分散式、可擴展的 AI 應用做了最佳化。

我們來看它一個最基本的範例:做一個會「問候人」的工具。這支伺服器腳本做的事情,描述起來很簡單。它先初始化一個 FastMCP 伺服器物件,然後定義一個叫 greet 的函式,在這個函式上方貼一個 tool 裝飾器。這個裝飾器一貼,就把這個普通的 Python 函式,註冊成了一個 MCP 工具。這個 greet 函式,輸入是一個人的名字,輸出是一句個人化的問候語,比方說你給它「John」,它就回你一句「Hello, John,很高興認識你」。特別要注意的是,函式裡那段說明字串,會直接變成「這個工具給語言模型看的描述」,也就是模型靠這段文字來理解「這個工具是幹嘛的、要餵它什麼、它會吐什麼回來」。腳本的最後一段,是讓這支腳本被直接執行時,就啟動伺服器,並指定用 HTTP 這種傳輸方式,監聽在本機位址、連接埠 8000 上。換句話說,當你執行這支腳本,它就會在 localhost 的 8000 埠上待命,把 greet 這個函式變成一項網路服務對外提供,而且它會一直跑,直到你手動把它停掉為止。

伺服器有了,接下來怎麼讓一個 ADK 代理去「當用戶端」、來用這個伺服器?這需要設定一個叫 HttpServerParameters 的東西,顧名思義,它是用 HTTP 來連線的參數,你要把 FastMCP 伺服器的網路位址填進去,通常就是 localhost 的 8000 埠。這個範例的代理腳本做的事情是這樣:它一樣建立一個語言模型代理,用 Gemini 模型,取名叫「fastmcp greeter agent」,給它一道指令,要它扮演一個友善的助理,專門依照人們的名字去問候他們,並明確叫它「使用 greet 這個工具」。然後它在工具清單裡放進一個 MCPToolset,連線參數用的就是剛剛說的 HttpServerParameters,指向那個 8000 埠的伺服器。這裡它也用了 tool filter 工具過濾器,把代理能用的工具限定在 greet 這一個上面。整段程式的用意,就是把系統的「用戶端那一半」設好,建立出一個聰明的代理:它知道自己的目標是問候人,也清楚知道該用哪個外部工具來達成。它的運作邏輯是,當你給它一句像「Greet John Doe」這樣的請求,代理內嵌的語言模型會辨識出「透過 MCP 有個 greet 工具可用」,於是用「John Doe」這個引數去呼叫它,再把伺服器回來的結果吐給你。這個範例,完整示範了怎麼把一個你自訂、並透過 MCP 揭露的工具,跟一個 ADK 代理整合起來。

實際要跑這個雙邊範例,順序是這樣:你先開一個終端機,執行那支 FastMCP 伺服器腳本,讓它在 8000 埠上跑起來;然後再開另一個終端機,切到代理的上層目錄,執行 adk web;介面載入後,選「fastmcp greeter agent」這個代理,輸入「Greet John Doe」,代理就會去呼叫你伺服器上的 greet 工具,產生回應。對了,別忘了在這個代理資料夾裡,同樣要放一個 init 檔案,確保它被當成 ADK 能發現的套件,這跟前一個範例是一樣的道理。

兩個範例都走完了,我們把整章的脈絡用幾句話再串一次,幫你把重點沉澱下來。先講「是什麼」這個層次:語言模型要當好代理,就必須超越單純的文字生成,需要跟外部環境互動的能力。如果沒有一套標準化的溝通方法,那模型跟每一個外部工具或資料來源之間的整合,都會變成一件客製化、複雜又無法重用的苦差事,這種臨時拼湊的做法,英文叫 ad-hoc,會嚴重阻礙可擴展性。再講「為什麼」要用 MCP:它提供一套標準化的解法,當語言模型跟外部系統之間的萬用介面,用主從式模型運作,讓伺服器向任何符合規範的用戶端,揭露工具、資料資源跟互動式提示;由模型驅動的應用扮演用戶端,以可預測的方式動態發現資源並使用,於是孕育出一個由可互通、可重用元件構成的生態系。

那什麼時候該用、什麼時候不必用?給你一個經驗法則。當你在建構複雜的、可擴展的、或企業級的代理系統,而且這個系統需要跟「一大堆多樣且不斷演進」的外部工具、資料來源跟 API 互動時,就該用 MCP;尤其當「不同模型跟工具之間的互通性」是你的優先考量,或當你希望代理能「不重新部署就動態學會新能力」時,MCP 就是理想選擇。反過來,如果你的應用很簡單,要用的函式數量固定、有限、而且事先就定義好了,那直接用工具函式呼叫,可能就綽綽有餘,不必動用 MCP 這麼重的標準。

這一章原本還附了一張示意圖,我用講的幫你描述一下它要傳達的訊息:使用者透過提示,跟一個「扮演 MCP 用戶端的代理」互動;這個代理再經由 MCP 伺服器,連接到外部的各種資源,包括工具、API、舊式服務、資料服務,還有那些把舊系統包起來的包裝層;最後代理把處理過的輸出,回傳給使用者。你只要記得這個「使用者到代理、代理透過伺服器到外部世界、再把結果繞回使用者」的環狀流動就好。

最後我幫你把幾個一定要帶走的重點收束一下。第一,MCP 是一套開放標準,促進語言模型跟外部應用、資料來源、工具之間的標準化溝通。第二,它採用主從式架構,定義了怎麼揭露、怎麼消費資源、提示跟工具這三種東西。第三,ADK 這套開發套件,同時支援兩個方向:既能去運用既有的 MCP 伺服器,也能反過來透過 MCP 伺服器把自己的工具揭露出去。第四,FastMCP 大幅簡化了 MCP 伺服器的開發跟管理,尤其適合用來揭露你用 Python 寫的工具。第五,前面提到的那套 Genmedia 媒體工具,讓代理能整合 Google Cloud 的生成式媒體能力,也就是 Imagen、Veo、Chirp 3 HD 跟 Lyria。第六,也是最根本的一點:MCP 讓語言模型跟代理,能跟真實世界的系統互動、存取動態資訊,並執行超越文字生成的各種動作。

總結來說,模型情境協定就像是 AI 代理通往真實世界的那個萬用轉接頭。它用一套開放、標準化的主從式架構,讓語言模型得以存取資源、運用提示、執行動作,從跟資料庫互動、協調生成式媒體,到控制物聯網裝置、自動化金融服務,範圍極廣。而本章的兩個動手範例,一個連到現成的檔案系統伺服器,一個用 FastMCP 自己做一個問候伺服器,都示範了它跟 ADK 開發套件怎麼整合。掌握了 MCP,你手上的代理,能耐就能真正超越基本的語言能力,開始替你在真實世界裡做事。
