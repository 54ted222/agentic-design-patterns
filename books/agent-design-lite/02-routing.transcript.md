這一章我們要談的,是讓代理系統懂得「分流」的關鍵技巧,也就是路由,英文叫 Routing。接下來我會先帶你弄懂為什麼需要路由,再說明路由到底在做什麼、有哪四種常見的做法,接著看它在真實世界裡可以用在哪些地方,然後我會用口語帶你走過兩個實際的程式範例,一個用 LangChain,一個用 Google 的代理開發套件,最後再把整章的脈絡幫你串一次。準備好了,我們就開始。

我們先從「為什麼需要路由」說起。在上一章我們談過提示鏈,英文是 Prompt Chaining,它的精神是把一連串步驟一個接一個串起來,像生產線一樣,前一步的輸出餵給下一步。這種做法在處理確定性、而且是直線型的工作流程時非常好用,因為流程是固定的,你很清楚每一步會發生什麼。但問題來了:真實世界的代理系統,英文叫 agentic system,往往沒有那麼乖。它常常得看當下的環境狀態、看使用者到底輸入了什麼、或是看前一步算出來的結果,然後在好幾種可能的行動之間做選擇。這種「要看情況才能決定下一步」的能力,固定的直線流程是給不了的。而能夠提供這種動態決策能力的,就是路由。

那路由到底在做什麼?簡單講,路由就是替代理引入「條件邏輯」,英文是 conditional logic。有了條件邏輯,系統就不再只會走那一條寫死的路徑,而是能夠先動態地評估眼前的條件,再從多個可能的後續行動裡頭挑一個出來執行。這樣一來,代理的行為就變得更靈活,也更懂得看情境,而不是不管三七二十一都走同一條路。

讓我用一個生活化的例子幫你把這個概念落地。想像一個客服查詢的代理,使用者各式各樣的問題都會丟進來。一個有路由能力的客服代理,會先做一件事,就是分析使用者的查詢,判斷他這次到底想幹嘛,也就是判定意圖。判定完之後,它就會依照意圖把這通查詢導向不同的地方。比方說,如果使用者想問的是「我的訂單到哪了」,也就是查詢訂單狀態,它就把問題交給一個專門跟訂單資料庫打交道的子代理,或是一串專門處理訂單的工具鏈。如果使用者想問的是產品資訊,它就導向一個負責去搜尋產品型錄的子代理。如果是技術支援的問題,它就導向疑難排解的指南,真的搞不定就升級交給真人客服。那萬一意圖根本不清楚呢?它也有對策,就是導向一個專門負責「把問題問清楚」的子代理,反問使用者來釐清需求。你看,同樣一個入口,因為有路由,就能把不同的人帶往最合適的窗口,這就是路由的價值。

接下來談一個比較有料的部分:路由具體要怎麼實作?這個機制的核心,其實就是一段「執行評估、然後導引流程」的程式邏輯,而常見的做法有四種,我一個一個說給你聽。

第一種,是以大型語言模型為基礎的路由。大型語言模型的英文是 Large Language Model,簡稱 LLM,後面我就用 LLM 來稱呼它。這種做法是直接給 LLM 一段提示,要它分析輸入,然後吐出一個識別字,告訴你該往哪去。舉個具體的例子,你可以這樣指示它:「請分析這則查詢,而且只能輸出底下四個類別的其中一個,分別是訂單狀態、產品資訊、技術支援、或是其他。」LLM 回你一個類別字串,系統就照著這個字串去導引後續流程。它的好處是很懂語意、很有彈性。

第二種,是以嵌入為基礎的路由。嵌入的英文是 embedding,更完整地說是向量嵌入,vector embedding。這個觀念在後面講檢索增強生成,也就是 RAG 那一章會更詳細。它的做法是,把使用者的查詢轉換成一串數字向量,同時你也事先替每一條路由準備好一個代表它的向量,然後拿查詢的向量去跟每一條路由的向量比相似度,哪一條最像,就導向哪一條。這種方式特別適合所謂的語意路由,英文叫 semantic routing,也就是依照「意思」來決定去向,而不是死板地比對關鍵字。比方說使用者打「我的包裹卡住了」,就算他沒講「訂單」兩個字,語意上它跟訂單狀態最接近,系統照樣能正確分流。

第三種,是以規則為基礎的路由。這個最直覺,就是你事先寫好一堆規則,像是 if-else 這種判斷,或是 switch case 這種多重分支,依照關鍵字、特定模式、或是結構化的資料來判斷該走哪條路。它的優點是比 LLM 路由更快,而且結果是確定的、可預測的,你寫什麼規則它就照什麼跑。但缺點也很明顯,對於比較細膩、或是你沒預料到的新奇輸入,它就會顯得僵硬、轉不過彎。

第四種,是以機器學習模型為基礎的路由。這裡用的是一個經過監督式微調的判別式模型,監督式微調英文是 supervised fine-tuning,判別式模型講白話就是一個分類器。它的路由邏輯不是寫在提示裡,而是被訓練進模型的權重裡。這裡有一個跟第一種 LLM 路由很容易混淆、但很關鍵的差別,我特別講清楚:這種做法在真正做決策的那一刻,靠的不是一個邊推論邊跑提示的生成式模型,而是這個已經訓練好的分類器直接給答案。LLM 在這裡頂多用在前置作業,例如幫忙生成一些合成的訓練資料來訓練這個分類器,但它不參與即時的決策。換句話說,前面三種裡的 LLM 路由是「現場問 LLM 該怎麼分」,而這一種是「事先訓練好一個專職分類器,現場直接判」。

講完四種做法,還有一個觀念要補上:路由可以放在代理運作週期的很多個不同位置,不是只能放在最前面。它可以在一開始就先替主要任務做分類;也可以放在處理鏈的中間,根據前面的結果判定接下來該怎麼走;甚至可以放在某個子程序裡,負責挑出當下最合適的那個工具。在工具層面,現在已經有不少現成的框架幫你管理這類條件邏輯,像是 LangChain、LangGraph,還有 Google 的代理開發套件,英文全名是 Agent Development Kit,簡稱 ADK。其中 LangGraph 特別值得一提,它是用一種「以狀態為基礎的圖」來組織流程,所以特別適合那種「我下一步怎麼決定,要取決於系統一路累積下來的狀態」的複雜情境。

我們接著看路由在實務上能用在哪些地方。路由是設計「適應性代理」的關鍵控制機制,它就靠這一層條件邏輯,讓系統能因應多變的輸入跟內部狀態,動態地改變執行路徑。它的用處橫跨很多領域,我挑三個有代表性的講。

第一個領域是人機互動,像是虛擬助理、AI 家教這類。在這裡,路由負責解讀使用者的意圖,判定接下來該做什麼:可能是去呼叫某個特定的檢索工具,可能是判斷這題太難要升級交給真人,也可能是替學生挑出下一個該學的學習模組。有了它,系統就不再只會跟你一問一答地走直線對話,而是能依情況轉彎。

第二個領域是資料與文件的處理管線。想像每天有大量的郵件、客服工單、或是 API 傳進來的資料載荷,英文叫 payload,一股腦湧進系統。路由可以依照內容本身、依照中繼資料,也就是 metadata,或是依照格式,把這些東西先分類,再分派到對應的工作流程。比方說,這封是潛在客戶要匯入、那份要做特定格式的轉換、這件是緊急問題得馬上升級,各走各的處理線。

第三個領域是多工具或多代理的複雜系統。在這種系統裡,路由扮演的是一個高階分派器的角色,英文叫 high-level dispatcher。舉例來說,一個研究型系統可以用它把任務分派出去,該搜尋的交給搜尋代理,該摘要的交給摘要代理,該分析的交給分析代理。又或者一個 AI 編碼助理,可以用路由先辨識這是哪一種程式語言、使用者的意圖是除錯、是要解釋程式、還是要做語言翻譯,然後再把任務交給正確的工具去處理。

把這些串起來,你會發現路由真正帶來的轉變是這樣的:它把代理從一個「照著預定順序乖乖執行的靜態執行者」,升級成一個「能在變動條件下,自己選出最有效方法的動態系統」。這句話請你記住,它是整章的精神所在。

接下來我們進入動手實作的部分。我會帶你看第一個範例,它用的是 LangChain 搭配 Google 的生成式 AI 服務。這個範例要建立一個簡易的類代理系統,核心是一個叫做「協調者」的角色,英文是 coordinator。這個協調者會依照請求的意圖,把請求路由到不同的模擬子代理去處理,意圖分成三類:訂位、一般資訊查詢、還有不明確。這正好展示了多代理架構裡很常見的「委派」模式。

實作上,第一步當然是安裝需要的函式庫,並且替你選用的模型設定好 API 金鑰,這個 API 指的就是應用程式介面,金鑰就是讓你的程式有權限去呼叫模型服務的通行證。範例裡用的是 Google 的 Gemini 模型,但程式也支援換成 OpenAI 或 Anthropic 的模型。

我用口語幫你描述這段程式在做什麼,你不用去記任何符號。它的開頭先初始化一個語言模型物件,選的是 Gemini 的一個快速版本,並且把溫度參數設成零。這個溫度設成零的用意很重要:溫度代表回答的隨機性,設成零就是要模型盡量穩定、每次都給一致的答案。因為這裡 LLM 的工作是分類,我們要的是可靠、可重複的判斷,而不是天馬行空的創意,所以要把隨機性壓到最低。如果初始化失敗,它也有錯誤處理,會印出錯誤訊息,讓你知道是不是金鑰沒設好。

再來,程式定義了三個模擬的處理常式,你可以把它們想成三個假裝在工作的小幫手。第一個是訂位處理常式,收到請求時會印出「委派給訂位處理常式」,然後回傳一句模擬已完成訂位的訊息。第二個是資訊處理常式,收到請求會印出「委派給資訊處理常式」,回傳一句模擬已擷取資訊的訊息。第三個是處理不明確請求的常式,它會回一句「協調者無法委派這個請求,請進一步釐清」。注意這三個都只是模擬,重點不在它們真的做了什麼,而在於「請求最後被正確地送到了哪一個」。

整個範例真正的靈魂,是中間那段「協調者路由鏈」。它的做法是用一個聊天提示樣板,給 LLM 一段很明確的指示,大意是:請你分析使用者的請求,判斷該由哪個處理常式來處理;如果跟訂機票或訂飯店有關,就輸出 booker 這個字;如果是其他一般資訊問題,就輸出 info;如果不明確或不屬於前兩類,就輸出 unclear;而且強調一句,你只能輸出這三個字當中的一個。然後把這段提示、LLM、跟一個負責把模型輸出轉成純字串的解析器,串成一條鏈。這條鏈的輸入是使用者請求,輸出就是 booker、info、unclear 三者之一,這個輸出我們把它叫做「決策」。

光有決策還不夠,還要根據決策把請求真的送出去,這就靠一個叫做「可執行分支」的元件,在 LangChain 裡叫 RunnableBranch。你可以把它想成一個交通警察:它去看那個決策字串是什麼。如果是 booker,就把原始的請求交給訂位處理常式;如果是 info,就交給資訊處理常式;如果都不是,就走預設那條路,交給不明確處理常式。這裡有一個很實務的小細節值得一提,程式在比對決策字串之前,特地做了去除前後空白的處理。為什麼?因為 LLM 有時候會在回答前後夾帶看不見的空白字元,如果不清掉,字串比對就會對不上,明明答對了卻被判成不符合。這種小地方,正是把範例真正跑起來時會踩到的坑。

最後,程式把這兩塊組裝成一個完整的協調者代理:先讓路由鏈算出決策,同時把原始請求原封不動往下帶,兩者一起交給那個分支元件去委派,委派完再把最終的輸出擷取出來。為了示範,它在主程式裡丟了三個請求進去:第一個是「幫我訂一張飛往倫敦的機票」,這應該被導向訂位;第二個是「義大利的首都是哪裡」,這應該被導向資訊;第三個是「跟我說說量子物理」,這比較模糊,會落到不明確那條。整段程式合起來,就完整模擬了一個「中央協調者依照意圖把任務委派出去」的基本多代理框架。

看完 LangChain,我們再看第二個範例,它用的是 Google 的代理開發套件,也就是 ADK,而且它思考路由的方式跟前一個範例很不一樣,值得對照著聽。前一個範例是以運算圖的方式在組流程,而 ADK 這套範式,通常是透過定義一組離散的「工具」來實作路由,工具的英文是 tools。也就是說,你把各種能力做成一個一個工具,而「到底要選哪個工具」這件事,是交給框架內部的邏輯去運用底層模型,自動把使用者意圖對應到正確的處理常式。你不用自己寫那一大串分支判斷,框架幫你決定。

這個範例同樣建立一個協調者代理,底下帶兩個專門的子代理:一個叫 Booker,負責訂位;一個叫 Info,負責一般資訊。每個子代理都配備自己專屬的工具來模擬處理。我一樣用口語帶你走過它的運作。

程式先定義了三個工具函式,跟前面類似,分別模擬訂位、資訊、跟處理不明確請求的動作,收到請求就印一行字、回一句模擬結果。比較值得注意的是,這裡多了一個動作,就是把訂位函式跟資訊函式分別「包裝成工具物件」。在 ADK 裡,你得先把一個普通函式包成工具,框架才認得它、才知道這是一個可以被選用的能力。

接著定義兩個子代理。訂位代理取名 Booker,配上訂位工具,並且給它一段描述,說明它是一個專門透過呼叫訂位工具來處理所有機票與飯店訂位的代理。資訊代理取名 Info,配上資訊工具,描述它是一個專門提供一般資訊、回答使用者問題的代理。這裡的「描述」不是寫給人看的註解而已,它其實是給框架判斷用的線索,框架會根據這些描述來決定該把請求交給誰,所以描述寫得清不清楚,直接影響路由準不準。

然後是重頭戲,父代理,也就是協調者,取名 Coordinator。它被賦予一段很明確的委派指令,大意是:你是主要的協調者,你唯一的任務就是分析進來的使用者請求,然後把它委派給適當的專門代理,你絕對不要自己直接回答使用者;只要是跟訂機票或飯店有關的,就委派給 Booker;其他所有一般資訊問題,就委派給 Info。這段指令的精神,跟前一個範例那段「只輸出一個字」的分類提示其實是同一回事,都是在告訴模型怎麼分流,只是表達方式不同。而最關鍵的一行設定是:這個協調者掛上了它的子代理清單。在 ADK 裡,只要你替一個代理定義了子代理,框架就會自動啟用一種叫做 Auto-Flow 的機制,也就是由 LLM 驅動的自動委派。換句話說,你不必像前一個範例那樣手寫分支,框架看到你有子代理,就自動幫你把請求路由到對的子代理去,這是 ADK 跟前面圖式做法在體感上最大的差別。

至於執行的部分,程式建立了一個記憶體內的執行器,英文是 InMemoryRunner,先替使用者開一個工作階段,也就是 session,然後把請求送進去跑,執行器會一路產出一連串事件,程式再從這些事件裡把最終回應的文字擷取出來。這裡有個小提醒,範例裡那個處理不明確請求的常式雖然定義了,但目前的執行邏輯在委派失敗時並沒有真的去用到它,它比較像是一個預留的後備。為了示範,主程式丟了四個請求:在巴黎訂飯店、世界最高的山、講個冷知識、查下個月飛東京的航班,前者跟最後一個應該被導向訂位,中間兩個應該被導向資訊。

兩個範例都看完了,我們把它們放在一起做個重點整理,我用「是什麼、為什麼、什麼時候用」三個角度幫你收斂。

先講「是什麼」。代理系統常常得回應各式各樣、沒辦法用單一直線流程處理的輸入。單純的循序工作流程沒辦法依情境做決策,也沒辦法替特定任務挑出正確的工具或子流程,所以它就會變得僵化,很難駕馭真實世界那種又複雜又多變的請求。這就是路由要解決的痛點。

再講「為什麼」。路由替代理引入了條件邏輯,讓系統能先分析查詢、判定意圖,再把控制流程動態地導向最適切的工具、函式或子代理。而做這個決策,可以靠提示 LLM、可以靠事先寫好的規則,也可以靠以嵌入為基礎的語意相似度,這正好呼應我們前面講的那幾種實作方式。它的效果,是把一條靜態寫死的執行路徑,轉成一個靈活、有情境感知、能挑出最佳行動的工作流程。

最後是「經驗法則」,也就是什麼時候該用路由。原則很簡單:當你的代理必須依照使用者輸入、或是依照當前狀態,在多個工作流程、多個工具、或多個子代理之間做抉擇時,就用路由。它對那種需要「分流」或「分類請求」的應用特別關鍵。分流的英文是 triage,就像醫院急診先幫病人分輕重緩急那樣。一個最典型的例子,就是能夠區分業務諮詢、技術支援、跟帳戶管理的客服機器人,進來的問題先分好類,再送到對的隊伍。

這一章原本附了一張圖,我用講的讓你在腦中把它畫出來。那張圖呈現的就是路由器模式,英文是 router pattern,畫面中央是一個 LLM 扮演路由器的角色,它從左邊接收進來的請求,經過判斷之後,把流程分岔導向右邊好幾條不同的下游路徑。你只要記住那個「一個入口、中間一個判斷者、分岔出多條去路」的形狀,就抓到這張圖的精神了。

最後我們把整章的脈絡再串一次做個收束。路由是建構動態、有回應能力的代理系統的關鍵一步,它讓代理能夠針對「該怎麼處理資訊、怎麼回應輸入、要不要動用工具或子代理」做出明智的決策,應用範圍從客服聊天機器人,一路延伸到複雜的資料處理管線。而它的決策邏輯,我們講過,可以用 LLM、用規則、或用嵌入相似度來實作,藉此處理多樣的輸入並調整行為,讓代理徹底擺脫只能走直線的限制。

本章那兩個範例,剛好展現了兩條都很有效、但風格不同的路:LangGraph 這種以圖為基礎的結構,提供的是視覺化、把狀態跟轉換都明確定義出來的方式,特別適合那種錯綜複雜、有很多步驟的工作流程;而 Google 的 ADK 則著重在「定義好一項一項能力,也就是工具」,然後仰賴框架自動把請求路由到適當的工具,對於那種本來就擁有一組明確、離散動作的代理來說,反而更簡單。沒有哪一個一定比較好,而是看你的場景比較像哪一種。能夠把路由這個模式真正掌握住,就是你打造出多功能、又夠穩健的代理應用的關鍵所在。
