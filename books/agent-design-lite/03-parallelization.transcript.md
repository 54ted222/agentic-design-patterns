這一章我們要談的,是讓代理系統跑得更快的一個關鍵設計,叫做平行化,英文是 Parallelization。我會先帶你理解平行化到底在解決什麼問題、它的核心理念是什麼,接著帶你看七種真實世界的應用場景,然後用兩個動手實作的例子,分別是 LangChain 跟 Google 的代理開發套件,把它的運作方式講清楚,最後再幫你把整章的脈絡收束起來。整段我們都用聽的,所以遇到程式碼,我不會去唸符號,而是描述它在做什麼。

我們先從前情提要說起。在前面幾章,我們學過兩個控制流程的模式。第一個叫提示鏈,英文是 Prompt Chaining,它的精神是循序,一步接一步,前一步的輸出餵給後一步。第二個叫路由,英文是 Routing,它的精神是動態切換,根據情況決定走哪一條路徑。這兩個都很有用,但它們有一個共同的特點,就是同一時間只在做一件事。

問題來了。很多代理任務,其實內部包含了好幾個彼此不相干、可以同時做的子任務。如果你硬要它們一個排一個地等,就太浪費時間了。這就是平行化要登場的地方。

那平行化到底是什麼?簡單講,就是同時執行多個元件。這裡的元件可以是好幾次對大型語言模型的呼叫,大型語言模型英文叫 Large Language Model,簡稱 LLM;也可以是好幾次工具的使用;甚至可以是一整個獨立的子代理,英文叫 sub-agent。重點在於,彼此獨立的任務不需要排隊互相等待。所以對於那些可以拆成好幾個獨立部分的任務來說,平行化能大幅縮短整體的執行時間。

它最划算的場景,是當你要跟有延遲的外部服務打交道的時候。延遲英文叫 latency,意思就是你發出請求之後,要等一段時間對方才會回你。比方說你要呼叫外部的應用程式介面,也就是 API,或者要去查資料庫,這些動作本身就慢。如果你能同時把好幾個請求一次發出去,大家一起等、一起回來,而不是發一個等一個,那省下來的時間是非常可觀的。

我用一個很具體的例子讓你體會這個差別。假設我們有一個代理,任務是研究某個主題,然後做摘要。循序的做法是這樣:先去搜尋第一個來源,搜到了再摘要第一個來源,接著去搜尋第二個來源,搜到了再摘要第二個來源,最後把兩份摘要綜合起來。你看,這中間每一步都在等上一步。那平行的做法呢?是同時去搜尋第一個跟第二個來源,兩邊都搜回來之後,再同時摘要這兩份,最後才綜合。注意最後綜合這一步通常還是得循序做,因為它必須等前面兩個平行的步驟都完成,手上有東西了才能整合。

所以平行化的核心理念,你可以記住一句話:找出工作流程裡那些不依賴別人輸出的環節,把它們平行化。一個環節只要不需要等別人的結果,它就有資格跟其他環節同時跑。

那實作上需要什麼?通常你需要一個支援非同步,或者支援多執行緒、多行程的框架。非同步英文叫 asynchronous,意思是程式不必呆呆地卡在原地等一件事做完,而是可以先去推進別的事。好消息是,現在的代理框架大多已經把非同步設計內建好了,你不太需要自己從零造輪子。

接著說幾個主流框架各自怎麼做平行。第一個是 LangChain,它有一套叫做 LangChain 表達式語言的東西,英文縮寫是 LCEL。在這套語言裡,你可以用運算子把好幾個可執行的物件組合起來。可執行物件,英文叫 runnable,你可以把它想成一個能接收輸入、產生輸出的處理單元。其中有一個直線符號代表的是循序串接,意思是把前一個的輸出接到後一個。而要做平行,你就是把多個可執行物件並排組在一起,讓它們同時跑出好幾條分支。第二個框架是 LangGraph,它是用圖的結構來思考的。圖英文叫 graph,你可以把它想像成一張由節點跟連線構成的流程圖。它的巧妙之處在於,單一一次狀態轉移就可以同時觸發好幾個節點,這樣就自然開出了平行的分支。第三個是 Google 的代理開發套件,英文叫 Agent Development Kit,簡稱 ADK。它提供了原生的機制來管理代理的平行執行,讓複雜的多代理系統跑得更有效率、也更容易擴展。總之你可以把平行化記成一句話:它是最佳化複雜代理工作流程效能的關鍵技術。

好,理念講完了,我們來看它在實務上到底能用在哪裡。這裡有七個典型的應用場景,我一個一個帶你過,你會發現它們的共同點都是「有好幾件不相干的事要做」。

第一個是資訊蒐集與研究。想像一個要研究某家公司的代理。它可以同時做這幾件事:搜尋這家公司的新聞、拉取它的股價、檢查社群媒體上有沒有人提到它、再去查公司內部的資料庫。這四件事彼此沒有先後關係,同時做,你就能比一條一條查更快地拿到一個全面的視野。

第二個是資料處理與分析。假設有一個代理要分析顧客的回饋。對著同一批回饋,它可以同時做情緒分析、同時擷取關鍵字、同時做分類、同時找出哪些是緊急問題。一口氣從好幾個面向切進去,很快就能給出多角度的分析。

第三個是跟多個 API 或工具互動。最好懂的例子是旅遊規劃代理。它可以同時去查機票、搜尋飯店有沒有空房、查當地有什麼活動、再找餐廳推薦。本來這些都是要分開查的服務,同時查,使用者就能更快看到一份完整的行程。

第四個是生成有多個元件的內容。比方說一個幫你寫行銷郵件的代理。一封行銷郵件其實是由好幾塊拼起來的:主旨列、內文、配圖、還有最後催使用者行動的那句話,也就是行動呼籲,英文叫 call-to-action。這幾塊可以同時生成,因為它們彼此不依賴,寫主旨的時候不需要先知道內文長怎樣。

第五個是驗證與查核。一個要驗證使用者輸入的代理,可以同時檢查電子郵件格式對不對、電話號碼合不合理、地址核不核得上、有沒有夾帶不雅字眼。這些檢查互不相干,同時跑,就能更快回饋這筆輸入到底有沒有效。

第六個是多模態處理。多模態的意思是同時處理不只一種型態的資料,像文字加影像。假設有一則貼文同時有文字跟圖片,代理可以一邊對文字做情緒分析跟關鍵字分析,另一邊同時對影像做物件辨識跟場景描述,兩邊各做各的,最後再把多模態的洞見整合起來,速度自然快很多。

第七個是 A 比 B 測試,或者說多選項生成。A 比 B 測試英文是 A/B testing,精神是同時準備好幾個版本來比較。舉例來說,一個生成創意文案的代理,可以用稍微不一樣的提示,或者用不一樣的模型,同時幫一篇文章生出三個標題,然後讓你快速比較、挑出最好的那一個。

把這七個場景串起來,你會發現平行化是代理設計裡一個很基礎的最佳化技術。它的做法很單純,就是把獨立的任務同時執行,但帶來的效果是讓你的應用效能更高、回應更靈敏。

理解了它能用在哪,我們現在動手看第一個實作,用的是 LangChain。我要先說清楚這段程式在做什麼、它的輸入跟輸出是什麼、為什麼這樣設計,你不用看到程式碼也能在腦中想像它的運作。

它的整體構想是這樣:當使用者丟進來一個查詢,我們同時跑兩件以上彼此獨立的操作,然後把各自的輸出彙整成一個統一的結果。在 LangChain 裡,要做平行,你就是把多個可執行的處理單元放進一個字典或清單的結構裡;當這個結構被當成輸入往後傳的時候,LangChain 的執行階段就會聰明地把裡面的單元同時跑起來。

具體來說,這個例子先建立了一個語言模型的實例,用的是一個叫 gpt-4o-mini 的模型,而且建立的時候用了防錯的包裝,萬一模型起不來,程式不會直接崩潰,而是會記下來、繼續往下。

接著它定義了三條彼此獨立的處理鏈,這三條就是等一下要平行跑的三個任務。第一條負責把主題做精簡摘要。第二條負責針對這個主題生出三個有趣的問題。第三條負責從主題裡辨識出五到十個關鍵詞,用逗號分隔。這三條鏈的結構都長一樣:先有一段提示樣板,告訴模型該做什麼,然後接上語言模型,最後接上一個輸出解析器,把模型回傳的東西整理成乾淨的純文字字串。

關鍵的一步來了。程式用一個叫做平行執行器的結構,把這三條鏈綁在一起,讓它們同時跑。同一時間,它還偷偷多塞了一樣東西進去,叫做直通器。直通器的作用,是把使用者原本輸入的那個主題原封不動地傳到後面去。為什麼要這樣?因為等一下做綜合的時候,我們不只要摘要、問題跟關鍵詞,還想保留原始主題當參考,所以要讓它一路直通到底。

三條平行鏈都跑完之後,我們就有四樣東西在手上:摘要、三個問題、一串關鍵詞,還有原始主題。接著程式定義了一段綜合用的提示,它把這四樣東西當成輸入,要求模型根據摘要、相關問題跟關鍵詞,綜整出一個全面的答案。

最後,整條端到端的流程就這樣串成了:先是那個讓三件事同時發生的平行區塊,接到綜合提示,再接到語言模型,最後接到輸出解析器。整條鏈是用一個非同步的函式去執行的,執行的時候你只要丟進一個主題字串,這個字串就會被送進平行區塊裡的每一個處理單元。範例裡用的測試主題是「太空探索的歷史」。

這裡有一個觀念我要特別幫你釐清,因為很多人會搞混,就是並行跟平行的差別。這個例子底層用的非同步機制,嚴格來說提供的是並行,英文叫 concurrency,而不是真正的平行,英文叫 parallelism。差別在哪?真正的平行是好幾件事在不同的處理器核心上真的同時在跑。而並行是在單一一條執行緒上,靠一個叫事件迴圈的東西在調度。事件迴圈的運作方式是這樣:當某個任務閒下來了,比方說它正在等一個網路請求回來,在那段空檔裡,事件迴圈就切換去推進別的任務,於是營造出一種大家好像同時在前進的效果。但它本質上同一瞬間還是只有一件事在動。這背後還有一個限制,叫做全域直譯器鎖,英文縮寫是 GIL,它讓 Python 在這條路上沒辦法真正同時執行多段運算。

那你可能會問,既然不是真平行,那還有用嗎?非常有用。因為我們這個場景的瓶頸,根本不在運算,而在等待。三條鏈都在等語言模型回話,這就是大量的閒置空檔。事件迴圈正好趁這些空檔互相穿插,所以三次模型呼叫實際上幾乎是疊在一起進行的,整體時間就被壓縮了。這就是這個例子展現的 LangChain 平行化核心理念:讓多次語言模型的呼叫同時發生,再用最後一次呼叫把它們組合起來。

講完 LangChain,我們來看第二個實作,用的是 Google 的代理開發套件,也就是剛剛說的 ADK。這個例子的味道不太一樣,它是用一整個一整個的子代理當積木來組裝的,會用到三種基本元件:語言模型代理、平行代理,還有循序代理。

這個系統的任務,是研究並綜整永續科技的進展。它的設計是這樣分工的。首先有三個研究員代理,每一個都是一個語言模型代理,各自有專長。第一個專門研究再生能源的最新進展。第二個專門研究電動車技術的最新發展。第三個專門研究碳捕捉方法的現況。這三個代理都用同一個 Gemini 模型,也都配備了 Google 搜尋這個工具,讓它們能上網查資料。它們收到的指令都很一致:用搜尋工具去查、然後用一到兩句話精簡地摘要關鍵發現、而且只輸出摘要本身,不要囉嗦。

這裡有一個很重要的設計細節,叫做輸出鍵,英文是 output key。每個研究員代理查完之後,不是把結果直接吐給使用者,而是把摘要存進一個共享的地方,叫做工作階段狀態,英文是 session state。你可以把工作階段狀態想成一塊大家都能讀寫的共用白板。第一個研究員把它的結果寫在白板上「再生能源」那一格,第二個寫在「電動車」那一格,第三個寫在「碳捕捉」那一格。這塊白板,等一下負責綜合的代理會去讀。

接著,程式用一個平行代理,把這三個研究員包起來。這個平行代理的職責就是協調這三位同時開工,然後它會一直等,直到三位研究員全部都查完、也全部都把結果寫上白板了,它才算結束。這就對應到我們前面說的:平行的步驟全部完成,才能進下一步。

下一步是一個負責合併的代理,我們叫它綜合代理,它也是一個語言模型代理,而且它一定是在平行代理跑完之後才動作。它的工作是去白板上把三格結果都讀回來,然後綜整成一份結構化、而且標明出處的報告。這個代理的指令下得非常嚴格,嚴格到我要特別提醒你:它被要求整份回應只能根據白板上那三段摘要來寫,絕對不准加入任何外部知識、任何沒出現在這些摘要裡的事實或細節。為什麼要這麼嚴?因為這能防止模型自己腦補、亂編內容,確保報告裡每一句話都有來源、可以追溯。它輸出的報告會分成幾個帶標題的區塊,再生能源一塊、電動車一塊、碳捕捉一塊,每一塊都註明是哪位研究員查的,最後再附一段簡短的整體結論。

最後,怎麼把整個流程串起來呢?用的是循序代理。這個循序代理當總指揮,它裡面排了兩個步驟,有先後順序:第一步先跑那個平行代理,讓三位研究員同時把白板填滿;第二步再跑綜合代理,把白板上的東西整理成最終報告。這個循序代理被設定成根代理,英文是 root agent,意思就是整個系統的執行進入點,你從這裡按下開始,整套流程就跑起來。

你有沒有注意到這個設計很漂亮的地方?它把平行跟循序兩種控制流程套在一起用了。研究這一段,因為三個主題互不相干,所以用平行;但研究跟綜合之間有明確的先後,必須先有資料才能整理,所以這層用循序。這正是真實系統的樣子:不是非平行即循序,而是該平行的地方平行、該排隊的地方排隊。

兩個實作都看完了,我幫你把這一章的重點用三個問題收斂一下,分別是「是什麼」、「為什麼」,還有「什麼時候用」。

先說「是什麼」。很多代理工作流程裡頭,有好幾個都得完成的子任務。如果你純循序執行,也就是每一項都死等前一項做完,往往會很慢。尤其當這些任務要依賴外部的輸入輸出,比方說呼叫不同的 API、查好幾個資料庫,延遲就會變成嚴重的瓶頸。在這種純循序的情況下,總處理時間等於所有任務耗時加總起來,這會狠狠拖累整體的效能跟回應速度。

再說「為什麼」。平行化提供了一個標準化的解法,讓那些彼此獨立的任務同時跑。它的精神,就是找出工作流程裡那些不需要等別人即時輸出的元件,可能是某次工具使用,也可能是某次語言模型呼叫,然後用 LangChain、Google ADK 這些框架內建的結構去定義跟管理它們。一個很典型的樣貌就是:主行程觸發了好幾個平行的子任務,然後它耐心等到全部完成,才進入下一步,靠這個方式把總執行時間大幅壓低。

最後是「什麼時候用」,也就是經驗法則。當你的工作流程裡包含好幾個可以同時執行的獨立操作時,就該用這個模式。具體一點的訊號,比方說你要從好幾個 API 同時擷取資料、要同時處理不同的資料區塊、或者要一次生成好幾份內容好讓後面去綜合比較,這些都是平行化該出場的時機。

最後我們來把整章收束起來,順便補幾個前面沒特別強調、但很重要的提醒。

第一,平行化的本質,是同時執行獨立任務來提升效率。它在任務需要等待外部資源、像是等 API 回應的時候,效益特別突出,因為你把一堆等待的時間疊在一起花掉了。

第二,這裡要潑一點冷水,提醒你天下沒有白吃的午餐。採用並行或平行架構,會引入可觀的複雜度跟成本。它會影響到開發的好幾個階段:設計變難了,因為你要想清楚哪些能平行、彙整點在哪;除錯變難了,因為好幾件事同時在跑,問題不一定每次都重現;就連系統的日誌記錄也變麻煩了,因為各路結果交錯回來,順序不固定。所以平行化不是無腦套上去就好,你要先確認任務真的彼此獨立、也真的值得這個複雜度。

第三,框架是有現成支援的,你不用自己硬幹。LangChain 跟 Google ADK 都內建了平行執行的能力。在 LangChain 的表達式語言裡,那個負責讓多個可執行物件並排運行的結構,就是這個模式落地的關鍵。

第四,Google ADK 還有一條很有代理味道的路,叫做大型語言模型驅動的委派,英文是 LLM-Driven Delegation。它的玩法是這樣:有一個協調者代理,英文叫 Coordinator,它背後的語言模型會自己去判斷,這個大任務裡哪幾塊是彼此獨立的子任務,然後把這些子任務分派給各有專長的子代理,讓它們同時開工。換句話說,要不要平行、怎麼拆,這件事本身也可以交給模型去動腦,而不是全由工程師寫死。

第五,把這些好處總結成一句:平行化降低了整體的延遲,讓代理系統在面對複雜任務的時候,反應更快、更靈敏。

那整章最大的一個收穫是什麼?平行化是透過同時執行彼此獨立的子任務,來最佳化你的運算工作流程。它在那種需要好幾次模型推論、或者好幾次外部服務呼叫的複雜操作裡,效益最明顯。不同框架的機制各有風味:LangChain 是用那個並排執行的結構,明確地定義好幾條鏈一起同時跑;Google ADK 則是走多代理委派的路線,由協調者模型把子任務分派給各有專長、能同時運作的代理。

而真正高明的系統,不會只用平行這一招。你要做的,是把平行處理,跟我們前面學過的循序也就是鏈式控制、還有條件也就是路由控制,三者整合在一起。該排隊的地方排隊、該分岔的地方分岔、該同時做的地方同時做。這樣你才能建構出一個精密、高效能、而且能從容應對各種複雜任務的運算系統。這,就是平行化這個設計模式留給我們最重要的一課。
