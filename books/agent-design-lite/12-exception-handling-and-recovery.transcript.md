這一章我們要談的,是怎麼讓 AI 代理在出錯的時候,還能站得住腳。這裡說的 AI 代理,英文叫 AI agent,指的是那種能自己規劃、自己呼叫工具、自己完成任務的智慧程式。這一章的主題,叫做「例外處理與復原」,英文是 Exception Handling and Recovery。我會帶你先弄清楚這個模式到底在解決什麼問題,接著拆解它的三大支柱,也就是錯誤偵測、錯誤處理、跟事後復原;然後我們會走過一連串很貼近實務的使用案例;最後一起看一段用 Google 的代理開發套件寫出來的範例,理解它的設計巧思。

我們先從「為什麼需要它」說起。想像一個情境:你寫了一個很聰明的代理,在你的電腦上、在你準備好的乾淨資料上,跑得漂漂亮亮。可是一旦把它丟到真實世界,事情就完全不一樣了。真實環境是混亂的、是無法預測的。它依賴的某個工具可能突然壞掉,網路可能斷線,它拿到的資料可能格式錯誤,它呼叫的某個外部服務可能剛好在維護。這些狀況,你事先全部都想不到。人遇到突發的障礙,會臨機應變、會繞路、會想辦法;一個可靠的代理,也必須具備同樣穩健的機制,能夠察覺問題、啟動復原,或者退一步說,至少要能用一種「可控的方式」失敗,而不是整個當掉。

所以這個模式的核心精神,其實是兩條腿走路:一條腿是「主動準備」,在事情還沒爆之前就先設想哪裡可能出包、先把應對方案準備好;另一條腿是「反應策略」,問題真的發生時,有一套清楚的章法去處理它,把代理拉回到一個穩定的狀態。除此之外,還有一個常被忽略但很關鍵的角色,就是完善的監控與診斷工具。它的價值在於「防微杜漸」,在小問題還沒滾成大災難之前,就先把它辨識出來。

這裡先補充一個很實用的搭配。這個例外處理模式,常常跟另一個模式一起用,那個模式叫「反思」,英文是 reflection。它們怎麼合作呢?舉個例子:代理第一次嘗試一個任務,失敗了,觸發了一個例外,英文叫 exception。這時候反思這個機制就上場,它會回頭分析「我這次為什麼失敗」,然後用一個更精煉的做法重新來過,比如說,把當初給語言模型的提示詞改寫得更清楚一點,再試一次。所以例外處理負責「接住失敗」,反思負責「從失敗裡學到東西再出手」,兩者是很自然的拍檔。

接下來,我們把這個模式攤開來看它的整體樣貌。它存在的理由,就是為了回應「AI 代理需要管理運作失敗」這個需求。它的做法很有條理:先預先設想可能出什麼問題,比如工具出錯,或是某個服務根本連不上;然後針對這些情況,擬好緩解跟復原的策略,目的就是把代理恢復到穩定。為了讓你有畫面,給你三個很生活化的例子:一個是會處理資料庫錯誤的聊天機器人,資料庫一時連不上,它知道該怎麼辦;一個是會應對金融錯誤的交易機器人,遇到錢不夠或市場狀況不對,它有對策;還有一個是會處理裝置故障的智慧家庭代理,某個裝置罷工了,它不會跟著一起癱瘓。

現在我們進到這個模式最重要的骨架,它由三個關鍵環節組成,你可以把它想成處理危機的三個階段:先是發現問題,再是應對問題,最後是收拾善後。我們一個一個來講。

第一個環節是「錯誤偵測」,英文是 Error Detection。它要做的事很單純,就是在問題發生的當下,把它認出來。那代理要怎麼知道「出事了」?有好幾種徵兆可以抓。第一種,是工具吐回來的輸出無效,或者格式根本不對,你期待它給你一個整齊的答案,它給你一坨亂七八糟的東西。第二種,是收到特定的 API 錯誤。這裡的 API 是應用程式介面,英文是 Application Programming Interface,你可以理解成程式跟程式之間溝通的窗口。常見的錯誤碼像是 404,代表「找不到」,也就是你要的東西不存在;還有 500,代表「伺服器內部錯誤」,也就是對方的伺服器自己出狀況了。第三種徵兆,是反應時間異常地長,正常幾秒就回來的,結果等了老半天都沒動靜,這本身就是個警訊。第四種,是回應內容前言不對後語,偏離了你預期的格式,語無倫次。除了代理自己察覺,你還可以更主動一點,額外安排其他代理,或是專門的監控系統,在旁邊盯著,幫忙攔截這些異常。

第二個環節是「錯誤處理」,英文是 Error Handling。偵測到問題之後,接下來就是擬定一個回應計畫,該怎麼接招。這裡有五種常見的招式。第一招叫「記錄」,就是把錯誤的細節寫下來,留作日後除錯用,這是最基本的功課,因為如果你連發生了什麼都不知道,後面什麼都談不上。第二招叫「重試」,就是再試一次,必要的時候還可以微調一下參數再試。這一招對付「暫時性錯誤」特別有效,暫時性錯誤英文叫 transient errors,意思是那種一閃即逝的故障,比如網路抖一下、伺服器忙一下,你過幾秒再試,它可能就好了。第三招叫「後備」,英文是 fallback,就是此路不通就改走別條路,換一個替代策略,至少維持住一部分功能。第四招叫「優雅降級」,意思是當你沒辦法立刻完全復原時,不要乾脆兩手一攤,而是退而求其次,還是提供一部分價值給使用者。打個比方,影音串流卡頓時,自動把畫質從高清降成標清,讓你至少還看得下去,而不是直接黑屏,這就是優雅降級的精神。第五招叫「通知」,當情況需要人介入時,就主動發警示,通知操作人員,或是通知其他代理來幫忙。

第三個環節是「復原」,英文是 Recovery。這一步是收拾善後,目標是把整個系統恢復到一個穩定、可以正常運作的狀態。它有幾種手段。第一種叫「狀態回滾」,意思是把最近做的變更或交易還原掉,撤銷它造成的影響,就像文書軟體裡的「復原」鍵,把你剛剛搞砸的那一步退回去。第二種,是去調查問題的根本成因,搞清楚為什麼會出這個錯,目的是防止它再犯。第三種,是透過自我修正,或是「重新規劃」來調整。重新規劃英文叫 replanning,意思是代理把原本的計畫、邏輯或參數重新盤算一遍,改出一條新路。第四種,是當問題嚴重到代理自己搞不定的時候,就「升級轉介」,把問題往上交給人類,或是交給更高層級的系統來處理。

把這三個環節合起來,你就能體會這套模式真正的威力:它能夠把一個 AI 代理,從原本那種一碰到意外就崩潰、不可靠的脆弱系統,徹底改造成一個穩健、值得信賴的元件,讓它在難以預測的環境裡照樣有效運作,而且把停機的時間壓到最低。

講完了原理,我們來看它在真實世界裡長什麼樣子。原則上,任何一個真正部署到實際情境的代理,都需要例外處理與復原,沒有例外。我給你六個具體的場景,你會發現同一套思路,在各行各業都用得上。

第一個場景,客戶服務聊天機器人。假設它背後的客戶資料庫暫時離線了。一個沒有處理能力的機器人會直接當機;但一個穩健的機器人,會偵測到這個 API 錯誤,然後溫和地告訴使用者「這只是暫時的問題,請您稍後再試」,或者乾脆把這通對話升級轉介給真人客服。你看,它用到的就是我們前面講的偵測、通知、跟升級轉介。

第二個場景,自動化金融交易。當它遇到「資金不足」或「市場休市」這類錯誤時,關鍵在於兩件事:一是要把錯誤記錄下來,二是千萬不要傻傻地對著同一筆無效的交易反覆重試,因為那不但沒用,還可能造成麻煩。它應該做的,是通知使用者,或是調整自己的策略。這裡的重點是,重試這一招要用對地方,對暫時性的故障有效,但對「資金不足」這種根本性的問題,再試一百次也是錯,反而要避免重試。

第三個場景,智慧家庭自動化。一個負責控制燈具的代理,因為網路或裝置故障,沒辦法把燈打開。它應該先偵測到這個失敗,然後重試;如果重試還是不行,就通知使用者,並且建議他手動去開燈。

第四個場景,資料處理代理。它在批次處理一堆文件的時候,中間碰到一個損毀的檔案。正確的做法,不是讓這一個壞檔案搞垮整個流程,而是略過它、記下這個錯誤、繼續處理剩下的檔案,最後再回報「這幾個檔案我跳過了」。這其實就是優雅降級的精神,我不是全有或全無,我盡量把能做的事做完。

第五個場景,網頁爬取代理。它在抓取網頁時,可能會遇到 CAPTCHA,就是那種「請證明你不是機器人」的驗證;也可能遇到網站改版、結構變了;或是遇到伺服器錯誤,像是剛剛說過的 404 找不到,還有 503,代表「服務暫時無法使用」。面對這些,它要優雅地處理,比如暫停一下、改用代理伺服器,代理伺服器英文叫 proxy,可以理解成一個中轉的替身來繞過封鎖,或者把抓失敗的那些網址回報出來。

第六個場景,是機器人技術跟製造業。想像一隻機械手臂,因為對位沒對準,夾不起一個零件。它要透過感測器的回饋偵測到「我夾失敗了」,然後嘗試重新校準、再夾一次;如果問題一直存在,就警示操作員,或是改去抓另一個零件。你會發現,這跟前面軟體的例子根本是同一套邏輯,只是換到了實體世界。

所以總結這一段,這個模式,是打造一個「不只是聰明,而且在面對真實世界的複雜跟混亂時,還能可靠、有韌性、對使用者友善」的代理的根本基礎。聰明很重要,但能扛得住現實,才真正能用。

接下來,我們一起看一段範例程式,理解這些概念實際上怎麼落地。這段範例是用 ADK 寫的,ADK 是 Agent Development Kit,也就是代理開發套件,是 Google 提供的一個用來打造代理的工具。我不會逐字唸程式碼,我會用講的,讓你在腦中把它的運作想像出來。

這段範例要示範的,是代理怎麼回應「一次失敗的工具呼叫」。失敗的原因可能有兩種:一種是工具的輸入給錯了,另一種是工具背後依賴的外部服務出了問題。它的設計,是把三個子代理串成一條生產線。子代理英文叫 sub-agent,就是大代理底下分工的小代理。

第一個子代理,叫做「主要處理器」。它的職責範圍故意設計得很窄、很明確,就是專心做一件事:嘗試取得精確的地點資訊。它手上握著一個工具,專門用來根據使用者給的地址,查出精準的位置。這是第一線的嘗試,我們希望它成功,但我們也預期它可能失敗。

第二個子代理,叫做「後備處理器」。它的角色就是我們前面講的「後備」這一招的具體實現。它做的第一件事,是去檢查一個共用的狀態標記,這個標記記錄著「主要的地點查詢到底失敗了沒」。如果這個標記顯示「真的失敗了」,它就啟動備案:從使用者最原始的問題裡,把城市名稱抓出來,然後改用另一個比較粗略的工具,去查一個「大概的區域資訊」。意思是,精確地址查不到沒關係,那我退一步,至少給你個大概的範圍。但如果那個標記顯示「主要查詢其實成功了」,那這個後備處理器就什麼都不做,安靜地讓路。這就是後備機制的優雅之處:只在需要的時候才出手。

第三個子代理,叫做「回應代理」。它本身不呼叫任何外部工具,它唯一的工作,是針對最終的狀態做推理跟呈現。它會去看那個存放地點結果的狀態,如果裡面有東西,就把它清楚、簡潔地講給使用者聽;但如果那個結果根本不存在、或者是空的,代表前面兩關都沒查到,那它就誠懇地向使用者道歉,說明這次無法取得這個地點。這一步,正好對應到我們前面說的「優雅地失敗」,就算最後真的沒結果,也要好好地、有禮貌地告訴使用者,而不是丟出一堆看不懂的錯誤。

那這三個子代理怎麼確保照順序乖乖執行呢?靠的是 ADK 裡一個叫做「循序代理」的元件,英文是 SequentialAgent。它的作用,就是把這三個處理器依照「先主要、再後備、最後回應」這個固定的順序串起來,保證它們一個接一個跑,不會亂套。所以整個流程合起來,就是一個分層式的地點檢索:先試精確的,不行就退而求其次拿大概的,最後不管結果如何,都好好地回應使用者。這短短一段程式,其實把偵測、後備、優雅降級這幾個概念,完整地演了一遍。

我們把整章的精華再收斂一次,從三個角度幫你定錨:這個模式是什麼、為什麼需要它、以及什麼時候該用它。

先講「是什麼」。在真實環境運作的 AI 代理,無可避免會遇到各種無法預見的狀況、錯誤跟系統故障,從工具失敗、網路問題,到無效的資料,每一樣都在威脅代理完成任務的能力。如果沒有一套有結構的方式去管理這些狀況,代理一碰到意外就會變得脆弱、不可靠,這樣的東西,你根本不敢把它部署到那些需要穩定表現的關鍵應用上。

再講「為什麼」。這個模式提供的,是一套標準化的解法,它賦予代理三種能力:預先設想失敗、管理失敗、以及從失敗中復原。它一方面涵蓋主動式的錯誤偵測,也就是持續監控工具的輸出跟 API 的回應;另一方面也包含反應式的處理策略,像是記錄、對暫時性失敗做重試、以及後備。而對於那些嚴重的問題,它還定義了復原協定,包括還原到穩定狀態、自我修正、或是升級轉介給人類。這樣一來,就能確保代理始終維持運作的完整、能夠從失敗中學習,並且在難以預測的環境裡可靠地運作。

最後是「經驗法則」,幫你判斷什麼時候該掏出這個模式。簡單講:當你的 AI 代理要部署到一個動態的、真實的環境,那裡隨時可能發生系統失敗、工具錯誤、網路問題,或是各種你預料不到的輸入,而且「運作可靠」對你來說是一個不能妥協的關鍵需求時,你就該用它。

也順著把幾條核心重點幫你串起來。第一,這個模式的核心,就是偵測錯誤、優雅地處理錯誤、再實作復原策略,這三件事。第二,錯誤偵測這一塊,涵蓋了驗證工具的輸出、檢查 API 的錯誤代碼、以及善用逾時機制,逾時英文叫 timeout,就是設一個等待的上限,超過了就當作失敗,不要無止盡地空等。第三,處理策略,就是我們反覆講的那五招:記錄、重試、後備、優雅降級、跟通知。第四,復原這一塊,聚焦在透過診斷、自我修正、或升級轉介,把系統帶回穩定的運作。

我們做個總收尾。例外處理與復原這個模式,對於開發出穩健、可信賴的 AI 代理來說,是至關重要的。它讓代理有能力辨識並管理那些意料之外的問題,實作出合適的回應,並且復原到穩定的狀態。這一章,我們從錯誤偵測講起,接著走過記錄、重試、後備這些處理錯誤的機制,再到各種事後復原的策略,最後用橫跨客服、金融、智慧家庭、資料處理、網頁爬取到機器人製造這麼多領域的實例,讓你看見同一套模式,是如何在面對真實世界的複雜與潛在失敗時,實實在在地提升一個代理的可靠性跟適應力。記住這個核心:真正能用的代理,不是那個永遠不出錯的代理,而是那個出了錯還能優雅地接住自己、站回來的代理。
