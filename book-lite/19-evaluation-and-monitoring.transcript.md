這一章我們要談的,是怎麼幫一個會自己做事的 AI 系統「打分數」跟「盯著它」。換句話說,當你部署了一個智慧代理之後,你怎麼知道它做得好不好、有沒有愈做愈差、會不會突然出包?這一章就是要回答這件事。

先說一下這裡講的智慧代理是什麼。智慧代理,英文叫 intelligent agent,簡單講就是一個能自己理解任務、自己規劃步驟、自己呼叫工具去完成目標的 AI 程式。它不是寫死的程式,而是會「思考」跟「行動」的東西。

我先幫你把這一章的路線圖講一遍,讓你心裡有個底。我們會先談為什麼要評估跟監控,以及它在真實世界裡有哪些用途。接著進入動手實作的部分,看四個很具體的衡量方法:怎麼評回應準不準、怎麼監控延遲、怎麼追蹤 token 用量、還有怎麼用一個 AI 去當另一個 AI 的評審。然後我們會比較人工評估、AI 評審跟自動化指標三種做法的優缺點。再來談一個很關鍵的概念,叫「代理軌跡」,也就是不只看結果,還要看它一步一步是怎麼走到答案的。之後我們會聊多代理系統怎麼評估,還有一個很前瞻的構想,叫做從「代理」進化成「承包商」。最後簡單帶一下 Google 的 ADK 這個工具怎麼支援評估,再做總結。

那我們先從「為什麼要評估跟監控」說起。

核心的想法是這樣:你不能只在上線前測一次就放著不管。一個代理要持續地、而且常常是從外部去衡量它三件事。第一是有效性,英文叫 effectiveness,也就是它到底有沒有把事情做對、做好。第二是效率,英文叫 efficiency,也就是它花了多少時間、多少資源去做這件事。第三是它對需求的遵循程度,也就是它有沒有照你定下的規則跟目標走。

為了做到這三件事,你需要三樣東西配套。第一,你要先定義「指標」,也就是你要量什麼、用什麼數字來代表好壞。第二,你要建立「回饋迴路」,英文叫 feedback loop,意思是把量到的結果再餵回去,變成改進的依據,形成一個循環。第三,你要有「報告系統」,把這些結果整理出來給人看、或給另一個系統用。指標、回饋迴路、報告系統,這三個就是整個評估與監控的骨架。

接下來講它在實務上有哪些用途。這裡有七個很典型的使用案例,我一個一個說。

第一個是即時效能追蹤。意思是對正式環境裡正在跑的代理,持續監控它的準確度、延遲跟資源消耗。延遲就是反應的快慢。舉個例子,如果你有一個客服機器人,你會想隨時知道它的「解決率」,也就是有多少比例的問題它真的幫客戶解決了,還有它的回應時間有多長。這些是你天天都要盯的數字。

第二個是 A/B 測試。意思是你同時跑兩個不同版本的代理,或是兩種不同的策略,讓它們平行比較,看哪一個表現比較好。這就像餐廳同時試兩種菜單,看哪一種客人比較買單。

第三個是合規與安全稽核。也就是自動產生稽核報告,長期追蹤這個代理有沒有守倫理、守法規、守安全規範。這個驗證的工作,可以由「人在迴路」來做,英文叫 human-in-the-loop,意思是流程中安排真人來把關;也可以由另一個代理來驗證。驗證的結果可以變成關鍵績效指標,英文縮寫是 KPI,全名是 key performance indicator,也就是用來衡量成效的關鍵數字;或者一旦出問題就觸發警示。

第四個用途是在企業系統裡。當你要治理一整間企業裡那些會自己做事的 AI,你會需要一份 AI「合約」。這個合約是一份動態的協議,把任務目標、規則跟控制機制都白紙黑字寫清楚。這個「合約」的概念很重要,我們後面會花一大段專門講它,你先記住這個詞。

第五個是漂移偵測,英文叫 drift detection。漂移的意思是,世界變了,但你的代理沒跟著變,結果它就慢慢退化了。具體來說,你要監控輸出的相關性跟準確度,去偵測因為「輸入資料的分布變了」而造成的退化,這個現象叫概念漂移;或是因為環境本身改變造成的退化。打個比方,你訓練一個推薦系統的時候大家都在瘋某個東西,過了一年流行變了,如果它還照舊推,那就是漂移了。

第六個是異常偵測。意思是去辨識代理那些「不在預期內」的動作。這種非預期的行為,可能代表程式有錯誤、可能代表系統被攻擊、也可能是它自己衍生出來的奇怪行為。

第七個是學習進度評估。意思是去追蹤代理的學習曲線、它在某個特定技能上的進步,還有它的「泛化」能力,英文叫 generalization。泛化指的是它能不能把學到的東西用到沒看過的新任務跟新資料上,而不是只會背它練過的題目。

好,用途講完了,我們進到動手實作的部分。

這裡要先講一句很實際的話:一個完整的評估框架,理論上要同時顧到模型的表現、使用者的互動、還有倫理跟社會層面的衝擊,這實在太龐大、太艱鉅了。所以實務上落地的時候,我們會把焦點收斂到幾個關鍵案例。接下來就講這幾個關鍵案例。

第一個關鍵案例,叫做代理回應評估,英文是 agent response assessment。它要回答的問題是:代理給出的這個回應,到底切不切題、正不正確、合不合邏輯、有沒有帶偏見?可以拿來衡量的指標包括:事實正不正確、講得順不順、文法精不精準、還有它有沒有符合使用者真正的意圖。

書裡給了一段很簡單的程式來示範這件事,我用聽得懂的方式講它在做什麼。這段程式是一個函式,你餵給它兩個東西:一個是代理實際產出的回答,另一個是你心中的標準答案。它做的事情很單純,就是把這兩段文字的前後空白去掉、全部轉成小寫,然後一個字一個字嚴格比對。如果完全一樣,它就回傳一分,代表滿分;只要有一點不一樣,它就回傳零分。

這段程式的用意,是讓你看到「最陽春的評估」長什麼樣子,以及它的問題在哪。書裡舉的例子很傳神。代理回答的是「法國的首都是巴黎」,而標準答案寫的是「巴黎是法國的首都」。你我都知道這兩句意思一模一樣,對吧?但是因為字串排列不同,這個函式就會死板地回傳零分,判它答錯。這就是逐字比對的致命傷:它只看字面,不看意思。

所以結論是:直接做字串比較,沒辦法評估「語意」上像不像,你需要更進階的自然語言處理技術,自然語言處理的英文縮寫是 NLP,全名是 natural language processing。書裡列了幾種更精密的指標,我幫你說清楚。第一類是字串相似度,裡面包含萊文斯坦距離,它衡量的是把一個字串改成另一個字串要動幾個字;還有傑卡德相似度,它看的是兩段文字共用的詞佔比。第二類是關鍵字分析。第三類是用嵌入模型算「餘弦相似度」,英文叫 cosine similarity。這裡的嵌入,是把文字轉成一串數字向量,讓意思相近的句子在數學空間裡靠得很近,然後用餘弦相似度去量它們的夾角有多接近,夾角愈小就愈像。這一招就能解決剛剛「巴黎是法國首都」那種語意相同但字面不同的問題。第四類是「LLM 即評審」,這個我們等一下會專門講。第五類是 RAG 專屬的指標,RAG 是檢索增強生成,英文全名是 retrieval-augmented generation,它的專屬指標包括忠實度,也就是回答有沒有忠於它找到的資料,還有相關性。

第二個關鍵案例是延遲監控,英文是 latency monitoring。延遲就是代理從收到你的請求,到吐出結果,中間花了多少時間。為什麼要盯這個?因為延遲太高會直接傷害使用者體驗,特別是在那種需要即時互動的場合,你問一句它要想五秒才回,體驗就崩了。

這裡有個很重要的實務提醒:延遲不應該只是印在主控台上看一眼就算了。你應該把它「記錄」下來,寫到一個能持久保存的地方。書裡列了幾種選擇給你參考。一種是結構化日誌,比如用 JSON 格式存。一種是時序資料庫,這是專門存「隨時間變化的數字」的資料庫,書裡點名了 InfluxDB 跟 Prometheus。一種是資料倉儲,適合做大規模分析,例如 Snowflake、BigQuery 跟 PostgreSQL。還有一種是可觀測性平台,專門幫你監控系統健康狀況,例如 Datadog、Splunk 跟 Grafana Cloud。重點不是要你背這些名字,而是要建立一個觀念:延遲這種數據要長期留存,你才能回頭分析趨勢。

第三個關鍵案例是追蹤 token 用量。這裡的 token,你可以理解成大型語言模型處理文字的基本單位,大致上是一個詞或一小段詞。大型語言模型英文叫 LLM,全名是 large language model。為什麼要追蹤 token?因為 LLM 的計費,通常就是看你輸入了多少 token、它輸出了多少 token 來算錢的。所以追蹤 token 用量,對控制成本、對最佳化資源都非常關鍵;而且看它哪裡花了特別多 token,你還能反過來找出提示工程或回應生成可以改進的地方。

書裡同樣給了一段示意程式。我講它的設計意圖。它做了一個監控器物件,這個物件裡面記著兩個累加的數字,一個是輸入 token 的總數,一個是輸出 token 的總數。每當發生一次互動,你就把這次的提示跟回應交給它,它會算出這次用了多少 token,然後加進總帳裡,並且即時印出這次的用量。最後你可以隨時跟它要目前的累計總數,拿去做成本管理。

這裡要特別提醒一個誠實的細節:書裡這段程式為了示範,是用「把句子按空白切成幾個詞」這種很粗略的方式去估 token 數的,它只是一個佔位用的算法。真實情況下,你必須改用 LLM 官方 API 提供的 token 計數器或斷詞器來算,因為真正的 token 切法跟「按空白數詞」差很多。所以這段程式重點是讓你理解「累加追蹤」這個設計概念,而不是它算 token 的方法。

第四個關鍵案例,就是剛剛預告過的「LLM 即評審」,英文是 LLM-as-a-Judge。我們先說它要解決什麼問題。有些品質是很主觀的,比方說「有用性」,英文叫 helpfulness。一個回答有不有用,這種感覺很難用那種客觀的數字指標去量。那怎麼辦?一個聰明的做法,就是請另一個 LLM 來當評估者,依照你預先訂好的準則,去評估前一個代理的輸出。

為什麼這招行得通?因為 LLM 本身語言能力很強,它對這種主觀特質,能給出「接近人類」的細緻判斷,比起單純的關鍵字比對或死板的規則式評估高明多了。當然,這個技術還在發展中,不是萬靈丹,但是在「自動化、又能大規模做質性評估」這件事上,它的潛力非常可觀。質性評估的意思,就是評那種沒辦法簡單用對錯數字表達的品質。

書裡用了一個很完整的例子來示範這招,是拿來評「法律問卷題目」的品質。我把這段程式的設計講清楚,你不用看到程式碼也能懂它怎麼運作。

整個東西的核心,是一份叫做「評分準則」的長文字,英文叫 rubric。這份準則先給 LLM 設定一個身份,告訴它:你現在是一位專精法律問卷的方法學專家,也是一位嚴格的法律審查者,你的任務是評估一道法律問卷題目的品質,要打一到五分,而且要附上詳細理由跟具體回饋。然後這份準則列出五個評分面向,每一個都從一分到五分定義得很清楚。第一個面向是清晰度與精確性,一分代表極度含糊、讓人困惑,五分代表完全清楚、毫不含糊。第二個面向是中立性與偏見,一分代表帶有強烈誘導性、明顯把人帶往某個答案,五分代表完全中立客觀。第三個面向是相關性與聚焦,看它跟問卷主題相不相關、夠不夠聚焦。第四個面向是完整性,看它有沒有給足受訪者作答需要的資訊。第五個面向是對受眾的適切性,看它的用語對目標讀者來說會不會太難或太淺。

這份準則最後還規定了輸出格式,而且這點很關鍵:它要求 LLM 的回應「必須」是一個結構化的 JSON 物件,裡面要包含幾個固定欄位。一個是整體分數,一個是評分理由的摘要,一個是針對五個面向逐項的詳細回饋,一個是列出法律、倫理或方法學上的疑慮清單,最後一個是建議採取的行動,例如「為求中立而修改」、「照原樣核准」或是「釐清範疇」。為什麼要逼它輸出 JSON?因為這樣機器才好自動解析跟後續處理,而不是收到一段散文還要再去拆。

那程式本身在做什麼?它包了一個評審類別。你建立這個評審的時候,可以指定要用哪個 Gemini 模型,還有一個叫「溫度」的參數。Gemini 是 Google 的生成式 AI 模型。溫度控制的是回答的隨機程度,這裡刻意設得很低,因為評分這種事我們要它穩定、要它確定性高,不要每次評分都飄。書裡也提到模型的取捨:有一個比較快、比較便宜的版本,還有一個品質最高的版本,看你需求選。

實際評分的時候,程式會把那份長長的評分準則,跟你要評的那道問卷題目,串接成一個完整的提示,送給模型。然後它會處理幾種狀況。如果模型因為內容審核之類的原因回了空白,它會記錄下安全評級並回報失敗。如果模型回的東西沒辦法被當成 JSON 解析,它會記錄這個解碼錯誤。如果出了其它意外,它也會捕捉起來。這些錯誤處理的設計意圖,是讓這個評審在真實環境裡跑的時候不會一出狀況就整個掛掉,而是能優雅地回報問題。

書裡還很貼心地用三個對照範例來示範它怎麼運作。第一個是「良好範例」,一道措辭嚴謹、中立、聚焦的瑞士智慧財產權法相關問卷題目。第二個是「帶偏見的不佳範例」,題目開頭就用「你難道不同意嗎」這種句式,明顯在誘導受訪者往某個答案走。第三個是「含糊範例」,題目只問「你對法律科技有什麼想法」,空泛到不行。透過這三個對照,你就能直觀感受到這個 LLM 評審怎麼分辨好題目跟壞題目。

講完這四個關鍵案例,書裡用一個小表格,比較了三種評估方法的優缺點,我用講的幫你比一比。

第一種是人工評估,英文是 human evaluation。它的優點是能捕捉到很細微的行為,人就是敏感,什麼怪怪的地方都看得出來。但缺點是很難規模化、成本高又耗時,而且因為是人在評,難免帶主觀。

第二種是 LLM 即評審。它的優點是一致、有效率、而且可以大規模地做。缺點是它可能會忽略中間的步驟,只看頭尾;而且它的評估品質會受限於這個 LLM 本身的能力,LLM 不夠強,評出來的東西就不夠準。

第三種是自動化指標,英文是 automated metrics,就是前面講的那些字串相似度、餘弦相似度之類的客觀算法。它的優點是可規模化、有效率、又客觀,不帶感情。缺點是它在捕捉「完整能力」這件事上可能有侷限,因為它畢竟是死的公式,抓不到太細緻或太全面的東西。

你會發現這三種各有長短,實務上常常是搭配著用,而不是只挑一種。

好,接下來進到這一章一個很核心的概念,叫做「代理軌跡」,英文是 agent trajectory。

這裡要先打破一個迷思:傳統的軟體測試,對代理來說是不夠用的。為什麼?因為傳統程式碼的產出是可預測的,你給一樣的輸入,它就給一樣的輸出,結果非黑即白,不是通過就是失敗。但代理是「以機率方式運作」的,同樣的問題它可能給出不太一樣的回答。所以對代理,你不能只看最終的那個輸出,你還要對「軌跡」做質性評估。軌跡,指的就是它為了達成答案,一步一步走過的那一連串步驟。

而多代理系統又更難評估。多代理系統英文是 multi-agent system,就是好幾個代理一起協作完成任務。它難在哪?難在它不斷變動,所以你需要更精密的指標去衡量代理之間的「溝通」跟「協作」,而且你的測試方法還得隨著環境一起調整。

那評估軌跡到底在評什麼?在評三件事:決策的品質、推理的過程,還有整體的結果。它超越了只看結果的層次。而且書裡特別點出,當你的代理過了原型階段、要正式放大規模的時候,這種自動化的軌跡評估就格外有價值。

那具體怎麼比?做法是,把代理「實際走的動作」跟一條「基準真相」軌跡去比對。基準真相,英文是 ground truth,你可以理解成「標準答案路線」,也就是專家認為理想上應該怎麼走。書裡舉了一個很清楚的例子:假設任務是處理產品查詢,那條理想的軌跡可能是這樣四步——先判定使用者的意圖,接著用資料庫搜尋工具去查,然後審閱查到的結果,最後生成一份報告。

那比對的方式也有好幾種,我講給你聽。第一種是完全比對,英文是 exact match,要求代理走的每一步都跟標準路線一模一樣。第二種是依序比對,英文是 in-order match,它要求關鍵步驟的順序對,但是允許代理中間多做一些額外步驟。第三種是任意順序比對,英文是 any-order match,只要該做的步驟都做了,順序不拘。除此之外還可以用精確率跟召回率,精確率英文是 precision,召回率英文是 recall,這兩個是衡量「抓對的比例」跟「該抓的有沒有漏」的指標。也可以單純看單一工具有沒有正確使用。

那什麼時候用哪一種?原則是這樣:在高風險的情境,你可能會要求完全比對,一步都不能錯;但在比較有彈性的情境,用依序比對或任意順序比對就可以了,給代理多一點發揮空間。

接著書裡介紹了做代理評估的兩種主要做法,這兩種要分清楚。

第一種叫「測試檔」,英文是 test files。它是 JSON 格式,代表的是一次單一、簡單的互動,或者說一個工作階段,工作階段英文是 session。測試檔的特色是適合開發期間做「單元測試」,強調快速執行。它的結構是:每個測試檔包含一個工作階段,裡面有多個「回合」,回合英文是 turn。一個回合,就是使用者跟代理的一次來回互動,裡面包含使用者的查詢、預期的工具軌跡、中間的回應跟最終的回應。書裡舉的例子是:使用者說「把臥室的二號裝置關掉」,這個測試檔會指定代理應該要呼叫一個設定裝置資訊的工具,而且參數要對——地點是臥室、裝置編號是二號、狀態設成關閉——然後預期最終回應是「我已經把二號裝置的狀態設成關閉了」。你看,它連工具參數跟最終回話都規定好了,這樣才能精準檢查。

第二種叫「評估集檔」,英文是 evalset files。它用的是一個叫「評估集」的資料集來評估互動,評估集英文是 evalset。它跟測試檔的差別在於:它包含多個、而且可能很長的工作階段,所以它適合用來模擬複雜的多回合對話,做「整合測試」,整合測試英文是 integration test。它的結構是:一個評估集由多個「評估」組成,每個評估英文是 eval,代表一個工作階段,裡面有一個或多個回合,並且附上參考的最終回應。書裡舉的例子是:使用者先問「你會做什麼?」,接著說「丟一顆十面骰子兩次,然後檢查九是不是質數」,而這個評估集會定義出預期要呼叫的擲骰工具、檢查質數的工具,以及把結果彙整起來的最終回應。

所以你可以這樣記:測試檔像是快速的隨堂小考,單純、跑得快,用在開發期;評估集檔像是期末的綜合大考,情境複雜、跨多輪,用在整合測試。

接著書裡專門談多代理系統的評估。它打了一個很好的比方:評估多代理系統,很像在評量一個「團隊專案」。在團隊專案裡,眾多的步驟跟交接,英文叫 handoff,反而是你的優勢,因為這讓你能在每一個階段都去檢查品質。重點是,除了看每個代理「個別」表現好不好,你「更必須」評估整個系統「合在一起」運作得好不好。書裡提了四個關鍵問題,我一個一個講。

第一個問題:代理之間有沒有有效協作?書裡的例子是,一個「訂機票代理」有沒有把正確的日期跟目的地,交給「訂飯店代理」?如果協作失敗,可能就會把飯店訂在錯的那一週,整趟行程就亂了。

第二個問題:它有沒有擬定計畫並貫徹到底?比方說計畫是「先訂機票、再訂飯店」,結果飯店代理在航班都還沒確認之前就先把房訂了,這就是偏離了計畫。同時你也要檢查代理會不會「卡關」,書裡舉的例子很生動:一個代理為了找「完美的」租車,無止盡地一直搜尋下去,卻一直不往前走,這也是個問題。

第三個問題:有沒有為任務選對代理?例子是,如果使用者問「我這趟旅程天氣如何」,那就應該用能提供「即時資料」的天氣代理;如果你用的是「一般知識代理」,結果它回你「夏天通常很溫暖」這種籠統廢話,那就是選錯工具了。

第四個問題:增加代理到底有沒有讓表現變好?例子是,你新增了一個「餐廳訂位代理」,它是真的讓整個規劃變得更好,還是反而製造衝突、拖慢系統?如果是後者,那就暴露了「可擴展性」的問題,可擴展性英文是 scalability,指的是系統能不能順利地長大、加更多東西進去還跑得動。

好,接下來是這一章我覺得最有意思的一段,叫做「從代理到進階承包商」。

這個構想來自一份叫《Agent Companion》的白皮書。它提出的願景是:讓 AI 從現在這種「機率性的、常常不太可靠的」代理,演進成一種專為高風險環境設計、更具「確定性」跟「當責性」的角色,這個角色叫做「承包商」,英文是 contractor。當責性,英文是 accountable,意思是它要為自己的產出負責、可以被追究。

為什麼要這樣演進?因為現在的代理,是靠很簡短、規格嚴重不足的指令在運作的。這種代理拿來做展示很漂亮,可是一放到正式環境就很脆弱,因為只要指令有模糊地帶,它就容易失敗。而「承包商」這個模型,是在使用者跟 AI 之間,建立一種嚴謹、形式化的關係,就像人類社會裡的法律服務協議一樣正式。這個承包商模型,由四大支柱撐起來,我一根一根講。

第一根支柱是「形式化合約」,英文是 formalized contract。它是一份詳細的規格,扮演「單一事實來源」的角色,單一事實來源英文是 single source of truth,意思是大家都以這份文件為準,不會各說各話。它遠遠超過一句簡單的提示。書裡的對比超級傳神:一份財務分析合約,不會只說「分析上一季的銷售」這麼含糊;它會明確要求——要一份二十頁的 PDF 報告,分析二零二五年第一季歐洲市場的銷售,裡面要含五個特定的資料視覺化圖表,要跟二零二四年第一季做對比分析,還要根據隨附的供應鏈中斷資料集做風險評估。你聽聽,這份合約把交付成果、精確規格、可以用哪些資料來源、範疇有多大,甚至連預期的運算成本跟完成時間都定義清楚了。這樣做的目的,就是讓最後的結果變得「客觀可驗證」,而不是各憑感覺。

第二根支柱是「協商與回饋的動態生命週期」,英文是 dynamic lifecycle of negotiation and feedback。關鍵觀念是:合約是一場「對話的起點」,而不是一道靜態的、死的命令。承包商可以分析合約的條款,然後跟你協商。書裡的例子是:當合約要求它去用一個它根本存取不到的專有資料庫時,它會回饋說——你指定的那個資料庫我存取不了,請你提供憑證,或者核准我改用替代的公開資料庫,但要先提醒你,這樣資料的粒度可能會稍微不一樣。這個階段的價值在於,它讓代理在「真正動手執行之前」,就先把模糊的地方跟風險標示出來,這樣就能防止那種代價高昂的失敗,也確保最後的產出真的契合你的本意。

第三根支柱是「以品質為核心的迭代執行」,英文是 quality-focused iterative execution。這跟前面那種一味追求低延遲、追求快的代理很不一樣。承包商優先重視的是「正確性」跟「品質」,它遵循自我驗證跟自我校正的原則,自我驗證英文是 self-validation。書裡用程式碼生成的合約來舉例:這個承包商代理會先生成好幾種不同的演算法做法,然後拿合約裡定義好的單元測試去編譯、去執行,再依照效能、安全性、可讀性來評分,最後「只提交」那個通過了所有驗證的版本。這種「生成、審閱、改進,一直到符合規格為止」的內部迴路,對於建立信任是非常關鍵的。

第四根支柱是「透過子合約進行階層式分解」,英文是 hierarchical decomposition via subcontracts。意思是,面對複雜任務的時候,主承包商代理可以扮演「專案經理」的角色,把大目標拆成一個個更小的子任務,然後為每個子任務生成形式化的「子合約」,子合約英文是 subcontract。書裡的例子是:一個「建構電子商務行動應用程式」的大任務,可以分解成好幾個子合約——設計使用者介面與體驗、開發使用者驗證模組、建立產品資料庫綱要、整合金流閘道等等。重點是,每一個子合約本身都是一份完整、獨立的合約,可以指派給專門的代理去做。這種分解方式,讓系統能夠有組織、可規模化地去處理龐大的專案,這標誌著 AI 從一個「工具」,轉變成一個「自主又可靠的問題解決引擎」。

把這四根支柱合起來看,承包商框架做的事情,就是把「形式化規格、協商、可驗證的執行」這三件事,嵌進代理的核心邏輯裡。它把 AI 從一個難以預測的助手,提升成一個能用「可稽核的精確度」自主管理複雜專案的可靠系統。這也為「在那些信任跟當責至上的關鍵任務領域部署 AI」鋪好了路。

講完這個前瞻構想,我們回到比較落地的工具層面,看一下 Google 的 ADK。ADK 是 Google 推出的代理開發套件。

ADK 在做代理評估這件事上,提供了三種方法,搭配三種使用情境。第一種是基於網頁的使用者介面,你下一個叫 adk web 的指令就能開,它用在「互動式的評估」跟「資料集的生成」,適合你一邊操作一邊看。第二種是用 pytest 做程式化整合,pytest 是 Python 常用的測試框架,這種方式可以把評估直接納入你的測試管線裡,讓它自動跑。第三種是命令列介面,你下一個叫 adk eval 的指令,適合做自動化評估,比方說每次建置後自動驗證。

我把這三種再講細一點。網頁介面讓你能互動式地建立工作階段,把它存進評估集,同時還會顯示評估的狀態給你看。pytest 整合,是透過呼叫一個評估器的 evaluate 功能,你告訴它代理模組在哪、測試檔在哪,它就把測試檔當成整合測試來跑。命令列介面則是用代理模組的路徑加上評估集檔來做自動化評估,你可以指定設定檔,也可以叫它印出詳細結果;而且在比較大的評估集裡,你還可以在檔名後面用逗號分隔,列出你只想跑的那幾個特定評估,不必每次全部跑一遍。

好,我們把整章的脈絡再串一次,做個收尾。

先講「是什麼」。代理式系統跟大型語言模型,是運作在複雜又動態的環境裡的,它們的表現會隨時間退化。而且它們的本質是機率性的、非確定性的——非確定性英文是 non-deterministic,意思就是同樣輸入不保證同樣輸出——這個特性讓傳統軟體測試不足以確保它們的可靠性。多代理系統因為跟環境不斷互動變動,需要可調適的測試方法,還有能衡量「協作成功」的精密指標。而像資料漂移、非預期的互動、工具呼叫出錯、偏離目標這些問題,常常是「部署之後」才浮現的,所以持續評估有它的必要。

再講「為什麼」。一套標準化的評估與監控框架,給你的是一個有系統的方式,去確保代理的表現能持續穩定。具體來說,它幫你為準確度、延遲、資源消耗——像是 token 用量——定義清楚的指標;也包含那些進階技術,例如分析軌跡來理解推理過程、用 LLM 即評審來做質性評估。然後透過回饋迴路跟報告系統,這個框架能促成持續改進、能做 A/B 測試、能偵測異常跟效能漂移,確保代理始終跟你的目標對齊。

最後講「經驗法則」,也就是什麼時候該用這套模式。有五種情況。第一,當你在即時的正式環境部署、而且可靠性至關重要的時候。第二,當你需要系統化地比較不同代理或不同模型版本、好推動改進的時候。第三,當你身處受監管或高風險的領域、需要做合規、安全跟倫理稽核的時候。第四,當表現可能因為資料或環境變化、也就是漂移而退化的時候。第五,當你要評估複雜的代理行為、像是動作軌跡或是有用性這種主觀的輸出品質的時候。這五種,都該用上評估與監控這個設計模式。

那我幫你把整章最核心的精神濃縮成一句話收尾。要有效評估一個 AI 代理,你必須超越「單純檢查它答對沒」這個層次,轉而對它在動態環境中的表現,做持續的、多面向的評估:既要實務地監控延遲跟資源消耗,也要透過比對「實際軌跡」跟「理想基準真相路徑」,去精密分析它的決策過程。對於「有用性」這種細緻特質,LLM 即評審會愈來愈不可或缺;而像 Google ADK 這樣的框架,則用測試檔來做單元測試、用評估集檔來做整合測試,提供結構化的工具。面對多代理系統,焦點就轉向評估它們的協作成功與否。而放眼未來,為了確保關鍵應用的可靠性,整個典範正在從「提示驅動的代理」,轉向受形式化「合約」約束的「承包商」——讓 AI 以精確、可驗證的條款運作,能協商、能分解任務、能自我驗證,從一個不可預測的工具,進化成一個當責又可信的系統。這項演進,正是我們要在關鍵任務領域,部署精密代理式 AI 所需要的那份「信任」的基礎。
