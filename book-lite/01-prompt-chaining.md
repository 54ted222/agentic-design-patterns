# 第 1 章:提示鏈(Prompt Chaining)

## 提示鏈模式總覽

提示鏈(Prompt Chaining),又稱管線模式(Pipeline pattern),是運用 LLM 處理複雜任務的強大範式。它不期望 LLM 在單一龐大步驟中解決問題,而採「分而治之」策略:把大問題拆成一連串較小、易掌控的子問題,各以專門的提示處理,並把前一個提示的輸出當作下一個提示的輸入。

這種循序處理為與 LLM 的互動帶來模組化與清晰度。拆解後,每一步都更容易理解、除錯與最佳化,專注於問題的特定面向,產出更精準的結果。前一步輸出作為後一步輸入,建立起「依賴鏈」——「鏈」之名由此而來——讓先前的情境引導後續處理,使模型逐步逼近解答。

提示鏈也讓外部知識與工具的整合成為可能:每一步都可指示 LLM 與外部系統、API 或資料庫互動,讓能力超越內部訓練資料的範圍。因此它不只是問題解決技巧,更是建構 AI 代理(AI agent)的基礎——代理能藉此在動態環境中自主規劃、推理與行動,以更貼近人類思考歷程的方式完成多步推理與決策任務。

**單一提示的侷限:** 面對多面向任務,單一複雜提示往往效率不彰,容易導致:指令忽略(部分內容被忽視)、情境漂移(逐漸偏離原情境)、錯誤傳播(早期錯誤被放大)、情境視窗不足,以及幻覺(認知負荷增加而產生錯誤資訊)。例如一個要求「分析市場研究報告、摘要發現、找出有數據支撐的趨勢,並草擬郵件」的查詢,模型也許摘要做得不錯,卻無法妥善擷取數據或草擬郵件。

**透過循序拆解提升可靠性:** 提示鏈把上述任務拆成聚焦的工作流程:

1. **第一個提示(摘要):**「請摘要以下市場研究報告的關鍵發現:[文字]。」唯一焦點是摘要,提高起始步驟的準確度。
2. **第二個提示(趨勢辨識):**「根據此摘要,找出前三大新興趨勢,並擷取支撐每項趨勢的具體數據:[步驟 1 的輸出]。」此提示受更明確約束,且建立在已驗證的輸出之上。
3. **第三個提示(撰寫郵件):**「請撰寫一封簡潔的電子郵件給行銷團隊,概述以下趨勢及其支撐數據:[步驟 2 的輸出]。」

每一步更簡單、更不模稜兩可,降低認知負荷,帶來更可靠的最終輸出,就像運算管線中各函式各司其職、依序傳遞結果。我們還可為每階段賦予明確角色,例如依序指定為「市場分析師」「交易分析師」「專家文件撰寫者」。

**結構化輸出的角色:** 提示鏈的可靠性高度仰賴步驟間傳遞資料的完整性。若某步輸出含糊或格式不佳,後續步驟就可能因輸入有誤而失敗。因此指定結構化輸出格式(如 JSON 或 XML)至關重要。例如趨勢辨識步驟的輸出可格式化為:

```json
{
  "trends": [
    {
      "trend_name": "AI 驅動的個人化",
      "supporting_data": "73% 的消費者偏好與會運用個人資訊、使購物體驗更貼近自身需求的品牌往來。"
    },
    {
      "trend_name": "永續且符合道德的品牌",
      "supporting_data": "過去五年間,帶有 ESG 相關訴求的產品銷售成長了 28%,相較之下不帶此類訴求者為 20%。"
    }
  ]
}
```

結構化格式確保資料是機器可讀的,能被精確解析並插入下一個提示,把詮釋自然語言可能產生的錯誤降到最低,是建構穩健多步驟 LLM 系統的關鍵。

## 實務應用與使用案例

提示鏈的核心效用在於把複雜問題拆成循序、可掌控的步驟,適用範圍廣泛:

**1. 資訊處理工作流程:** 許多任務涉及對原始資訊的多次轉換,例如摘要文件、擷取實體,再用實體查詢資料庫或生成報告:

- 提示 1:從指定 URL 或文件擷取文字內容。
- 提示 2:摘要清理過後的文字。
- 提示 3:從摘要或原文擷取特定實體(人名、日期、地點)。
- 提示 4:用這些實體搜尋內部知識庫。
- 提示 5:整合摘要、實體與搜尋結果,生成最終報告。

這套方法用於自動化內容分析、AI 研究助理與複雜報告生成。

**2. 複雜問題回答:** 適用於需多步推理或多次檢索的問題,例如:「1929 年股市崩盤的主要成因為何?政府政策又如何回應?」

- 提示 1:辨識查詢中的核心子問題(崩盤成因、政府回應)。
- 提示 2:研究 1929 年崩盤的成因。
- 提示 3:研究政府對該崩盤的政策回應。
- 提示 4:綜整步驟 2 與 3 的資訊,回應原始查詢。

當查詢無法靠單一資料點回答,而需一連串邏輯步驟或整合多來源資訊時,就需要這類系統。例如自動化研究代理會先平行檢索並擷取多篇文章的關鍵資訊(獨立子任務適合平行處理以最大化效率);擷取完成後,流程轉為循序——彙整資料、綜整成初稿、審閱精煉成最終報告,後階段各自依賴前一階段。因此複雜操作常把「用於獨立蒐集的平行處理」與「用於相依步驟的提示鏈」結合。

**3. 資料擷取與轉換:** 把非結構化文字轉成結構化格式,通常需迭代修正以提升準確度:

- 提示 1:從發票文件擷取特定欄位(姓名、地址、金額)。
- 處理:檢查必填欄位是否齊全且符合格式。
- 提示 2(條件式):若有缺漏或格式錯誤,擬新提示專門尋找缺漏資訊,可附前次失敗的情境。
- 處理:再次驗證,必要時重複。
- 輸出:已擷取、已驗證的結構化資料。

此法特別適合從表單、發票、郵件等來源擷取資料。例如處理 PDF 表單的 OCR 問題:先用 LLM 對文件影像擷取文字,再正規化資料(如把「一千零五十」轉為 1050)。由於精確數學運算對 LLM 是挑戰,後續可把算術委派給外部計算機工具——LLM 辨識需計算之處,餵入正規化數字,再整合精確結果。這條「文字擷取、資料正規化、外部工具使用」的鏈,能達成單次查詢難以穩定取得的準確結果。

**4. 內容生成工作流程:** 複雜內容撰寫是程序性任務,通常拆成發想、大綱、初稿與修訂等階段:

- 提示 1:根據使用者興趣生成 5 個主題構想。
- 處理:由使用者或自動挑選最佳構想。
- 提示 2:根據選定主題生成詳細大綱。
- 提示 3:根據大綱第一個重點撰寫初稿段落。
- 提示 4:根據第二個重點撰寫初稿,並附前段作為情境;持續處理所有重點。
- 提示 5:審閱並精煉整份初稿的連貫性、語氣與文法。

此法用於創意敘事、技術文件等結構化文字內容的自動撰寫。

**5. 具狀態的對話式代理:** 完整狀態管理雖比循序串接複雜,提示鏈仍提供維繫對話連續性的基礎機制:把每一輪建構成新提示,有系統地納入先前互動的資訊或實體以維持情境。

- 提示 1:處理使用者第一段話語,辨識意圖與關鍵實體。
- 處理:以該意圖與實體更新對話狀態。
- 提示 2:根據當前狀態生成回應,並辨識下一個所需資訊。
- 後續各輪重複,每段新話語都啟動一條鏈,並運用累積的對話歷史(狀態)。

保留對話歷史讓系統能理解並回應依賴先前交換資訊的輸入,是對話式代理在多輪對話中維持情境的根本。

**6. 程式碼生成與精煉:** 生成可運作程式碼通常是多階段過程:

- 提示 1:理解使用者需求,生成虛擬碼或大綱。
- 提示 2:根據大綱撰寫初版程式碼。
- 提示 3:找出潛在錯誤或可改進之處(可用靜態分析或另一次 LLM 呼叫)。
- 提示 4:根據問題重寫或精煉程式碼。
- 提示 5:加入文件說明或測試案例。

提示鏈把複雜編碼任務拆成可掌控的子問題,降低每一步的操作複雜度。關鍵在於它允許在模型呼叫之間插入確定性(deterministic)邏輯,進行中介資料處理、輸出驗證與條件分支,把單一多面向請求轉換成由執行框架管理的結構化操作序列。

**7. 多模態與多步推理:** 分析多模態(modality)資料集需把問題拆成以提示為基礎的子任務。例如解讀一張含嵌入文字、標籤與表格資料的影像:

- 提示 1:從影像請求中擷取並理解文字。
- 提示 2:把擷取的影像文字與對應標籤連結。
- 提示 3:運用表格詮釋蒐集到的資訊,判定所需輸出。

## 動手實作範例

提示鏈的實作可小至腳本中循序呼叫函式,大至運用專門框架管理控制流程、狀態與元件整合。LangChain、LangGraph、Crew AI 與 Google 的 Agent Development Kit(ADK)等框架提供結構化環境,對複雜架構尤其有利。

本範例採用 LangChain 與 LangGraph:其核心 API 明確針對「組合操作的鏈與圖」設計。LangChain 提供線性序列的基礎抽象,LangGraph 則擴展為支援具狀態與循環的運算。以下示範一個基本線性序列——一條兩步驟提示鏈:第一階段解析非結構化文字並擷取資訊,第二階段把輸出轉換成結構化格式。

先安裝所需函式庫:

```bash
pip install langchain langchain-community langchain-openai langgraph
```

`langchain-openai` 可替換成對應其他模型供應商的套件。接著須在執行環境設定所選供應商(OpenAI、Google Gemini 或 Anthropic)的 API 憑證。

```python
import os
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# 為提高安全性,從 .env 檔案載入環境變數
# from dotenv import load_dotenv
# load_dotenv()
# 請確認 OPENAI_API_KEY 已設定在 .env 檔案中

# 初始化語言模型(建議使用 ChatOpenAI)
llm = ChatOpenAI(temperature=0)

# --- 提示 1:擷取資訊 ---
prompt_extract = ChatPromptTemplate.from_template(
    "從以下文字中擷取技術規格:\n\n{text_input}"
)

# --- 提示 2:轉換成 JSON ---
prompt_transform = ChatPromptTemplate.from_template(
    "請將以下規格轉換成一個 JSON 物件,"
    "以 'cpu'、'memory' 與 'storage' 作為鍵:\n\n{specifications}"
)

# --- 使用 LCEL 建構鏈 ---
# StrOutputParser() 會把 LLM 的訊息輸出轉換成單純的字串。
extraction_chain = prompt_extract | llm | StrOutputParser()

# 完整的鏈會把擷取鏈的輸出,傳入轉換提示的 'specifications' 變數。
full_chain = (
    {"specifications": extraction_chain}
    | prompt_transform
    | llm
    | StrOutputParser()
)

# --- 執行鏈 ---
input_text = "這款新筆電搭載 3.5 GHz 八核心處理器、16GB 記憶體,以及 1TB NVMe SSD。"

# 以含輸入文字的字典執行這條鏈。
final_result = full_chain.invoke({"text_input": input_text})

print("\n--- Final JSON Output ---")
print(final_result)
```

這段程式碼運用兩個獨立提示:一個從輸入字串擷取技術規格,另一個把規格格式化為 JSON。互動使用 `ChatOpenAI`,`StrOutputParser` 確保輸出為可用字串。範例以 LangChain 表達式語言(LCEL)串接提示與模型:`extraction_chain` 負責擷取規格,`full_chain` 取其輸出作為轉換提示的輸入,讓筆電描述文字經兩步處理,最終印出包含已擷取並格式化規格的 JSON 字串。

## 情境工程與提示工程

情境工程(Context Engineering,見圖 1)是一門系統化學科,旨在模型生成 token 之前,為其設計、建構並交付完整的資訊環境。其核心主張是:模型輸出的品質,較少取決於模型架構本身,更多取決於所提供情境的豐富程度。

![圖 1:情境工程是一門為 AI 建構豐富、完整資訊環境的學科,因為此情境的品質是促成進階代理表現的首要因素。](assets/01-prompt-chaining/fig-1-context-engineering.png)

*圖 1:情境工程是一門為 AI 建構豐富、完整資訊環境的學科,因為此情境的品質是促成進階代理(Agentic)表現的首要因素。*

相較於主要著重最佳化使用者當下查詢措辭的傳統提示工程(Prompt Engineering),情境工程是一項重大演進,把範圍擴展到多個資訊層次:系統提示(system prompt,定義 AI 運作參數的基礎指令,如「你是技術文件撰寫者;語氣須正式且精確」)、外部資料(檢索到的文件、工具輸出),以及關鍵的隱性資料(使用者身分、互動歷史、環境狀態)。其核心原則是:即使先進模型,在只獲得對運作環境之拙劣視野時也會表現不佳。

這把任務從「回答問題」重新定位為「為代理建構完整的運作全貌」。例如經情境工程的代理會先整合使用者的行事曆空檔(工具輸出)、與郵件收件人的專業關係(隱性資料),以及先前會議筆記(檢索文件),才產生高度相關、個人化且實用的輸出。其中「工程」涉及建立穩健管線在執行階段擷取並轉換這些資料,並建立回饋迴路持續改善情境品質。

實作上可運用專門調校系統大規模自動化改善過程。例如 Google 的 Vertex AI prompt optimizer 能依據範例輸入與評估指標有系統地評估回應,藉此提升表現,對於跨模型調適提示與系統指令而無需大量人工重寫相當有效。

這種結構化做法把情境本身視為首要元件,重視代理「知道什麼、何時知道、如何運用」,正是區分「簡陋 AI 工具」與「具情境感知系統」的關鍵,也是把無狀態(stateless)聊天機器人推進為高能力系統的核心方法。

## 重點速覽

**是什麼(What):** 把複雜任務塞進單一提示時,往往讓 LLM 不堪負荷:認知負荷增加會提高各種錯誤的可能性,如忽略指令、遺失情境、產生錯誤資訊。單體式提示難以管理多重限制與循序推理,造成不可靠、不準確的輸出。

**為什麼(Why):** 提示鏈把複雜問題拆成一連串較小、相連的子任務,每一步以聚焦提示執行特定操作,輸出作為下一步輸入,構成邏輯工作流程,大幅提升可靠性與可控性。這種模組化、分而治之的策略更易管理與除錯,並允許整合外部工具或結構化資料格式,是建構能規劃、推理並執行複雜工作流程之多步驟代理系統的基礎。

**經驗法則(Rule of thumb):** 當任務複雜到單一提示無法勝任、涉及多個處理階段、需在步驟間與外部工具互動,或正在建構需多步推理並維持狀態的代理系統時,就使用此模式。

## 結論

提示鏈把複雜問題解構成一連串較簡單的子任務,為引導 LLM 提供穩健框架。這種「分而治之」策略讓模型一次專注一個操作,大幅提升輸出的可靠性與可控性。重點整理:

- 提示鏈把複雜任務拆成較小、聚焦的步驟,有時稱為管線(Pipeline)模式。
- 每一步涉及一次 LLM 呼叫或處理邏輯,並以前一步輸出作為輸入。
- 此模式提升複雜 LLM 互動的可靠性與可管理性。
- LangChain/LangGraph 與 Google ADK 等框架提供穩健工具來定義、管理並執行這些多步驟序列。

作為基礎模式,提示鏈使開發具備多步推理、工具整合與狀態管理能力的精密 AI 代理成為可能,是建構穩健、具情境感知系統的關鍵,能耐遠超單一提示所能企及。
