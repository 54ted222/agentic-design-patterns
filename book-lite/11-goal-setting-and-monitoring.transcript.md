這一章我們要談的,是讓 AI 代理真正「有方向、知進退」的一個設計模式,叫做目標設定與監控,英文是 Goal Setting and Monitoring。在開始之前,我先幫你畫一條路線圖:我們會先弄清楚這個模式到底要解決什麼問題,接著用一個旅行的比喻把核心觀念講透,然後一口氣走過六個實際的應用場景,再用一個會自己寫程式、又會自己評分的 AI 代理範例,讓你親耳「聽懂」整段程式在做什麼。最後我們會收斂出幾個你一定要記住的原則,包括一個叫做 SMART 的目標設定法則。

我們先從問題本身說起。一個 AI 代理,光是會處理資訊、會呼叫工具,其實還不夠。你可以把它想成一個很會用各種軟體、反應很快的助理,但如果你沒告訴他「我們今天到底要達成什麼」,他就只能被動地一件事接一件事地處理,沒辦法判斷自己是不是真的在往成功靠近。這種「有刺激才有反應」的狀態,我們叫它反應式系統,英文是 reactive。目標設定與監控這個模式,核心就是兩件事:第一,給代理一個具體、明確的目標;第二,給它一套能追蹤進度、判斷自己達成了沒有的手段。有了這兩樣,代理才從被動的反應式,變成主動朝著目標前進的系統。

那要怎麼理解這個模式呢?我用一個你一定有經驗的場景來打比方,就是規劃一趟旅行。你規劃旅行的時候,腦子裡其實做了好幾件事。首先你會定下目的地,這就是所謂的「目標狀態」,也就是你希望最後抵達的地方。接著你會確認你現在人在哪裡、手上有什麼資源,這是「初始狀態」,也就是你的出發點。然後你會開始考量各種選項,要搭什麼交通工具、走哪條路線、預算多少。最後,你會把這些考量排成一連串顧及先後順序、顧及各種限制的步驟。這整個「從現在這裡,一步一步推進到目的地」的過程,在代理系統裡就叫做規劃,英文是 planning。

實務上,代理通常是先接到一個很高層次、比較籠統的目標,然後它會自己、或者半自動地,把這個大目標拆解成一連串比較小的子目標,英文叫 sub-goal。拆解完之後,它再一個一個照順序執行,或者用更複雜的流程去跑。在這個過程中,它可以搭配使用工具、可以做路由判斷該把任務交給誰、也可以找其他代理一起協作。而「怎麼規劃」這件事本身,可以用傳統的搜尋演算法、可以用邏輯推理,而現在愈來愈常見的做法,是直接讓大型語言模型,也就是我們常說的 LLM,來生成這份計畫。

好的規劃帶來兩個關鍵好處。第一,它讓代理有能力處理那種要好幾個步驟才能完成的請求,而不是只能回答一句問一句。第二,它讓代理可以「重新規劃」,英文是 replanning。意思是當情況變了、原本的計畫走不通了,代理能夠當場調整、重新安排路線。你可以想像導航軟體,你開錯了一個出口,它不會當機,而是馬上幫你重算一條新路。就是這種重新規劃的能力,把一個只會被動反應的系統,真正轉變成一個會主動朝目標努力的系統。

觀念講完了,我們來看這個模式在真實世界裡到底用在哪些地方。我會帶你走過六個場景,你會發現它們有一個共同的骨架:都是先定一個目標,然後持續監控,再根據監控結果決定下一步。

第一個場景是客戶支援自動化。目標可能是「解決這位客戶的帳單疑問」。代理會一邊監控對話的進展,一邊去查資料庫核對資料,需要的時候用工具去調整帳單。那它怎麼知道自己成功了?它的判定標準是:帳單確實被改好了,而且客戶也給了正面的回饋。如果問題一直解不掉,它就會做一件很重要的事,叫做升級,英文是 escalate,也就是把這個案子轉交給真人處理。

第二個場景是個人化學習系統。目標可能是「提升某位學生對代數的理解程度」。這時候代理要追蹤的,是學生做練習的進度、答對的準確率、還有完成題目花的時間。一旦發現學生卡關了,它就調整教材的難度跟教學的方法。

第三個場景是專案管理助理。目標可能是「確保某個里程碑在某個日期前完成」。代理會監控每一項任務的狀態、團隊之間的溝通、還有資源夠不夠用。一旦它嗅到有延遲的風險,就會標示出來,並且主動建議該怎麼修正。

第四個場景是自動化交易機器人。目標可能是「在可以承受的風險範圍內,把收益最大化」。它會持續監控市場資料、整個投資組合現在值多少錢、以及各種風險指標。當條件符合,它就出手交易;一旦風險衝破了預設的門檻,它就調整策略。

第五個場景是機器人跟自駕車。目標可能是「安全地把乘客從 A 點載到 B 點」。它要不停地監控周遭環境、自己車輛的狀態、以及行程走到哪了,然後即時調整駕駛行為。

第六個場景是內容審核。目標可能是「辨識並移除有害的內容」。代理會監控傳進來的每一則內容,套用分類模型去判斷,同時還要追蹤自己的誤判狀況。這裡有兩個專有名詞要說清楚:一個是誤報,英文叫 false positive,意思是把好的內容錯當成壞的;另一個是漏報,英文叫 false negative,意思是把壞的內容放過去了。代理會根據這些誤判去調整過濾的標準,遇到模稜兩可、自己拿不準的案子,就升級交給真人來判斷。

你把這六個場景擺在一起看,會發現它們其實都在示範同一件事:這個模式提供了一個讓代理「自我管理」的框架,讓它能可靠地交出成果,又能跟著動態變化的條件去適應。

接下來是這一章最精彩的部分,我們要看一個真正動手做的範例。我會強調一次,因為你現在是用聽的,看不到螢幕上的程式碼,所以接下來我不會一行一行唸語法,而是告訴你「每一段在做什麼、輸入是什麼、輸出是什麼、為什麼要這樣設計」。

這個範例是用 LangChain 這個框架,搭配 OpenAI 的 API 寫成的一支 Python 腳本。它本身就是一個自主的 AI 代理,任務是生成並且精煉 Python 程式碼,同時確保產出的程式符合使用者事先定義好的品質標準。它運作的方式,是一個不斷重複的循環,這個循環有三個動作:先創作、再自我評估、然後改善。創作出來的東西好不好,是由代理自己用 AI 的判斷來衡量的,看看有沒有滿足目標。最後,它會輸出一個帶有註解、可以立刻拿來用的 Python 檔案。要跑這支腳本,你需要先安裝幾個套件,主要是 LangChain 跟 OpenAI 相關的函式庫,還有一個用來讀取環境設定的小工具;另外你還要準備一個設定檔,裡面放著你的 OpenAI 金鑰。

我給你一個貫穿整段的比喻,你抓住這個比喻,後面就都好懂了。你可以把這支腳本想像成一位被指派了專案的自主 AI 程式設計師。你交給他兩樣東西:第一樣是一份詳細的專案說明書,也就是你要他解決的那個具體的程式設計問題;第二樣是一份嚴格的品質檢查清單,英文叫 quality checklist,它代表最終的程式碼一定要滿足的那些目標,比方說「程式必須簡單好懂」「功能要正確」「要能處理各種邊界情況」。記住這兩樣東西:說明書是任務,清單是目標。

現在我帶你聽聽這位 AI 程式設計師的內部是怎麼組裝起來的,它由幾個分工明確的小零件構成。

第一個零件,負責「組裝提示詞」。它的輸入是那份使用案例、那份目標清單,還有上一輪產出的舊程式碼跟上一輪收到的回饋。它做的事,是把這些材料拼成一段給 LLM 的指令,大意是說:你是一個 AI 程式設計代理,請根據這個使用案例寫 Python 程式,而且你的程式要滿足以下這幾個目標。如果這不是第一輪,它還會貼心地把上一版的程式碼跟上一版的回饋一起附上去,讓模型知道「上次做到哪、上次哪裡被嫌」,這樣才能在原有基礎上改進,而不是每次都從零開始。它最後還會交代一句:請只回傳修改後的程式碼,不要多寫說明。這樣輸出才乾淨。

第二個零件,負責「拿到對程式碼的回饋」。它的角色設定是一位 Python 程式碼審查員。輸入是一段程式碼,加上那份目標清單。它要 LLM 去評論這段程式,並且明確指出:這些目標到底達成了沒有,還需不需要在清晰度、簡潔性、正確性、邊界情況的處理、或是測試覆蓋率這幾個面向上再改進。它的輸出,就是一段文字形式的審查意見。

第三個零件最關鍵,它負責「判斷目標到底達成了沒」。它的輸入是上一步那段審查意見,還有目標清單。它的巧妙之處在於:它要求 LLM 扮演一位審查員,讀完那段回饋之後,只用一個詞回答,要嘛是 True、要嘛是 False。為什麼要硬性規定只能回答這兩個字?因為這是一個讓程式能自動判斷「該不該停下來」的開關。程式拿到這個回答之後,只要比對它是不是「true」,就知道要不要結束這個循環。這一招,就是把「監控」跟「成功判定」這件事,具體落實到程式裡的做法。

剩下還有幾個比較雜務性的小零件,我快速帶過,你知道它們存在就好。有一個負責清理程式碼,因為 LLM 回傳的內容常常會被包在標記符號裡,它就把那些多餘的外框去掉。有一個負責在最終程式碼的最上面,加上一段註解標頭,說明這段程式實作的是哪個使用案例。還有一個負責把一段文字轉成適合當檔名的格式。最後一個負責存檔,它甚至會再請 LLM 幫忙,把使用案例濃縮成一個很短的字當檔名,再加上一串隨機數字避免重複,然後把程式碼寫進這個檔案。

把這些零件組合起來的,是主代理的那段流程。它的運作是這樣的:先把使用者輸入的那串目標,用逗號切開成一條一條的目標。然後它進入一個最多跑五次的迴圈。為什麼要設上限五次?因為這是一個安全閥,免得它沒完沒了地一直跑下去。在每一輪裡,它做的事情就是我們剛剛說的那條鏈:先組裝提示詞、生成程式碼、清理乾淨、然後把程式碼送去做回饋審查、拿到審查意見、再丟給那個只回答 True 或 False 的判定零件。如果回答是 True,代表 LLM 確認目標達成了,迴圈就立刻停下來。如果還沒達成,它就把這一輪的程式碼記下來當成「上一版」,帶進下一輪繼續精煉。等到迴圈結束,不管是達標而停、還是跑滿五次而停,它都會把最終的程式碼加上註解標頭,然後存成檔案。腳本最後還附了幾個現成的測試例子,其中一個是請它寫一段程式去找出某個正整數的「二進位間隙」,並且要求程式簡單好懂、功能正確、能處理各種邊界情況、只接受正整數輸入、還要印出幾個範例。

我們把這位 AI 程式設計師的工作態度再串一次你就更清楚了。他先產出一份草稿,但他不會立刻交出去。他會拿著那份品質清單,一項一項地對照、自我審查,英文叫 self-review。這時候他其實是一人分飾兩角,自己又當作者、又當品管檢查員,然後做出裁決:如果每一項都符合,就回答 True;只要有一項不符合,就回答 False。回答 False 的話,他就進入修訂階段,把程式碼重寫一遍。這個「起草、自我審查、精煉」的循環會一直跑,直到拿到 True、或是撞到次數上限為止。通過之後,才把程式碼加上註解、存成檔案。

不過這裡有一段非常重要的注意事項,我希望你務必聽進去,因為它點出了這個簡單做法的真正限制。首先,這只是一個示範,它不是可以直接搬到正式生產環境去用的程式。在真實應用裡,你必須提防幾件事:第一,LLM 可能根本誤解了你的目標;第二,它可能把表現得不好的東西錯誤地評為成功;第三,就算它真的理解了目標,它仍然可能產生幻覺,英文叫 hallucinate,也就是一本正經地胡說八道。而且最關鍵的盲點是:當同一個 LLM 既負責寫程式、又負責評判自己寫得好不好的時候,它特別難以察覺自己其實走錯了方向。這就像讓學生自己改自己的考卷,他很容易看不到自己的盲點。所以結論是,你還是必須自己把產出的程式真的拿去執行、真的去測試。另外要提醒,這個範例裡的「監控」其實相當基礎,而且它還帶著一個風險,就是有可能會「永遠跑下去」停不下來。

那為了示範什麼叫好的審查,書裡還給了一段審查員的提示詞範本,我用講的幫你抓重點。它要求 LLM 扮演一位資深的程式碼審查員,而且核心使命寫得很重,是「消除程式碼裡的幻覺」,也就是確保每一項建議都立基於現實跟最佳實務。它具體要求審查員做四件事:第一,找出並修正錯誤,包括邏輯瑕疵、臭蟲、還有可能在執行時才爆出來的錯誤;第二,在不犧牲正確性的前提下,簡化跟重構程式碼,讓它更好讀、更有效率、更好維護;第三,對每一項建議都要給清楚的說明,解釋它為什麼算是改進,並且援引乾淨程式碼、效能或安全性的原則;第四,要同時展示「修改前」跟「修改後」,讓改進的地方一目瞭然。整體精神是:回饋要直接、要有建設性,而且永遠以提升品質為目標。

講到這裡,你應該已經感覺到剛剛那個盲點的解法了。既然「自己改自己」靠不住,那更穩健的做法,就是用一組代理,英文叫 crew,讓它們各司其職,把職責分開來。書裡舉了一個用 Gemini 建立的團隊當例子,這個團隊有五種角色。第一位是同儕程式設計師,負責協助寫程式跟一起腦力激盪。第二位是程式碼審查員,專門抓錯誤、提改進建議。第三位是文件撰寫者,負責生成清楚又精簡的文件。第四位是測試撰寫者,負責建立全面的單元測試。第五位是提示精煉者,負責把跟 AI 互動的提示詞最佳化。這樣分工最大的價值在於:因為審查員是獨立於程式設計師之外的另一個角色,它對品質的客觀評估就大幅提升了,不再是自己改自己的考卷。而且測試撰寫者的存在,也自然而然地滿足了「要為產出的程式碼寫單元測試」這個需求。

好,我們把整章的精華收斂一下。先講這個模式「是什麼」。AI 代理常常缺乏明確的方向,所以在簡單的反應式任務之外,它就沒辦法帶著目的性去行動。沒有既定的目標,它就獨立處理不了複雜的多步驟問題、協調不了精密的工作流程,也沒有任何機制去判斷自己的行動到底有沒有導向成功。

再講「為什麼」要用它。這個模式做的事,是把「目的感」跟「自我評估」這兩樣東西嵌進代理裡面。它明確定義出清楚、可衡量的目標,並且建立一套會持續對照目標、追蹤進度跟環境狀態的監控機制。這兩者合起來,形成了一個關鍵的回饋迴路,英文叫 feedback loop。就是靠這個迴路,代理才能評估自己的表現、修正方向、在偏離的時候調整計畫,從而把一個被動的反應式代理,轉變成一個自主、可靠、主動以目標為導向的系統。

那什麼時候該用它?經驗法則是這樣:當你的 AI 代理必須自主執行多步驟的任務、必須適應動態變化的條件,而且要在沒有人持續盯著、持續介入的情況下,可靠地達成某個高層次的目標時,你就該用這個模式。

最後,有幾個重點我希望你帶走。第一,目標設定與監控,本質上就是替代理配備了「目的感」跟「追蹤進度的機制」這兩件法寶。第二,你設定的目標最好符合一個叫做 SMART 的法則。SMART 是五個英文字的縮寫,我幫你逐一拆開:S 是 Specific,具體的;M 是 Measurable,可衡量的;A 是 Achievable,可達成的;R 是 Relevant,相關的;T 是 Time-bound,有時限的。一個好目標,應該同時滿足這五點。第三,清楚地定義出指標跟成功的標準,是有效監控的關鍵;如果連「怎樣算成功」都講不清楚,監控就無從談起。第四,監控的內容,包含觀察代理的行動、環境的狀態、還有工具的輸出這三方面。第五,監控所產生的那個回饋迴路,正是讓代理能夠適應、能夠修訂計畫、或是能夠把難題升級處理的根源。第六,如果你用的是 Google 的 ADK,也就是 Agent Development Kit,代理開發套件,那麼在它裡面,目標是透過「代理指令」來傳達的,而監控則是藉由「狀態管理」跟工具互動來達成的。

我們用一句話來收束這一整章。目標設定與監控,做的事情就是把 AI 代理從一個被動的反應式系統,轉變成一個主動的、由目標驅動的實體。而這件事成不成功,關鍵就在於兩點:一是定義出清楚、可衡量的目標,二是建立嚴謹的監控程序去追蹤進度。賦予代理「自己制定目標、又自己監督目標」的能力,正是邁向一個真正智慧、而且能夠被究責的 AI 系統,最根本的一步。
