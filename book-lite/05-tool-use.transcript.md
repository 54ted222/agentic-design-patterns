歡迎來到第五章,這一章我們要談一個讓代理真正「動起來」的關鍵能力,叫做工具使用,英文是 Tool Use,它在技術上最常見的實作方式,叫做函式呼叫,英文是 Function Calling。

在進入正題之前,我先幫你把這一章的路線圖說一遍,讓你心裡有個底。我們會先弄清楚一件事:為什麼語言模型光會講話還不夠,一定要會用工具。接著我會帶你走過工具使用的完整運作流程,總共有六個步驟。然後我會說明一個比函式呼叫更寬廣的概念,叫做工具呼叫。再來,我們會看六個實務上的使用場景,讓你知道這個模式到底能拿來做什麼。之後,我會用純口語的方式,帶你理解四段實作範例,分別來自三個不同的框架:LangChain、CrewAI,還有 Google 的代理開發套件。最後,我們會用「是什麼、為什麼、什麼時候用」這三個角度,把整章的重點收斂起來。

我們先從一個根本問題說起。在前面幾章,我們討論過好幾種代理模式,像是把多個步驟串接起來、把任務路由到不同處理者、讓多個任務平行處理,還有讓模型回頭反思自己的輸出。這些模式有一個共通點:它們都是在「編排」語言模型之間怎麼互動、資訊怎麼流動。換句話說,它們處理的都是模型內部的世界。但問題來了,如果你希望代理不只是在自己腦袋裡打轉,而是要真的去跟外面的世界、跟外部系統互動,那它就必須具備使用工具的能力。這就是這一章要解決的核心問題。

那什麼叫工具使用?我給你一個生活化的比喻。你可以把語言模型想成一個知識淵博、口才很好的人,但他被關在一個沒有窗戶、沒有網路、沒有電話的房間裡。他能跟你聊很多事情,可是他不知道今天台北的天氣,因為他出不去;他也沒辦法幫你寄一封信,因為他碰不到郵局。工具使用,就是在這個房間裡裝上一支電話、一台電腦、一扇通往外界的門。有了這些工具,這個人就能查到即時資訊、能幫你做事,而不只是空談。

接下來,我帶你走過工具使用的完整運作流程,這是這一章最核心的骨架,總共六個步驟,你可以把它想成一條接力的流水線。

第一步,叫做工具定義。意思是,你要先把外部的某個功能,清楚地描述給語言模型聽。你要告訴它這個工具叫什麼名字、是做什麼用的、需要哪些輸入參數、每個參數又是什麼型別。你可以想像成,你要寫一張工具的使用說明書交給模型,讓它知道手上有哪些工具、各自能幹嘛。

第二步,叫做模型決策。當使用者提出一個請求,模型會根據這個請求,再對照它手上有哪些可用的工具,自己判斷:這件事我需不需要用工具?需要用哪一個?甚至需不需要連續用好幾個?這個決策是模型自己做的,這一點很關鍵。

第三步,叫做函式呼叫生成。如果模型決定要用某個工具,它不會直接去執行,它做的事情是「生成一份結構化的指令」。這份指令通常長得像一種叫做 JSON 的資料格式,你可以把它理解成一張填好的表單,上面寫清楚了要呼叫哪一個工具,以及從使用者的問題裡擷取出來的參數該填什麼。舉個例子,使用者問「倫敦的天氣如何」,模型就會生成一張表單,上面寫著「要呼叫天氣查詢工具,查詢的城市是倫敦」。

第四步,叫做工具執行。這時候,有一個我們稱之為「編排層」的東西會出場。它會攔截住模型剛剛生成的那張表單,看懂上面寫的內容,然後拿著裡面的參數,真正地去執行那個函式。請特別注意,真正動手執行的是這個編排層,不是模型本身。模型只負責「決定要做什麼、把表單填好」,實際動手的是外面的程式。

第五步,叫做觀察,或者說結果回傳。工具執行完之後,會產生一個輸出,這個輸出會被送回給代理。延續剛剛的例子,天氣工具查完之後,回傳「倫敦目前多雲,氣溫攝氏十五度」,這個結果就會交回到流程裡。

第六步,叫做模型處理。這一步是選用的,但實務上非常常見。模型拿到工具回傳的結果之後,會把它當成新的情境資訊,接著決定下一步怎麼走。它可能直接根據這個結果,擬出一個給使用者的友善回答;也可能覺得資訊還不夠,再去呼叫另一個工具;或是回頭反思一下再決定。

把這六步串起來你就會發現,這個模式之所以這麼基礎、這麼重要,是因為它打破了語言模型的一個天生限制。模型的知識,是來自它訓練時看過的資料,而那些資料是靜態的、有時間截止點的。透過工具,模型就能取得最新的資訊、能執行它自己腦袋裡算不出來的精確計算、能去存取使用者專屬的私人資料,甚至能在真實世界裡觸發一個行動。所以我們說,工具使用就是那座技術橋樑,一端是模型的推理與理解能力,另一端是外部世界的各種實際功能,它把這兩端接了起來。

講到這裡,我要幫你把概念再擴大一點。我們一直在講「函式呼叫」,但其實有一個更寬廣的講法,叫做「工具呼叫」。差別在哪?所謂的「工具」,不一定只是一個傳統的程式函式。它可以是一個複雜的網路 API 端點,可以是一次資料庫的查詢,甚至可以是「下達一個指令給另一個專門的代理」。你體會一下這個視角的轉變:當工具的定義被放寬到這種程度,代理的角色就不只是一個會查資料的小幫手,它更像是一個「總指揮」,英文叫 orchestrator,也就是編排者。它能橫跨各式各樣的數位資源,甚至調度其他有智慧的代理,讓大家一起協作完成任務。用這個角度去想,你才能真正抓住代理的全部潛力。

那這些能力,實務上要怎麼蓋出來?好消息是,我們不需要從零造輪子。目前有好幾個成熟的框架幫我們處理掉繁瑣的細節,像是 LangChain、LangGraph,還有 Google 的代理開發套件,英文全名是 Agent Developer Kit,簡稱 ADK。這些框架提供了穩健的支援,讓你可以很方便地定義工具、把工具整合進代理的工作流程裡。它們底層通常會運用像 Gemini 或 OpenAI 這類現代語言模型「原生內建」的函式呼叫能力。可以這麼說,工具使用是建構「強大、能互動、又對外界有感知能力」的代理的基石。

接下來,我們來看實務上的使用場景。一個簡單的判斷原則是:只要你的代理需要做的事情,超越了「生成一段文字」,而是要去「執行某個行動」或「檢索某種會變動的資訊」,工具使用就派得上用場。我幫你整理出六種典型場景。

第一種,從外部來源檢索資訊,也就是去拿訓練資料以外的即時資訊。經典例子就是天氣查詢:使用者問「倫敦的天氣如何」,模型用「倫敦」這個參數去呼叫天氣工具,拿到資料後再整理成一句友善的回答。

第二種,跟資料庫和 API 互動,對結構化的資料做查詢或更新。例如一個電子商務的代理,要回答「某某商品還有沒有庫存」,它就去呼叫庫存系統的 API 查一下。

第三種,執行計算與資料分析。語言模型本身其實不擅長精確的數學運算,所以我們可以讓它去用計算機、資料分析的函式庫或統計工具。例如一個金融代理,先呼叫股價 API 拿到目前股價,再呼叫計算機算出潛在的獲利。

第四種,發送通訊。也就是讓代理幫你寄電子郵件、傳訊息,或對外部服務發出請求。例如一個個人助理代理,從你的指令裡擷取出收件人、主旨、內文,然後呼叫電子郵件工具把信寄出去。

第五種,執行程式碼。在一個安全、隔離的環境裡,執行一小段程式。例如一個寫程式的助理代理,用所謂的程式碼直譯器,英文叫 code interpreter,去實際跑一段 Python 程式碼並分析結果。

第六種,控制其他系統或裝置。例如跟智慧家庭、跟物聯網平台互動。物聯網的英文是 IoT。舉個例子,智慧家庭代理依照你的指令,呼叫工具把客廳的燈關掉。

把這六種場景合起來看,你就能體會工具使用真正的意義:它把語言模型,從一個「只會生成文字的產生器」,轉變成一個能在數位世界、甚至實體世界裡,去感知、去推理、然後去行動的代理。

好,概念說清楚了,我們進入實作的部分。我會帶你看四段範例,但我不會逐字唸程式碼,而是用聽得懂的方式,告訴你每段程式在做什麼、輸入是什麼、輸出是什麼、為什麼這樣設計。

第一段範例,用的是 LangChain 框架,搭配 Google 的 Gemini 模型。在 LangChain 裡實作工具使用,基本上分成兩個階段:第一階段是定義工具,通常就是把一個你已經寫好的 Python 函式包裝起來;第二階段是把這個工具「綁定」到語言模型上,綁定的英文是 bind,綁定之後,模型在判斷需要用到外部功能時,就能自動生成那份結構化的工具使用請求。

這段範例先定義了一個工具,叫做「資訊搜尋」。但要注意,它其實是一個「模擬」的搜尋工具,並不是真的去上網查。它的內部其實只是一張對照表:如果你問的是「倫敦的天氣」,它就回你倫敦多雲、攝氏十五度;如果你問「法國的首都」,它就回你巴黎;如果你問「地球的人口」,它回你大約八十億人;如果你問的東西不在這張表裡,它就給一個預設的回答,說「沒有找到特定資訊,但這個主題看起來挺有趣的」。用模擬的好處是,你在學習階段不需要真的去申請外部服務,就能先把整個流程跑通。

定義好工具之後,程式建立了一個語言模型,而且特別把它的隨機性參數設成零,這個參數英文叫 temperature,設成零的意思是讓模型的回答盡量穩定、可重現,不要每次都不一樣,這在做工具呼叫時很重要,因為我們希望它的決策是確定的。

接著,程式用一個提示範本來引導對話。這個範本裡面有三個部分:一個是系統指示,告訴模型「你是一個樂於助人的助理」;一個是放使用者輸入的位置;還有一個特別的位置,叫做 agent scratchpad,直譯叫「代理的草稿紙」。這個草稿紙是用來放代理在思考過程中的內部步驟的,你可以把它想成模型一邊解題、一邊寫在旁邊的計算紙,記錄它呼叫了什麼工具、得到什麼結果。

再來,程式把語言模型、工具、提示範本這三樣東西組合在一起,建立出一個「工具呼叫代理」。然後它又建立了一個叫做 AgentExecutor 的東西,你可以把它理解成「代理的執行引擎」,或者說是代理實際跑起來的執行階段。它負責真正去呼叫代理、並且執行代理所選擇的工具。

最後,程式準備了三個問題,讓它們「平行」一起跑:第一個問「法國的首都是哪裡」,第二個問「倫敦的天氣如何」,第三個問「跟我說一些關於狗的事」。前兩個問題,工具的對照表裡剛好有答案,所以會觸發特定的回應;第三個關於狗的問題,表裡沒有,所以會觸發那個預設的回答。這樣的設計很巧妙,它一次就同時測試了「命中」和「沒命中」兩種情況,讓你看到代理在兩種狀況下都能正常運作。

第二段範例,用的是 CrewAI 框架,情境是一個金融分析的任務,代理被配備了一個查股價的工具。

這個範例的工具叫做「股價查詢工具」。它一樣是模擬的,內部有一張小表,記著幾支股票的價格,例如蘋果、谷歌、微軟。這裡有一個很值得學的設計細節:當你查的股票代碼存在時,這個工具會直接回傳一個「乾淨的數字」,也就是浮點數的股價;但當你查的代碼不存在時,它不是回傳一句「找不到」的文字,而是「拋出一個標準的錯誤」。為什麼這樣設計比較好?因為回傳乾淨的原始資料,工具的重用性更高;而主動拋出明確的錯誤,等於是逼著代理去妥善處理「失敗」這種情況,而不是把錯誤當成一段普通文字含混過去。這是一個工程上的好習慣。

定義好工具之後,程式建立了一個代理,給它設定了一個角色:資深金融分析師。CrewAI 的特色就是它很強調「角色扮演」。你會幫代理設定它的角色頭銜、它的目標,還有它的背景故事。這裡的背景故事大概是說,你是一位經驗豐富的金融分析師,擅長運用各種資料來源找股票資訊,你會給出清楚、直接的答案。設定這些,是為了引導模型用對應的口吻和專業度去回答。然後,把剛剛那個股價工具配備給這個代理。

接著,程式定義了一個「任務」。任務的描述寫得很具體,它要求代理去查蘋果公司、股票代碼 AAPL 目前的模擬股價,而且明確要求要使用那個股價查詢工具。更重要的是,任務還特別交代了:如果找不到這個代碼,你必須回報你無法取得股價。這就是在引導代理同時處理成功和失敗兩種情況。任務裡還給了期望的輸出範例,例如「AAPL 的模擬股價為一百七十八點一五美元」,讓代理知道答案應該長什麼樣子。

在 CrewAI 裡,代理和任務要靠一個叫做 Crew 的東西組起來,Crew 直譯就是「團隊」,它負責編排這些代理跟任務怎麼協同運作。最後,程式透過一個叫做 kickoff 的方法來啟動整個執行,kickoff 直譯就是「開球、啟動」的意思。另外,程式在開始跑之前,還會先檢查 API 金鑰有沒有設好,沒設好就提前報錯,避免跑到一半才出問題。

第三組範例,來自 Google 的 ADK,也就是代理開發套件。ADK 有一個很大的特色:它內建了一整套「原生整合」的工具函式庫,你可以直接拿來用,不用自己從頭包裝。我會帶你看其中四個。

第一個是 Google 搜尋工具。它的角色,就是 Google 搜尋引擎的一個直接介面,讓代理能去執行真正的網路搜尋、檢索外部資訊。這段範例建立了一個叫做「基本搜尋代理」的代理,給它一個描述說「我是使用 Google 搜尋來回答問題的代理」,再給它一段指示說「我可以透過搜尋網際網路來回答你的問題,儘管問我任何事」。然後把 Google 搜尋這個預建工具配備給它。程式裡還用到了一個叫做工作階段服務的東西,這裡用的是「記憶體內」的版本,意思是這個對話歷史只暫存在記憶體裡,關於工作階段更完整的說明會在第八章談到。最後,程式用一個輔助函式,把使用者的問題包裝成代理看得懂的格式送進去,再走訪代理回傳的一連串「事件」,從裡面找出「最終回應」並印出來。範例實際問的問題是「最新的 AI 新聞有哪些」,這正好需要即時的網路資訊,很適合展示搜尋工具。

第二個是程式碼執行工具。ADK 內建了一個工具,它為代理提供一個「沙箱化」的 Python 直譯器。沙箱這個詞,意思是一個被隔離、很安全的執行環境,程式在裡面跑,不會影響到外面的系統。有了這個,模型就能自己寫 Python 程式碼、然後執行它,用來做運算、操作資料、跑腳本。為什麼這很重要?因為語言模型的本質是「機率性」的文字生成,它是用猜下一個字的方式在運作的,這對需要「確定性邏輯」和「精確計算」的問題其實不太可靠。把計算這件事交給真正會跑程式碼的直譯器,結果才會精確。這段範例建立了一個計算機代理,給它的指示大意是:當你收到一個數學運算式,就寫並執行 Python 程式碼來算出結果,而且只回傳最終的數值,不要包成程式碼區塊那種格式。程式在跑的時候,用非同步的方式一個一個處理事件,而且會特地印出兩樣東西來方便除錯:一樣是「模型生成的程式碼」,另一樣是「程式碼執行的結果」,讓你清楚看到模型到底寫了什麼、算出了什麼。範例測試了兩個問題:一個是算「五加七再乘以三」,另一個是算「十的階乘」。這兩題都是純粹的計算,正好凸顯程式碼執行的價值。

第三個是企業搜尋工具,它背後接的是 Google 的 Vertex AI Search。這跟前面的 Google 搜尋不一樣:Google 搜尋是去搜公開的網際網路,而企業搜尋是去搜「你自己指定的私有資料儲存庫」。這段範例建立了一個代理,專門用來回答關於「第二季策略文件」的問題,它連接到一個資料儲存庫,這個儲存庫的識別碼是從環境變數讀進來的。當你問它問題,例如「摘要說明第二季策略文件的主要重點」,或是「文件中針對實驗室 X 提到了哪些安全程序」,它會去那個私有的資料儲存庫裡檢索,然後把答案串流回來。這裡有個值得注意的細節:它回答時還會附上「來源歸屬」,英文叫 source attribution,意思是它會告訴你這個答案是根據幾份來源文件得出的,這對企業應用的可信度很重要。如果資料儲存庫的識別碼填錯,或是權限沒設好,程式也會給出有用的錯誤訊息。整體來說,這段程式示範的是:怎麼打造一個對話式的 AI 應用,讓它根據你私有資料庫裡的內容來回答問題,重點就在於「從特定的資料儲存庫檢索並綜整資訊」。

第四個我要特別講一個觀念,叫做 Vertex 擴充功能,英文是 Vertex Extensions。它是一種「結構化的 API 包裝器」,讓模型能連接到外部的 API,去做即時的資料處理跟行動執行,而且它提供企業級的安全性、資料隱私和效能保證。它能拿來生成並執行程式碼、查詢網站、分析私有資料庫等等,Google 也為像程式碼直譯器、Vertex AI Search 這些常見場景提供了預先建好的擴充功能。這裡有一個跟「函式呼叫」最關鍵的差異,你一定要記住:差別在「誰來執行」。使用 Vertex 擴充功能時,是由 Vertex AI 平台「自動」幫你執行;而前面講的那種函式呼叫,執行這一步是要由使用者或客戶端「手動」去做的。換句話說,還記得我們前面講的六步驟流程嗎?其中第四步「工具執行」,在一般函式呼叫裡,是你自己的編排層去執行;但在 Vertex 擴充功能裡,這一步被平台接手自動完成了。這就是兩者最大的分野。

四段範例看完了,我們把整章的重點收斂一下。我用三個角度來幫你總結。

第一個角度,這到底「是什麼」。語言模型是很強的文字生成器,但它有一個天生的缺陷:它跟外部世界是脫節的。它的知識是靜態的、被限制在訓練資料裡,而且它本身沒有執行行動、也沒有檢索即時資訊的能力。如果不給它一座通往外部 API、資料庫或服務的橋,那它在解決真實世界問題上的能力,就會被嚴重限制住。

第二個角度,「為什麼」要用這個模式。工具使用模式,通常用函式呼叫來實作,它提供了一套標準化的解法。流程是這樣:先用模型能理解的方式,把可用的外部工具描述清楚;然後代理就能根據請求,自己判斷需不需要用工具,並生成一個結構化的物件,例如 JSON,指明要呼叫哪個函式、帶什麼參數;接著編排層去執行這個呼叫、把結果拿回來、再回饋給模型;模型最後就能把這些外部資訊或行動結果,整合進它最終的回答裡。一句話說,這個模式有效地「賦予了模型行動的能力」。

第三個角度,什麼時候該用,也就是經驗法則。原則很簡單:每當你的代理需要跳脫模型內部那點有限的知識、需要跟外部世界互動時,就用這個模式。具體來說,凡是任務需要即時資料,像天氣、股價;或需要私有、專屬的資訊,像公司內部資料庫;或需要精確計算;或需要執行程式碼;或需要觸發某個行動,像發郵件、控制智慧裝置,這個模式都是不可或缺的。

最後,我把這一章最該帶走的幾個要點,再幫你串一次。第一,工具使用,也就是函式呼叫,讓代理能跟外部系統互動、取用會變動的動態資訊。第二,它的核心動作是「定義工具」,而且要附上模型能理解的清楚描述跟參數說明。第三,要記住一個分工:是由模型來「決定」何時用工具、並生成那份結構化的函式呼叫,但「實際去執行」並把結果回傳給模型的,是代理框架。第四,對於要打造「能採取真實世界行動、能提供最新資訊」的代理來說,這個模式是不可或缺的。第五,在工具方面,LangChain 用一個叫 @tool 的裝飾器,讓定義工具變得很簡單,還提供了建立工具呼叫代理的函式跟那個執行引擎。第六,Google 的 ADK 內建了很多預先做好的工具,像我們剛剛看過的 Google 搜尋、程式碼執行,還有 Vertex AI Search 企業搜尋。

總結來說,工具使用是一項關鍵的架構原則,它把語言模型的能力,從它與生俱來的「文字生成」往外大大擴展。透過賦予模型對接外部軟體和資料的能力,代理就能執行行動、進行運算、從別的系統檢索資訊——而模型要做的,就是在它判斷有必要的時候,生成一個結構化的請求去呼叫工具。像 LangChain、Google ADK、CrewAI 這些框架,都提供了整合外部工具的結構化抽象和元件,它們幫我們處理掉兩件最麻煩的事:一是「向模型揭露工具的規格」,二是「解析模型發出的工具使用請求」。正是因為有了這些工具,我們才能比較輕鬆地,開發出能跟外部數位環境互動、能真正採取行動的精密代理系統。這,就是工具使用模式的全部精髓。
