這一章我們要談的,是怎麼讓不同的 AI 代理彼此對話、互相協作。這個主題的英文叫 Inter-Agent Communication,也就是「代理間通訊」,而它最具代表性的開放標準,叫做 A2A。A2A 是 Agent to Agent 的縮寫,直接翻成中文就是「代理對代理」。在開始之前,我先幫你把這趟路線圖說一遍。我們會先弄清楚為什麼需要這種通訊,接著拆解 A2A 的幾根核心支柱,包括參與的角色、代理卡、代理探索、任務與訊息、互動方式還有安全機制;然後我會帶你看 A2A 跟另一個常聽到的協定 MCP 有什麼不同;再來談幾個實務上的應用場景;最後我們會一起走過一段用 Google 的開發工具寫出來的真實程式,看看一個代理是怎麼被包裝成一個對外提供服務的伺服器。整章聽完,你應該就能在腦中清楚想像:兩個來自不同陣營的 AI 代理,是怎麼透過網路握手、交辦工作、再把成果回傳的。

我們先從問題本身說起。一個單獨的 AI 代理,在面對複雜、又牽涉很多面向的問題時,常常會力不從心。原因很單純:它的能力有邊界。如果你能讓多個各有專長的代理一起合作,讓它們彼此協調、把任務委派出去、互相交換資訊,那能處理的問題層級就完全不一樣了。但這裡有個關鍵的痛點,那就是這些代理往往是用不同的技術、不同的框架做出來的。有的用 LangGraph,有的用 CrewAI,有的用 Google 的 ADK。如果每兩個代理要合作,都得寫一套專屬的對接程式,那整合的成本會高到讓人卻步,而且非常耗時。這就是為什麼我們需要一個共通的協定,讓大家講同一種語言。Google 推出的 A2A 協定,就是為了解決這件事而生的開放標準。

那 A2A 到底是什麼?它是一個開源的標準,目的就是讓用不同技術開發出來的代理,能夠不受框架差異的影響,順暢地互通與協作。你可以把它想成代理世界裡的一種通用插座規格,不管你的電器是哪個牌子做的,只要符合這個規格,就能插上同一個插座一起運作。值得一提的是,這個標準背後有相當廣泛的業界支持,像是 Atlassian、Box、LangChain、MongoDB、Salesforce、SAP、ServiceNow 這些公司都站台支持。微軟更計畫把它整合進自家的 Azure AI Foundry 跟 Copilot Studio 這兩個平台,Auth0 跟 SAP 也都在跟進。我提這些名字不是要你背起來,而是想讓你感受到,這不是某個小團隊的實驗品,而是一個正在被產業認真採用的標準。

接下來,我們進入這章的核心,也就是 A2A 建立在哪幾根支柱上。整體來說有六根:核心參與者、代理卡、代理探索、通訊與任務、互動機制,還有安全性。我會一根一根帶你看清楚。

第一根支柱,是核心參與者,英文叫 Core Actors。在 A2A 的世界裡,主要有三個角色。第一個是使用者,也就是 User,他是發起需求的人,是那個說「我需要幫忙」的源頭。第二個角色是 A2A 用戶端,英文叫 Client Agent。它是代表使用者去提出請求的那一方,可能是一個應用程式,也可能本身就是一個代理,它的工作是替使用者去向外要動作或要資訊。第三個角色是 A2A 伺服器,英文叫 Remote Agent,也就是遠端代理。它用一個 HTTP 的端點來接收請求、處理、再把結果回傳。這裡有個很重要的觀念你要記住:對用戶端來說,這個伺服器是「不透明」的,英文叫 opaque。意思是用戶端只需要知道「我送什麼進去、會拿到什麼出來」,完全不需要知道對方內部是怎麼運作的、用了什麼模型、跑了什麼邏輯。打個比方,這就像你打電話去客服中心,你只在乎你的問題有沒有被解決,你不需要知道接電話那個人坐在哪棟大樓、用哪套系統。這種「黑盒子」式的設計,正是讓不同框架能互通的關鍵,因為大家彼此都不必窺探對方的內部。

第二根支柱,是代理卡,英文叫 Agent Card。這是整個 A2A 裡非常核心的東西,你可以把它理解成一個代理的「數位身分證」或「數位名片」。它通常是一個 JSON 格式的檔案,JSON 你可以想成一種有結構的純文字資料格式。這張卡片裡面,涵蓋了別人要跟它互動、還有要自動找到它所需要的所有資訊。包括它的身分、它的端點網址,也就是要去哪裡才能找到它;它的版本;它的能力,比方說它支不支援串流、支不支援推播通知;它有哪些技能;預設接受什麼樣的輸入跟輸出格式;以及它需要什麼樣的驗證。

書裡舉了一個叫 WeatherBot,也就是「天氣機器人」的例子,我幫你用聽的描述一遍這張卡片長什麼樣。卡片開頭先寫了它的名字叫 WeatherBot,描述是「提供準確的天氣預報跟歷史資料」,接著給出它的服務網址跟版本號一點零。然後是能力的部分,它宣告自己支援串流,但不支援推播通知,而且會保留狀態的轉換歷史。驗證方式上,它要求用 API 金鑰。輸入跟輸出預設都是純文字。最精彩的是技能這一塊,它列了兩項技能。第一項技能的代號叫 get current weather,也就是取得目前天氣,描述是能檢索任何地點的即時天氣,它還貼心地附了兩個範例問句,像是「巴黎現在的天氣如何」「東京目前的天氣狀況」,並且標上了 weather、current、real-time 這幾個標籤。第二項技能代號叫 get forecast,也就是取得預報,可以給你未來五天的天氣預測,範例問句像是「紐約未來五天的預報」「這個週末倫敦會不會下雨」。你看,這張卡片就像一份非常完整的自我介紹,別的代理光是讀這張卡,就能知道「喔,這傢伙會看天氣,我問它要怎麼問、它會回我什麼、我得用什麼鑰匙才能跟它說話」,全都一清二楚。

第三根支柱,是代理探索,英文叫 Agent Discovery。前面我們說代理卡是名片,那問題來了:用戶端要去哪裡拿到這張名片?這就是探索要解決的事。A2A 提供三種策略。第一種叫「眾所周知的 URI」,英文是 Well-Known URI。做法是把代理卡放在一個約定俗成的標準路徑上,書裡舉的例子是放在斜線點 well-known 斜線 agent 點 json 這樣的固定位置。這種方式適合公開的、需要自動化存取的場景,因為大家都知道去哪個固定地址翻名片。第二種叫「策展註冊中心」,英文是 Curated Registries。它是一個集中式的目錄,讓代理在上面發布自己、也讓別人在上面查詢。這比較適合企業,因為企業常常需要集中管理,還要做存取控制,決定誰能查、誰不能查。第三種叫「直接設定」,英文是 Direct Configuration。就是把代理卡直接嵌進去,或是私下分享給特定對象。這適合那種彼此緊密綁在一起、或是根本不需要動態去找對方的私有系統。不管你用哪一種方式,書裡都特別提醒:放代理卡的那個端點都要好好保護,可以用存取控制、用相互 TLS,也就是 mTLS,或是用網路層級的限制來把關,尤其是當卡片裡含有敏感資訊的時候。mTLS 這個詞我們後面講安全性會再展開,你先記得它是一種雙方都要驗證身分的加密連線就好。

第四根支柱,是通訊與任務。這是 A2A 實際運作的骨架,我們稍微花點時間講清楚。A2A 的通訊,是圍繞著「非同步任務」來設計的。什麼叫非同步?就是你交辦一件事之後,不必呆站在原地等它做完,可以先去忙別的。每一個任務,都是一段長流程裡的基本工作單位,它有一個唯一的識別碼,讓大家不會搞混是哪一件事。而且這個任務會經歷不同的狀態,書裡點名了幾個:submitted,也就是已提交;working,也就是處理中;completed,也就是已完成。因為是非同步的,所以系統可以同時平行處理很多個任務,不會卡死在一件事情上。

那代理之間具體是怎麼說話的呢?靠的是訊息,英文叫 Message。一則訊息裡包含兩種東西。一種是中介資料,也就是一些描述性的屬性,比方說這則訊息的優先順序、它是什麼時候建立的。另一種,是真正承載內容的部分,英文叫 parts,一則訊息可以有一個或多個 parts。這些 parts 很有彈性,它可以是純文字,可以是一個檔案,也可以是一段有結構的 JSON 資料。那任務做完之後產出的具體成果,有一個專門的名字叫「產物」,英文是 artifacts。產物一樣是由 parts 組成的,而且它可以一塊一塊地、增量地串流回來,不必等到全部做完才一次給你。所有這些通訊,底層都走 HTTP 或加密的 HTTPS,並且用一種叫 JSON-RPC 2.0 的規格來傳遞酬載。JSON-RPC 你可以理解成一種「用 JSON 格式來遠端呼叫對方功能」的約定。最後還有一個很實用的設計,叫 contextId,也就是情境識別碼,它是由伺服器產生的,作用是把相關的好幾個任務分成同一組,這樣即使跨越了好幾次互動,情境也能連貫下去,代理才不會忘記前面聊過什麼。

第五根支柱,是互動機制,英文叫 Interaction Mechanisms。A2A 提供四種互動模式,各自適合不同的情境,我幫你用生活化的方式區分。第一種是同步請求加回應。這適合很快就能處理完的操作,用戶端把請求送出去,然後就等在那裡,在這一來一回的單次交換裡,就拿到完整的答案。這就像你問朋友現在幾點,他馬上回你。第二種是非同步輪詢。輪詢的英文是 polling。這適合比較花時間的任務。伺服器收到請求後,會馬上回你一句「我在處理喔」,也就是 working 狀態,還會給你一個任務 ID。然後用戶端就每隔一段時間去問一次「好了沒、好了沒」,一直問到狀態變成 completed 完成,或是 failed 失敗為止。這就像你叫了外送,然後每隔幾分鐘自己打開 App 看一下送到哪了。第三種是串流更新,它用的技術叫 SSE,全名是 Server-Sent Events,也就是「伺服器主動推送的事件」。這適合需要即時、一點一點看到結果的情況。它會建立一條從伺服器到用戶端的持久連線,而且是單向的,伺服器就持續不斷地把更新推過來。這就像你看運動賽事的文字直播,畫面會自己一直跳出新的進度,你不用一直手動刷新。第四種是推播通知,用的是 Webhooks 這種機制。這適合那種極度耗時的任務。做法是用戶端先註冊一個 webhook 網址,等到任務狀態有了顯著的變化,伺服器才主動通知你。這就像你網購之後,包裹有重大進度時系統才發簡訊給你,平常它不吵你。這裡補充一點,一個代理到底支不支援串流或推播,會寫在它的代理卡裡,所以你查卡片就知道能不能用這些模式。另外,A2A 還有一個特性叫「與模態無關」,英文是 modality-agnostic。意思是它除了文字之外,還支援音訊、視訊等不同形式的內容,所以可以拿來做多模態的應用。

關於這四種互動,書裡用兩段請求範例來說明,我用聽的方式幫你描述它們的長相跟用意。第一段是同步請求,它呼叫的方法叫 sendTask,顧名思義就是「送出任務」,它期待的是一個單一、完整的回答。這段請求的內容大致是:遵循 JSON-RPC 2.0 的格式,帶了一個請求編號,指明要呼叫 sendTask 這個方法,然後在參數裡放了任務的 ID 叫 task-001、所屬的工作階段 ID,還有一則來自使用者的訊息,訊息的內容是一段純文字,問的是「美元兌歐元的匯率是多少」。它還註明了可接受的輸出格式是純文字,以及要保留多少筆歷史紀錄。整段的設計意圖很清楚:我問一個明確的問題,我期待你一次就給我完整答案。第二段是串流請求,它呼叫的方法換成了 sendTaskSubscribe,字面上是「送出任務並訂閱」,它的作用是建立一條持久的連線,然後回傳好幾筆增量的更新。它的結構跟前一段幾乎一樣,一樣帶了任務 ID、工作階段 ID 跟使用者訊息,這次問的是「今天日圓兌英鎊的匯率是多少」。兩者的差別不在問題本身,而在你選了哪個方法:選 sendTask 就是一問一答,選 sendTaskSubscribe 就是訂閱式的持續更新。我希望你抓到的重點是,同一個協定,光是換一個方法名稱,就切換了完全不同的互動節奏。

第六根支柱,是安全性。在 A2A 裡,安全不是事後補的,而是內建的根本面向,它有好幾種機制來確保資料交換既穩健又完整。第一個是相互傳輸層安全性,也就是我們前面提過的 mTLS,它的全名是 mutual TLS。這裡的關鍵字是「相互」,一般我們上網的加密只驗證伺服器這一方,但 mTLS 是雙方都要互相驗證身分,連線既加密又經過認證,這樣能防止未經授權的存取,也能防止連線被攔截。第二個是全面的稽核日誌。也就是把通訊的過程都詳細記錄下來:資訊流到哪裡去、牽涉到哪些代理、執行了哪些動作。這份紀錄的用處是讓事情有跡可循,出問題時可以追責、可以除錯,也可以拿來做安全分析。第三個是代理卡宣告。意思是把驗證的需求,明明白白地寫在代理卡裡,這樣驗證的管理就集中而且簡化了,大家一看卡片就知道規矩。第四個是憑證處理。代理通常會用 OAuth 2.0 的權杖,也就是 token,或是用 API 金鑰,透過 HTTP 的標頭來做驗證。這裡有個重要的安全細節:憑證要放在標頭裡,絕對不要把它暴露在網址裡、或塞在訊息的主體內,因為那些地方很容易被記錄下來或外洩。

講完六根支柱,我們來釐清一個很多人會搞混的問題:A2A 跟 MCP 是什麼關係?MCP 的全名是 Model Context Protocol,中文叫「模型情境協定」,是 Anthropic 提出的。這兩者其實是互補的,不是競爭關係,你要記住它們各管一塊。MCP 著重的是「為代理建構情境」,以及「代理怎麼去跟外部的資料和工具互動」。換句話說,MCP 處理的是一個代理向下、向外去接資源跟工具的那條線。而 A2A 著重的是「代理跟代理之間」的協調與通訊,讓任務委派跟協作變得可能,它處理的是代理之間橫向握手的那條線。打個比方,如果把代理想成公司裡的員工,MCP 像是給員工配備工具跟查資料庫的權限,讓他能完成自己的工作;A2A 則像是員工之間互相發信、開會、把工作交辦給同事的那套溝通規矩。A2A 的整體目標,就是提升效率、降低整合成本,並且推動複雜的多代理系統往互通與創新的方向走。

接著我們看幾個實務上的應用場景,感受一下代理間通訊到底能帶來什麼價值。它的好處主要是模組化、可擴展性,還有更強的整體智慧。第一個場景是多框架協作,這也是 A2A 最首要的使用案例。它讓用 ADK、用 LangChain、用 CrewAI 這些不同框架做出來的代理,能夠通訊協作,各自去專精問題的不同面向。第二個場景是自動化工作流程的編排。在企業裡,你可以讓代理彼此委派跟協調任務,書裡舉的例子很具體:一個代理負責蒐集資料,蒐集完委派給第二個代理去做分析,分析完再交給第三個代理去生成報告,像一條流水線一樣把工作傳下去。第三個場景是動態資訊檢索。主要的代理可以向一個專門負責擷取的代理,去請求即時的資訊,比方說市場資料,那個擷取代理就透過外部的 API 把資料抓回來、再回傳。這三個場景的共同精神都是:讓專精的代理各司其職,再透過 A2A 把它們串起來。

好,現在進入最實際的部分,我們一起走過一段真實的程式碼。提醒你,我不會逐行去唸程式的符號,我會告訴你每一段在做什麼、輸入什麼、輸出什麼、為什麼這樣設計。Google 有一個 A2A 的範例儲存庫,裡面提供 Java、Go、Python 三種語言的範例,涵蓋 LangGraph、CrewAI、Azure AI Foundry、AG2 等好幾種框架,而且都是開放的 Apache 2.0 授權。我們聚焦的這個例子,是怎麼用一個以 Google ADK 為基礎、又搭配 Google 驗證工具的代理,去設定出一個 A2A 伺服器。ADK 的全名是 Agent Development Kit,也就是「代理開發套件」。

第一段程式,是一個叫 create agent 的非同步函式,它的工作是建構出一個 ADK 代理。它接收兩個輸入,分別是用戶端 ID 跟用戶端密鑰,這兩個是用來向 Google 驗證身分用的。它做的第一件事,是拿這兩個憑證去初始化一個叫 CalendarToolset 的工具集,這個工具集讓代理能夠存取 Google 行事曆的 API。接著,它建立一個 LlmAgent,也就是一個由大型語言模型驅動的代理,並且幫它設定好幾樣東西:用的模型是 Gemini 2.0 Flash,名字叫 calendar agent,描述是「一個能協助管理使用者行事曆的代理」。最關鍵的是給它的指令,這段指令告訴代理:你的工作是幫忙管理使用者的行事曆,使用者可能會問行事曆的狀態、也可能要你去改它;請用提供的工具去跟行事曆 API 互動;如果使用者沒特別說,就預設用他的主要行事曆;而且用行事曆 API 的時候,時間要用一種叫 RFC3339 的標準格式來表示。這段指令還有一個很聰明的小設計:它會動態地把「今天是幾月幾號」這個當前日期塞進指令裡,這樣代理才有時間概念,知道「明天」「這個週末」到底是指哪一天。最後,它把工具集裡的工具都掛到這個代理身上,然後把整個代理回傳出去。整段的設計意圖,就是把一個會看行事曆、聽得懂自然語言、又有時間感的代理組裝完成。

第二段程式比較長,它是一個叫 main 的主函式,目的是把剛剛那個代理,真正包裝成一個對外提供服務的 A2A 伺服器。我幫你把它的流程串起來。它一開始會先做檢查,確認你有沒有設定好 API 金鑰;不過如果你用的是 Vertex AI 的 API,那就不需要這把金鑰。如果兩個都沒設好,它就會直接報錯停下來,避免後面白忙一場。檢查通過之後,它做的第一件正事,是定義一個技能,英文叫 AgentSkill。這個技能的代號是 check availability,也就是「檢查有沒有空」,描述是用使用者的 Google 行事曆去檢查某個時段他有沒有空,還附了一個範例問句「我明天早上十點到十一點有空嗎」。接下來,它建立那張我們講了很久的代理卡,英文是 AgentCard。卡片裡填了代理的名字叫 Calendar Agent、描述、它對外的網路位址也就是主機加上連接埠、版本號、預設輸入輸出都是文字,並且宣告它的能力支援串流,然後把剛剛那個技能掛上去。你看,前面講的理論,在這裡就具體變成了一張卡。

代理卡定義好之後,它才真正去呼叫前面那個 create agent 函式,把用戶端 ID 跟密鑰從環境變數裡讀出來、傳進去,生出實際的 ADK 代理。接著它建立一個執行器,英文叫 Runner,並且配給它幾個記憶體內的服務,分別管理產物、工作階段跟記憶。這裡「記憶體內」的英文是 in-memory,意思是這些東西都暫存在程式的記憶體裡,程式一關掉就沒了,這很適合用在示範跟測試,但正式上線通常會換成真正的資料庫。然後它建立一個代理執行器,叫 ADKAgentExecutor,把執行器跟代理卡交給它。它還特別定義了一個處理驗證回呼的函式,叫 handle auth,這是用在 OAuth 那種「使用者授權完之後,Google 會回頭通知我們」的流程上,驗證成功就回一句 Authentication successful。

再往下,它建立一個請求處理器,叫 DefaultRequestHandler,並且配一個記憶體內的任務儲存區給它,負責管理進來的任務。然後,真正的關鍵一步來了:它建立一個叫 A2AStarletteApplication 的應用,把代理卡跟請求處理器組進去。Starlette 是一個 Python 的網頁框架,你可以理解成搭建網路服務的鷹架。這個 A2A 應用會自動產生一組標準的路由,也就是各種網址對應到各種功能的對照表。程式接著把我們剛剛那個處理驗證的 authenticate 路徑,額外加到這組路由裡。最後,它用這些路由組成一個完整的 Starlette 應用,並透過一個叫 Uvicorn 的伺服器,把整個服務真正跑起來、對外公開。

我幫你把這第二段的全貌總結一下:這段程式做的,就是設定出一個符合 A2A 規範的 Calendar Agent。它先驗證金鑰或 Vertex AI 的設定,再在代理卡裡定義出 check availability 這項技能跟網路位址,然後建立 ADK 代理、並用記憶體內的服務去管理產物、工作階段跟記憶,最後把它包成一個含有驗證回呼跟 A2A 處理器的網頁應用,透過 Uvicorn 對外公開。這整個過程,完整展示了從「定義一項能力」到「把代理當成一個網頁服務真正執行起來」的全流程。換句話說,只要善用代理卡跟 ADK,開發者就能打造出能整合像 Google 行事曆這類工具、而且彼此可以互通的代理。

到這裡,主要的內容都講完了,我們用幾個層次幫你把整章收斂起來,把最該帶走的東西再串一次。

先說「這是什麼」。問題的根源在於:單一的 AI 代理,尤其是用不同框架建出來的代理,往往很難獨力解決複雜、多面向的問題,而最核心的挑戰,就是缺乏一個共通的協定,讓它們能有效地合作。沒有標準化的做法,要把這些迥異的代理整合在一起,既昂貴又耗時,也阻礙了那些專精代理把各自的技能組合起來去解決更大的任務。

再說「為什麼用它」。A2A 是一個以 HTTP 為基礎的開放標準,它讓不同的 AI 代理能無縫地協調、委派任務、共享資訊,而且不受底層技術影響。它最核心的元件就是代理卡,那是一份描述代理能力、技能跟端點的數位身分檔案,讓代理之間能彼此探索、彼此互動。A2A 還提供了同步跟非同步等多種互動機制,從而培育出一個模組化、可擴展的多代理生態系。

接著是一條好用的「經驗法則」。什麼時候該用 A2A?當你需要編排兩個以上的 AI 代理一起合作的時候,尤其是當它們是用不同框架建出來的,像 Google ADK、LangGraph、CrewAI 各做一塊的時候,它就特別合適。它很適合用來建構複雜、模組化的應用,讓專精的代理各自負責工作流程的一部分,比方說分析交給一個代理、報告生成交給另一個代理;它也適合那種代理需要動態去探索、然後運用彼此能力的場景。

最後,我把幾條最關鍵的重點幫你濃縮成幾句話。第一,A2A 是一個開放的、以 HTTP 為基礎的標準,它促成了不同框架代理之間的通訊與協作。第二,代理卡就像一張數位識別碼,讓其他代理能自動探索並理解它的能力。第三,A2A 同時提供同步的請求加回應,還有串流式的更新,而且支援多輪對話,比方說它有一個叫 input-required 的狀態,意思是「我還需要你補充資訊」,靠這種狀態就能維持對話的情境。第四,它鼓勵模組化的架構,讓各個專精代理可以在不同的連接埠上獨立運作,從而實現可擴展跟分散式。第五,有一些工具,像書裡提到的 Trickle AI,可以幫你把 A2A 的通訊視覺化跟追蹤,協助你監控跟除錯多代理系統。第六,請記住 A2A 跟 MCP 的分工:A2A 是管理跨代理任務的高層級協定,而 MCP 是為大型語言模型提供對接外部資源的標準化介面,一個管代理之間,一個管代理對工具。

總結來說,A2A 確立了一項開放標準,克服了單一 AI 代理與生俱來的那種孤立性。它用一個共通的、以 HTTP 為基礎的框架,讓不同平台上的代理能夠無縫協作。它的核心元件代理卡,扮演的是數位身分的角色,清楚定義能力、又能被動態探索;而這個協定的彈性,讓它能支援同步請求、非同步輪詢、即時串流等多種模式,使得模組化、可擴展的架構成為可能,也讓專精的代理能組合起來,編排出複雜的自動化工作流程。安全性是它的根本,內建了 mTLS 跟明確的驗證需求。再加上它跟 MCP 互補、各大科技公司的支持,還有現成範例的可得性,在在凸顯出它越來越重要,正在為更精密、更分散、也更智慧的多代理系統鋪平道路。這,就是代理間通訊與 A2A 協定要帶給你的整張藍圖。
