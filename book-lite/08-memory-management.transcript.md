這一講,我們要談的是一個讓 AI 代理真正「聰明起來」的關鍵能力,叫做記憶管理。英文是 Memory Management。我先把這一講的路線圖講給你聽,讓你心裡有個底。我們會先談為什麼代理需要記憶、記憶分成哪兩種;接著看記憶在實務上能拿來做什麼;然後我會帶你走過三套主流框架怎麼實作記憶,分別是 Google 的代理開發套件、LangChain 搭配 LangGraph,以及 Google 雲端上的記憶銀行;最後再把整章的重點串起來收尾。整段都是用聽的就能懂,不需要你看任何螢幕或程式碼。

我們先從一個最根本的問題說起:為什麼代理需要記憶?你可以想像一個沒有記憶的助理,每講完一句話就把前面全部忘光。你跟它說「我下星期要去紐約」,下一句它就完全不知道你剛剛講過什麼。這種代理我們叫它「無狀態的」,英文是 stateless。無狀態的代理只能做一次性的問答,沒辦法維持對話的連貫,沒辦法學習,也沒辦法為你做個人化。記憶,就是讓代理能夠保留並運用過去的資訊,進而做出更明智的決策、維持住情境脈絡,而且能隨著時間持續改善自己。

那代理的記憶到底長什麼樣子?它分成兩大類,這個區分非常重要,後面整章都圍著它打轉,你一定要先聽清楚。

第一類叫做短期記憶,也叫情境記憶。你可以把它想成人的「工作記憶」,就是你正在腦子裡同時轉的那些東西。對大型語言模型來說,短期記憶主要就活在它的「情境視窗」裡。情境視窗,英文是 context window,你可以理解成模型一次能讀進眼裡的那一段文字空間,裡面裝的是最近的訊息、最近的回覆、剛剛工具回傳的結果,還有代理自己的一些反思。這個空間的容量是有限的,所以管理短期記憶的重點,就是想辦法把「最相關」的資訊留在裡面,做法通常是把比較舊的片段做摘要,把關鍵的部分特別強調出來。這裡要破除一個迷思:有些模型號稱「長情境」,聽起來好像記得很多,但它其實只是把那個空間「變大」而已,本質上情境還是短暫的。為什麼說短暫?因為一旦這次的對話工作階段結束,這些內容就全部消失了;而且每一次要重新處理這一大段情境,成本都很高。正因為這樣,我們才需要另外一種記憶來達成「持久」。

第二類就是長期記憶,也叫持久記憶。它是一個跨越多次互動、多個任務都能保留資訊的儲存庫。它通常放在代理外部,可能是一般的資料庫、知識圖譜,或者最常見的「向量資料庫」。向量資料庫,英文是 vector database,它的做法是把資訊轉換成一串一串的數值向量,然後檢索的時候,不是靠關鍵字去比對,而是靠「語意上的相似度」去找。這種搜尋方式有個名字,叫做語意搜尋,英文是 semantic search。我舉個白話的例子:你問「我上次說我喜歡吃什麼?」就算你當初講的是「我超愛海鮮」,語意搜尋也能憑「意思相近」把那句話撈出來,不需要你的字眼一模一樣。所以長期記憶的運作流程是這樣:代理去查詢外部的儲存庫,把相關的資料取出來,再把它整合進當下的短期情境裡,即時拿來用。短期跟長期這兩種記憶搭配起來,代理才算有了完整的記性。

了解了這兩種記憶之後,我們來看看它們在真實世界裡能拿來做什麼。記憶能讓代理超越基本的問答,隨著時間展現出有智慧的行為。我列幾個典型的應用給你聽。

第一個是聊天機器人跟對話式 AI。短期記憶負責讓這一場對話講起來連貫,長期記憶則負責回想你的偏好、回想你們以前聊過的東西,做到個人化。

第二個是任務導向的代理。短期記憶用來追蹤現在做到第幾步、進度如何;長期記憶則用來存取那些不在當下對話裡、但又需要用到的使用者資料。

第三個是個人化體驗。用長期記憶把你的偏好跟行為存起來、需要時再取出來,代理就能調整它給你的回應跟建議。

第四個是學習與改善。把成功的策略、犯過的錯誤、新得到的資訊都存進長期記憶,代理就能不斷調適。這裡特別提一個例子,強化學習的代理,英文是 reinforcement learning,它就是靠這種方式把學到的策略儲存下來。

第五個是資訊檢索,這也就是大名鼎鼎的 RAG。RAG 的全名是檢索增強生成,英文是 Retrieval-Augmented Generation。它的做法是把一個知識庫當成長期記憶,需要回答的時候,先去檢索出相關的文件,再用這些文件來輔助產生回應。

第六個是自主系統,比如機器人或自駕車。它用短期記憶掌握眼前即時的周遭環境,用長期記憶儲存地圖、路線,還有過去學到的行為模式。

好,觀念講完了,接下來我們進入動手實作的部分。我會帶你看三套框架。第一套是 Google 的代理開發套件,英文叫 Agent Developer Kit,簡稱 ADK,之後我就用 ADK 來稱呼它。

ADK 用三個核心概念來有條理地管理情境跟記憶。在 ADK 的世界裡,每一次互動都可以看成一條獨特的對話執行緒。這三個概念,你一定要分清楚。

第一個是 Session,中文叫工作階段。它就是一條獨立的聊天執行緒,記錄這一次互動裡的所有訊息跟動作,這些東西在 ADK 裡叫做 Events,也就是事件;同時它也存放一些暫時性的資料,這些資料叫做 State,也就是狀態。

第二個是 State,也就是狀態。它是 Session 裡面、只跟「當前這條聊天執行緒」相關的資料。

第三個是 Memory,記憶。它是一個「可以被搜尋」的資訊儲存庫,來源可能是過去的聊天紀錄,也可能是外部資料,目的是讓代理能去檢索「超越當下這場對話」的東西。

你可以這樣記:Session 跟 State 處理的是「眼前這場對話」,屬於短期;Memory 處理的是「跨越很多場對話」的知識,屬於長期。

在 ADK 裡,有兩個服務分別管這些事。一個叫 SessionService,負責管理聊天執行緒的整個生命週期;另一個叫 MemoryService,負責監管長期知識的儲存跟檢索。這兩個服務都提供幾種不同的後端選擇:一種是「記憶體內」的版本,英文是 in-memory,它跑得快、很適合測試,但缺點是程式一重新啟動,資料就沒了;另一種是接資料庫或雲端的版本,它能持久保存、而且可以擴展。

我們先深入講 Session,也就是怎麼追蹤每一段聊天。

一個 Session 物件,會把某一段對話的所有資料都包在一起。裡面有幾樣東西:第一是各種識別碼,包括這個 Session 自己的編號、應用程式的名稱、還有使用者的編號;第二是依照時間先後排好的事件紀錄;第三是前面講過的那份暫時性狀態;第四是一個時間戳記,記錄這個 Session 最後一次更新是什麼時候。

實務上,開發者通常不會直接去戳 Session,而是透過 SessionService 來間接操作它,由這個服務去管理對話的生命週期。ADK 提供了好幾種實作。

第一種叫 InMemorySessionService,就是記憶體內的版本。它的用法很單純,你只要把這個服務建立起來就能用。它適合在你自己電腦上開發跟測試,但要記得,資料不會在重新啟動後保留。

第二種叫 DatabaseSessionService,適合用在正式環境,或是需要持久儲存的開發環境。用它的時候,你要給它一個資料庫的網址,告訴它資料要存到哪。書裡的範例是指向一個本機的 SQLite 檔案。提醒一下,要用這個服務,你得先安裝對應的套件跟資料庫驅動程式,比如說如果你用 PostgreSQL,就要裝它的驅動。

第三種叫 VertexAiSessionService,它運用的是 Vertex AI 的基礎設施,適合在 Google Cloud 上做可擴展的正式部署。用這個服務的時候,有個小細節:你要設定的應用程式名稱,要對應到一個叫「推理引擎」的資源編號或名稱,英文是 Reasoning Engine。而且在呼叫各種方法,像是建立、取得、刪除 Session 的時候,都要把這個推理引擎的名稱傳進去。

所以你選哪一種 SessionService,其實就決定了你的對話歷史跟暫時資料,要用什麼方式儲存、能不能持久。

那一次完整的訊息交換,在底層是怎麼跑的?它是一個循環。首先,有一個叫 Runner 的執行器,它去取得或建立一個 Session;接著代理依照當下的情境去處理這則訊息,過程中可能會更新狀態;然後 Runner 把整個過程包裝成一個事件,透過一個叫 append_event 的動作,把這個事件記錄下來,同時更新狀態;最後,當這次互動結束,理想上會用一個刪除 Session 的動作把它收掉。你只要記住這個節奏:取得 Session、處理訊息、附加事件、更新狀態,這是 ADK 記憶運作的心跳。

講完 Session,我們來細看 State,也就是狀態。我喜歡把 State 叫做「這段對話的暫存便條本」。

每一個 Session 都帶著一份 State,它就像這段對話的暫時工作記憶。這裡要分清楚兩樣東西:事件紀錄記的是「整段歷史」,而狀態記的是「為了讓對話連貫所需要的關鍵細節」。狀態長什麼樣子呢?它是一個字典,英文是 dictionary,也就是一組一組的「鍵」跟「值」配對。裡面裝的東西比如說:使用者的偏好、任務目前的進度、一點一點蒐集起來的資料、還有一些條件旗標。

State 裡的值,必須是「可以被序列化」的 Python 基本型別,講白話就是能被安穩存起來的單純資料,像是字串、數字、布林的是非值、串列,以及這些東西組成的字典。至於這些變更能保留多久、有多持久,要看你前面選的是哪一種 SessionService。

接下來這個點很重要,請你專心聽。State 的「鍵」可以加上前綴,而這個前綴會界定它的「作用範圍」跟「持久程度」。一共有四種情況。第一種,鍵沒有任何前綴,代表它是這個工作階段專屬的,只在這條對話裡有效。第二種,鍵加上「user 冒號」這個前綴,代表它跟某一個使用者編號綁在一起,會橫跨這個使用者的所有工作階段;比如記住「這位使用者登入過幾次」就適合放這裡。第三種,鍵加上「app 冒號」前綴,代表它是在所有使用者之間共享的。第四種,鍵加上「temp 冒號」前綴,代表它只在當前這一回合有效,英文叫 turn,而且不會被持久儲存,講完就丟。你可以用一個比喻記:沒前綴是「這場對話的便條」,user 是「這個人的專屬檔案」,app 是「公布欄」,temp 是「隨手寫完即丟的草稿」。

代理存取資料的時候,是透過單一一份狀態字典來拿所有東西,背後的檢索、合併、持久化,都由 SessionService 處理掉。還有一個原則:狀態應該在「附加事件」的那個時機一起更新,這樣才能正確追蹤。

那實際上要怎麼更新狀態呢?ADK 提供了兩種做法,從簡單到標準。

第一種是簡易做法,叫做 output_key,專門用在「代理的文字回覆」這種單純情況。如果你只是想把代理最後產生的那段文字回應存進狀態,你只要在設定代理的時候,給它一個 output_key,告訴它「請把回覆存到這個鍵底下」,Runner 就會在附加事件的時候自動幫你寫進去。書裡的範例做了一個打招呼的代理,它的任務就是產生一段簡短友善的問候語,然後設定 output_key 指向一個叫 last_greeting 的鍵。整個流程是:先把服務跟 Runner 建立起來,建立一個 Session,這時候你去看狀態,它是空的;接著送一句「Hello」進去執行;等 Runner 把所有事件都處理完之後,你再重新去取一次 Session 來看狀態,就會發現那段問候語已經自動被存進 last_greeting 裡了。這裡有個操作上的小提醒:你一定要等 Runner 全部跑完「之後」,再去檢查狀態,才看得到更新後的結果。背後的原理就是,Runner 一看到你設了 output_key,就會在附加事件的時候自動帶上一個叫 state_delta 的東西,也就是「狀態的變更內容」。

第二種是標準做法,直接操作這個 state_delta,適合比較複雜的更新。什麼時候需要它?當你要一次更新好幾個鍵、要存的不是文字資料、要鎖定特定的作用範圍像是 user 或 app、或是你要做的更新跟文字回覆無關的時候。這時候你就手動建一個包含所有變更的 state_delta 字典,把它放進你要附加的那個事件的動作裡。

書裡推薦一種更乾淨的寫法,叫做「以工具為基礎」的做法。它做了一個叫 log_user_login 的工具,意思是「記錄使用者登入」。這個工具會收到 ADK 自動提供的一個東西,叫 ToolContext,工具情境,透過它就能存取狀態。這個工具裡面做了四件事:第一,把「這位使用者的登入次數」加一,這個鍵用的是 user 前綴,所以會跨工作階段累積;第二,把任務狀態設成「進行中」;第三,記錄這次登入的時間;第四,加上一個 temp 前綴的暫時旗標,表示「需要驗證」,這個講完這回合就丟。工具最後回傳一個字典,確認動作成功,還告訴你目前總共登入幾次。這種寫法的好處是:把所有相關的狀態變更「封裝」在一個工具裡面,比起在工具外面東改一個、西改一個,程式碼更乾淨、也更穩健。在真實應用裡,是由大型語言模型代理自己決定什麼時候呼叫這個工具的;書裡為了示範,才手動模擬了一次呼叫。

講到這裡,有一個強烈的警告我一定要轉達給你。千萬不要在取得 Session 之後,直接去修改那份狀態字典。為什麼?因為這樣做會繞過標準的事件機制,後果有四個:你的變更不會被記進歷史、可能根本不會被持久化、可能造成並行衝突的問題、也不會更新相關的中介資料。正確的做法,永遠是用 output_key 或者 state_delta 來更新;那份狀態字典,你主要拿它來「讀」就好。

那設計狀態的時候有什麼原則?幫你濃縮成幾條:保持簡單;盡量用基本型別;給鍵取清楚的名字,而且正確使用前綴;避免層層巢狀的深層結構;最後,務必透過「附加事件」這個正規流程來更新。

State 講完,我們進到 ADK 的第三塊,也就是用 MemoryService 來管理長期知識。

前面講的 Session 跟 State,維護的都是「單一一場對話」的歷史跟暫時資料,屬於短期記憶。而 MemoryService 管的長期知識,則是一個持久的、可以搜尋的儲存庫,它可以包含很多次過去的互動,或者外部來的資料。

它一樣有記憶體內的版本,叫 InMemoryMemoryService,適合本機開發跟測試,缺點同樣是程式一停,記憶就沒了。MemoryService 的核心功能有兩個。第一個動作是 add_session_to_memory,意思是「把工作階段加進記憶」,它會從一個工作階段裡擷取內容、儲存起來。第二個動作是 search_memory,「搜尋記憶」,用查詢的方式把相關的資料找回來。

到了正式環境,通常會改用 VertexAiRagMemoryService。它運用 Google Cloud 的 RAG 服務,達成可擴展、持久、而且具備語意搜尋的檢索。用它的時候,你要指定一個 RAG 語料庫的資源名稱,告訴它記憶存在哪;另外還有兩個可以選的調整參數,一個是「要取回前幾筆最相似的結果」,書裡設五筆,另一個是「向量相似度的門檻」,用來決定多像才算數。設定好之後,前面講的那兩個動作,加進記憶跟搜尋記憶,就都會跟你指定的這個 RAG 語料庫互動了。

ADK 講完了,我們換第二套框架,看看 LangChain 跟 LangGraph 怎麼處理記憶。

在 LangChain 跟 LangGraph 裡,記憶的基本思路是這樣:先參考已經存起來的歷史,用它來「豐富」當前的提示;然後再把這次最新的交流記錄下來,留給以後用。

它也是分短期跟長期。短期記憶是「以執行緒為範圍」的,英文是 thread-scoped,意思是它只追蹤單一這場工作階段內的對話,用來提供即時的情境。但它有個老問題:完整的歷史可能會長到超出情境視窗,進而影響效能。LangGraph 的處理方式,是把短期記憶當成代理狀態的一部分,然後透過一個叫「檢查點機制」的東西來持久化,英文是 checkpointer。有了檢查點,一條執行緒就可以隨時被回復,接著之前的進度繼續跑。

長期記憶則是跨越工作階段,儲存使用者層級或應用層級的資料,而且可以在不同執行緒之間共享。它存放在一個叫「命名空間」的地方,英文是 namespace,你可以把命名空間想成一個個分門別類的資料夾。LangGraph 提供一個叫「儲存庫」的東西,英文是 store,用來儲存跟回想這些長期記憶,讓代理能無限期地保留知識。

接下來看 LangChain 裡幾個具體的記憶工具。

第一個叫 ChatMessageHistory,它是「手動的記憶管理」。適合用在正式的鏈之外,你想直接掌控對話歷史的時候。鏈,英文是 chain,你可以先理解成「一串串接起來的處理步驟」。這個工具的用法很直覺:你建立一個歷史物件,然後可以手動把使用者的訊息加進去、把 AI 的回覆加進去,最後也能把整串訊息讀出來。書裡的例子就是使用者說「我下星期要去紐約」,AI 回「太好了,那是個很棒的城市」,兩句都被記進歷史。

第二個叫 ConversationBufferMemory,它是「用在鏈裡面的自動化記憶」。它由兩個參數來自訂。第一個參數是 memory_key,它指的是「在提示裡面,要用哪個變數名稱來放聊天歷史」,預設叫 history。第二個參數是 return_messages,它決定歷史回傳的格式:如果設成否,也就是預設值,它會回傳一段「格式化好的字串」,這適合給標準的語言模型;如果設成是,它會回傳一串「訊息物件的串列」,而這是給聊天模型用的建議格式。這個差別等一下會再提到,你先記著。

那怎麼把記憶整合進鏈裡?書裡示範把記憶放進一個叫 LLMChain 的鏈。一旦整合進去,模型就能存取對話歷史,給出跟情境相關的回應。範例做了一個旅遊專員的角色,它的提示模板裡有一個放「先前對話」的位置,還有一個放「新問題」的位置;這裡的關鍵是,記憶設定的那個 memory_key 叫 history,必須跟提示模板裡那個變數名稱對得起來,兩邊要一致。然後它連續問了三句:「我想訂機票」、「對了我叫 Sam」、最後「我剛剛說我叫什麼名字?」。重點就在最後這句——因為記憶把前面講過的名字記住了,代理才答得出「你叫 Sam」。這就是記憶在發揮作用的最直接證據。

剛剛說那個 return_messages 參數很重要,這裡就見真章。對聊天模型來說,書裡建議把它設成「是」,改用結構化的訊息物件串列。範例又做了一個友善助理,提示裡用一個叫 MessagesPlaceholder 的東西,專門當作「聊天歷史的佔位插槽」,記憶的 memory_key 就設成 chat_history,而且 return_messages 設成是。對聊天模型而言,這個設定是不可或缺的。然後它先說「嗨,我是 Jane」,再問「你記得我的名字嗎?」,助理因為有記憶就答得出來。

接著是這一段裡我覺得最值得記住的觀念之一:長期記憶可以類比成人類的記憶,分成三種。我一個一個講。

第一種是語意記憶,英文是 Semantic Memory,它的關鍵字是「記住事實」。它保留的是事實跟概念,比如你的偏好、某個領域的知識,目的是為回應提供事實上的依據。實務上,它可以用兩種方式管理:一種是一份「持續更新的使用者設定檔」,通常用 JSON 這種格式;另一種是把一條一條獨立的事實,當成一個「文件的集合」來存。

第二種是情節記憶,英文是 Episodic Memory,關鍵字是「記住經驗」。它回想的是過去發生的事件或動作,常常用來記「我之前是怎麼完成某個任務的」。實務上,它最常見的實作方式是「少樣本範例提示」,英文是 few-shot example prompting,意思是給代理看幾個過去成功的範例序列,讓它照著學。

第三種是程序記憶,英文是 Procedural Memory,關鍵字是「記住規則」。它存的是「該怎麼執行任務」的核心指令跟行為準則,通常是寫在系統提示裡面。這裡有個很有意思的技巧,叫做「反思」,英文是 Reflection:你把代理「目前的指令」連同「最近的互動」一起交給代理本身,然後要求它去精煉、改寫自己的指令。換句話說,讓代理自己檢討自己、自己升級自己的行為守則。

書裡用一段虛擬碼示範了這個反思怎麼運作,我用講的帶你想像它的流程。它有兩個節點。第一個節點叫「更新指令」:它先從儲存庫裡把目前的指令取出來,接著組一段提示,要求語言模型「對這段對話進行反思,並且生成改進過的新指令」,模型吐出新指令之後,它再把新指令存回儲存庫。第二個節點叫「呼叫模型」:它從儲存庫裡檢索出最新的那份指令,用這份指令來組提示,然後應用程式的邏輯就接著往下跑。你把這兩個節點合起來看,就是一個自我改善的循環:用指令做事、反思、更新指令、下次用更好的指令做事。

那 LangGraph 的儲存庫實際上是怎麼存記憶的?它把長期記憶以 JSON 文件的形式存在儲存庫裡,每一則記憶都組織在一個自訂的命名空間底下,命名空間就像資料夾,然後每則記憶有一個獨特的鍵,就像資料夾裡的檔名。書裡用一個記憶體內的儲存庫示範了三個基本動作,我幫你口語化講過一遍。首先它定義了一個「嵌入函式」的替身,英文是 embedding function,真正的嵌入函式是把文字轉成向量用的,這裡只是放一個假的佔位版本方便示範。接著它為某個使用者、某個應用情境定義了一個命名空間。然後做三件事:第一,把一則記憶「放進」儲存庫,這則記憶記的是這位使用者的規則,比如「喜歡簡短直接的講話方式」、「只說英文跟 Python」;第二,透過命名空間跟鍵,把這則記憶「取出來」;第三,在命名空間裡「搜尋」記憶,可以依內容過濾,並且依向量相似度來排序。這三個動作——放進、取出、搜尋——就是長期記憶儲存庫最基本的操作。

第三套框架,我們來看 Vertex Memory Bank,中文可以叫記憶銀行。

它是 Vertex AI Agent Engine 裡的一個「受管理服務」,英文是 managed service,意思是這套東西由平台幫你顧好,為代理提供持久的長期記憶。它的特色是會運用 Gemini 模型,以「非同步」的方式去分析對話歷史,從裡面擷取出關鍵的事實跟使用者偏好。非同步的意思,就是它在背景默默處理,不會卡住你當下的對話。

這些擷取出來的資訊,會被持久儲存,依照像是使用者編號這樣的範圍來組織,而且會被「聰明地」更新——它會整併新的資料,還會去化解互相矛盾的地方。比如你以前說喜歡靠窗,後來改說喜歡走道,它會幫你把矛盾理清楚。等到一個新的工作階段開始,代理就透過兩種方式來檢索相關的記憶:一種是完整地回想,另一種是用嵌入向量的相似度搜尋。靠這個,它就能跨越不同的工作階段維持連續性,並且個人化它的回應。

運作上,代理的執行器會跟一個事先初始化好的服務互動,這個服務叫 VertexAiMemoryBankService。它會自動把對話產生的記憶儲存起來,而且每一則記憶都會標上獨特的使用者編號跟應用程式名稱,確保之後檢索得準確。書裡的範例流程是:先建立這個記憶銀行服務,綁定專案、位置跟一個代理引擎的編號;接著取得目前的工作階段;然後呼叫「把工作階段加進記憶」這個動作,把這場對話的記憶存進銀行。最後補充一點:記憶銀行跟 Google ADK 是無縫整合、開箱即用的;而對 LangGraph、CrewAI 這些其他框架,它也透過直接呼叫 API 的方式提供支援。

把三套框架都走過一遍之後,我們來做整章的收斂,我會用「是什麼、為什麼、什麼時候用」這三個角度幫你抓住骨幹。

先講「是什麼」。代理必須能記住過去的互動,才能執行複雜的任務、提供連貫的體驗。沒有記憶的代理是無狀態的,它無法維持情境、無法學習、也無法個人化,只能困在一次性的互動裡。所以這一章要解決的核心問題就是:怎麼同時管好兩種東西——一種是「單一對話裡的即時暫時資訊」,另一種是「跨越時間累積下來的龐大持久知識」。

再講「為什麼」,也就是標準解法。答案是一套「雙元件」的記憶系統。短期的情境記憶,把最近的互動保存在語言模型的情境視窗裡,維持對話的流暢;長期記憶,則用外部資料庫,通常是向量儲存庫,來做高效率的語意檢索。像 Google ADK 這樣的框架,就提供了對應的元件:Session、State,以及那個專門對接長期知識庫的 MemoryService。

最後講「什麼時候用」,給你一條經驗法則。當你的代理需要做的,不只是回答單一一個問題的時候,就該用記憶這個模式。具體來說,就是當它需要維持對話情境、需要追蹤多步驟的任務、需要回想偏好跟歷史來做個人化、或者需要根據過去的成敗跟新資訊來學習調適的時候。

接著我把這一章最該帶走的幾個重點,再串一次。第一,對話式 AI 同時仰賴短期跟長期兩種記憶——短期是單一聊天的即時情境,暫時而且常受限於情境視窗;長期是跨工作階段的持久知識,靠向量資料庫這類外部儲存來保存、靠搜尋來存取。第二,ADK 用三樣東西管記憶:Session 是聊天執行緒,State 是暫時的聊天資料,MemoryService 是可搜尋的長期知識,而 SessionService 負責工作階段的整個生命週期。第三,那份狀態字典是暫時資料,它的前綴 user、app、temp 界定了歸屬跟留存,更新它要用 state_delta 或 output_key,絕對不要直接去改字典。第四,LangChain 的 ConversationBufferMemory 會自動把對話歷史注入提示;LangGraph 則用儲存庫來儲存跟檢索三種東西——語意的事實、情節的經驗、還有可更新的程序規則,實現比較進階的長期記憶。第五,Memory Bank 是一個受管理服務,會自動擷取、儲存、回想使用者專屬的資訊,而且能跨 ADK、LangGraph 跟 CrewAI 提供個人化又連續的對話。

書裡還配了一張示意圖,我用講的描述給你聽,因為它其實很單純:使用者送出一個提示給代理,代理在跟記憶之間做了「雙向」的互動之後——也就是它既會去讀記憶、也會把新東西寫進記憶——接著才產生輸出,回傳給使用者。這張圖想傳達的核心就是:記憶不是放在旁邊的擺設,它是夾在「收到提示」跟「給出回應」中間、會被讀也會被寫的那一環。

最後做個總結。記憶管理這個模式,本質上就是去區分「短暫的情境」跟「需要長期留存的知識」,並且決定它們各自怎麼建立、怎麼被應用;而 Google ADK 就是用 Session、State 跟 MemoryService 這三樣來處理這件事。記住這個短期對長期的二分法,你就抓住了這一章的靈魂。而下一個要談的模式,叫做「學習與調適」,英文是 Learning and Adaptation,它會接著討論代理如何根據新的經驗,去改變自己的思考方式、行動方式,還有它所知道的內容。我們下一講見。
