歡迎來到第一章,我們要談的主題叫做「提示鏈」,英文是 Prompt Chaining。這一章其實是整本書的地基,因為你之後會看到的很多代理設計模式,骨子裡都用到了它。我會帶你走三條主線:第一,先搞懂提示鏈到底是什麼、為什麼需要它;第二,看它在實務上能解決哪些真實場景的問題;第三,我們會一起想像一段實際的程式怎麼把兩個步驟串起來,最後再延伸談一個更進階的觀念,叫做情境工程。準備好了,我們就開始。

先說說提示鏈是什麼。提示鏈又有另一個名字,叫做管線模式,英文是 Pipeline pattern。它的核心精神其實只有一句話,就是「分而治之」。意思是,當你面對一個很大、很複雜的任務時,不要妄想用一句話就叫語言模型一次做完,而是把這個大問題,拆成一連串的小問題,每一個小問題都用一個專門設計的提示去處理。這裡有個關鍵的動作:你會把前一步的輸出,直接拿來當作下一步的輸入。一棒接一棒,像接力賽一樣傳下去。

順帶解釋一下,我這裡說的「提示」,英文是 prompt,就是你交給語言模型的那段指令或問題。而「語言模型」,英文全名是 Large Language Model,大型語言模型,後面我會簡稱它為 LLM。

那麼這種一步接一步的循序處理,到底好在哪裡?第一個好處是「模組化」。因為每一步都只專心做一件事,只聚焦在單一面向上,所以這一步變得很容易理解、很容易除錯,也很容易單獨去最佳化。你可以把它想成工廠的生產線,每一個工位只負責鎖一顆螺絲,而不是要求一個人從頭到尾組裝整台車。哪個工位出問題,你一眼就看得出來。

第二個關鍵,是前一步的輸出會變成後一步的輸入,這就形成了一條環環相扣的「依賴鏈」。「提示鏈」這個名字裡的「鏈」,就是從這裡來的。這條鏈的意義在於,先前累積下來的情境,會一路引導後面的處理,讓整個流程不會走偏。

第三個能力很重要,就是在每一步裡面,你還可以指示 LLM 去跟外部的系統、API,或是資料庫互動。這一點讓它的能力突破了一個天花板。要知道,語言模型本身只懂它訓練時看過的資料,但只要讓它在某一步去查資料庫、去呼叫一個外部服務,它就能取得訓練資料裡根本沒有的、最新的或專屬的資訊。

也正因為有這些特性,提示鏈成了打造 AI 代理的基礎。AI 代理,英文是 AI agent,你可以把它理解成一個能自己作主的程式。代理能夠自主地規劃、推理,然後採取行動,去完成那種需要好幾個步驟的推理與決策。而它能這樣一步步走下去的底層機制,很多時候就是提示鏈。

講到這裡你可能會問:那為什麼不乾脆寫一句很長很完整的提示,把所有要求一次塞進去就好?這就要談到單一提示的侷限了。當任務是多面向的時候,一個又長又複雜的提示,很容易出五種毛病。第一是「指令忽略」,你交代了五件事,它只做了三件。第二是「情境漂移」,做著做著就離題了。第三是「錯誤傳播」,前面誤解了一個小地方,後面整串都跟著錯。第四是「情境視窗不足」,情境視窗你可以理解成模型一次能讀進去的記憶容量,東西塞太多就爆掉、就裝不下了。第五是「幻覺」,也就是模型開始一本正經地胡說八道,編造出根本不存在的資訊。

我舉個具體的例子你就懂了。假設你給模型一個任務:「請分析這份市場研究報告、摘要裡面的發現、找出有數據支撐的趨勢,最後幫我草擬一封郵件。」這一句話裡其實藏了四個任務。結果模型可能摘要做得還不錯,但輪到要精準擷取數據的時候就出包了,郵件也草擬得零零落落。因為你逼它一心四用,它就樣樣都做不精。

那提示鏈怎麼解決這個問題?它把剛剛那個四合一的任務,拆成幾個各自聚焦的步驟。第一個提示只做摘要,你告訴它:「請摘要以下這份市場研究報告的關鍵發現。」它唯一的焦點就是摘要,別的都不用管。第二個提示做趨勢辨識,你告訴它:「根據上一步給你的這份摘要,找出前三大新興趨勢,並且把支撐每一項趨勢的具體數據也擷取出來。」注意,這一步是建立在第一步已經驗證過的、乾淨的輸出之上。第三個提示才寫郵件:「請根據上一步整理出來的趨勢跟數據,寫一封簡潔的郵件給行銷團隊。」

你會發現,拆開以後,每一步都變得更簡單、更不模稜兩可,模型的認知負荷大幅下降,最後產出的品質自然就更可靠。而且還有一個加分技巧:你可以幫每一個階段都賦予一個明確的角色。比方說第一步請它扮演「市場分析師」,第二步請它扮演「交易分析師」,第三步請它扮演「專業的文件撰寫者」。角色一給,它的回答口吻跟專業度就更到位。

不過這裡有一個很容易被忽略、但其實攸關成敗的環節,就是「結構化輸出」。提示鏈的可靠性,完全建立在步驟與步驟之間,資料能不能完整、乾淨地交接。你想想看,如果某一步吐出來的東西含糊不清、格式亂七八糟,下一步接到這種爛輸入,當然就跟著失敗。所以,指定一個結構化的格式就非常重要,常見的格式像是 JSON 或 XML。

JSON 你可以理解成一種有固定欄位、機器一看就懂的資料寫法。舉個例子,剛剛那個趨勢辨識的步驟,輸出就可以要求它整理成這樣的結構:裡面有一個趨勢清單,清單裡每一筆都有兩個固定欄位,一個叫趨勢名稱,一個叫支撐數據。比方說第一筆,趨勢名稱是「AI 驅動的個人化」,支撐數據是「百分之七十三的消費者,偏好跟那些會運用個人資訊、讓購物體驗更貼近自己需求的品牌往來」。第二筆,趨勢名稱是「永續且符合道德的品牌」,支撐數據是「過去五年間,帶有 ESG 相關訴求的產品,銷售成長了百分之二十八,而不帶這類訴求的只成長了百分之二十」。順帶一提,ESG 指的是環境、社會與公司治理這三個面向的永續指標。重點不在於這些數字本身,而在於這種固定格式讓資料變得機器可讀,下一步可以精準地把它解析出來、塞進新的提示裡,把「誤讀自然語言」這種錯誤降到最低。

好,觀念的部分講完了,接下來我們進入第二條主線,看看提示鏈在實務上到底能用在哪些地方。我整理了七種典型的應用場景,我們一個一個來。

第一種,是資訊處理的工作流程。很多任務的本質,就是對原始資訊做好幾次轉換。比方說,第一個提示先從指定的網址或文件,把文字內容抓下來。網址的英文縮寫是 URL,就是網頁的地址。第二個提示,把清理過後的文字做摘要。第三個提示,從摘要或原文裡擷取特定的實體,像是人名、日期、地點。第四個提示,拿這些實體去搜尋公司內部的知識庫。第五個提示,把摘要、實體跟搜尋結果全部整合起來,生成一份最終報告。這整套流程,可以拿來自動化內容分析、做 AI 研究助理,或是產生複雜的報告。

第二種,是複雜問題的回答。有些問題,你沒辦法一步答完,它需要好幾步的推理,或是好幾次的查找。舉個例子:「一九二九年股市崩盤的主要成因是什麼?政府的政策又是怎麼回應的?」這時候你可以這樣拆:第一個提示先辨識出這個問題裡藏了哪幾個核心子問題,這裡顯然有兩個,一個是崩盤的成因,一個是政府的回應。第二個提示去研究一九二九年崩盤的成因。第三個提示去研究政府對這場崩盤的政策回應。第四個提示,把第二步跟第三步的資訊綜合起來,回答最一開始那個完整的問題。

這裡有一個很值得記住的觀念。研究型的代理,常常會先「平行」地去檢索好幾篇文章。平行的意思是同時一起做、而不是排隊一個一個做。因為這些檢索任務彼此獨立、互不相干,所以很適合一起並行處理,省時間。但是,等到所有文章都抓回來之後,流程就會轉成「循序」的:先彙整、再綜整成初稿、最後審閱精煉成最終報告,這些步驟前後相依,就必須一步接一步。所以你會發現,複雜的操作常常是兩種策略的混搭:獨立蒐集的部分用平行處理,彼此相依的步驟則用提示鏈。

第三種,是資料的擷取與轉換,就是把一堆雜亂的、非結構化的文字,變成整齊的結構化格式。這個過程通常需要反覆地修正。我用發票來舉例。第一個提示,先從發票文件裡擷取特定欄位,像是姓名、地址、金額。接著進入一個處理環節:檢查這些必填欄位有沒有齊全、格式對不對。然後是一個「條件式」的第二提示,意思是它不一定每次都跑,只有在發現有缺漏或格式錯誤的時候才啟動,專門去找那些缺掉的資訊,而且你還可以把上一次失敗的情境附給它參考。找回來之後再驗證一次,需要的話就重複這個循環,直到輸出一份已經擷取好、也驗證過的結構化資料。

這個場景特別適合處理表單、發票、郵件。再舉個更實際的:處理 PDF 表單的光學文字辨識。光學文字辨識的英文縮寫是 OCR,簡單說就是把圖片上的字認出來變成文字。你可以先用一步把文字抓出來,再用下一步做正規化,比方說把中文寫的「一千零五十」轉換成阿拉伯數字的一零五零。這裡還有一個很實用的提醒:精確的數學運算,其實是語言模型的弱項,它算數常常出錯。所以聰明的做法,是把算術這件事「委派」給一個外部的計算機工具去做,而不是硬要模型自己算。所以這條鏈,其實串起了三件事:文字擷取、資料正規化,再加上外部工具的使用。這樣組合起來,就能達到單純問一句話很難穩定拿到的準確結果。

第四種,是內容生成的工作流程。複雜的內容寫作,通常會拆成發想、大綱、初稿、修訂這幾個階段。比方說,第一個提示根據使用者的興趣,先生成五個主題構想。接著由使用者或程式自動挑出最好的那一個。第二個提示根據選定的主題,生成一份詳細的大綱。第三個提示根據大綱的第一個重點,寫出初稿的第一段。第四個提示寫第二個重點,而且要把前一段也附上去當作情境,讓文氣連貫,就這樣一個重點一個重點寫下去。最後第五個提示,通篇審閱、精煉整份初稿的連貫性、語氣跟文法。這套流程可以用在創意敘事,也可以用在技術文件這類結構分明的寫作。

第五種,是具狀態的對話式代理。這裡的「狀態」,你可以理解成「對話到目前為止累積下來的記憶」。提示鏈提供了維繫對話連續性的基礎機制:它把每一輪對話,都當成一個新的提示來處理,但會有系統地把先前互動裡的資訊或實體,納進來,以維持情境。具體來說,第一個提示先處理使用者講的第一句話,辨識出他的意圖跟關鍵實體。接著用這個意圖跟實體去更新對話狀態。第二個提示根據當前的狀態生成回應,同時辨識出「接下來還缺哪些資訊」。之後每一輪都這樣重複,使用者每講一句新的話,都會啟動一條新的鏈,而這條鏈會運用到前面累積下來的整個對話歷史。保留對話歷史,正是對話式代理能在多輪你來我往之間維持情境的根本。

第六種,是程式碼的生成與精煉。寫出一段能跑的程式碼,通常也是個多階段的過程。第一個提示先理解使用者的需求,產生虛擬碼或大綱。虛擬碼的意思是用接近白話的方式描述程式邏輯,還不是真正的程式語言。第二個提示根據大綱寫出初版的程式碼。第三個提示去找出潛在的錯誤或可以改進的地方,這一步可以用靜態分析工具,也可以再呼叫一次 LLM 來檢查。第四個提示根據找到的問題,重寫或精煉程式碼。第五個提示加上文件說明或測試案例。這個場景的關鍵價值在於,它允許你在一次次模型呼叫的「中間」,插入確定性的邏輯。確定性,英文是 deterministic,意思是這段邏輯的行為是固定的、可預期的,不像模型那樣每次回答都可能不太一樣。你可以靠這種確定性邏輯去做中介處理、驗證輸出、做條件分支。這樣一來,就把原本一個多面向的請求,轉變成一連串由執行框架嚴格管理的、結構化的操作序列。

第七種,是多模態與多步推理。多模態,英文是 multimodal,意思是資料不只一種型態,可能同時有文字、圖片、表格。分析這種資料,一樣要先把問題拆成子任務。比方說要解讀一張圖片,這張圖裡有嵌在裡面的文字、有標籤,還有表格資料。你可以這樣拆:第一個提示先從圖片裡擷取並理解那些文字。第二個提示把擷取到的圖片文字,跟對應的標籤連結起來。第三個提示運用表格去詮釋蒐集到的所有資訊,判斷出最終需要的輸出是什麼。

七種場景講完了,我們進到第三條主線:動手想像一段實際的程式長什麼樣子。提示鏈的實作,規模可大可小。小的時候,它可能就只是一段腳本裡,前後循序呼叫幾個函式而已。大的時候,你會用上專門的框架,來幫你管理控制流程、狀態,還有各種元件的整合。常見的框架有 LangChain、LangGraph、CrewAI,還有 Google 出的代理開發套件,英文叫 Agent Development Kit,縮寫是 ADK。對於複雜的架構,這些框架特別好用。

我們接下來要描述的範例,用的是 LangChain 跟 LangGraph 這一對。它們的分工是這樣:LangChain 提供的是「線性序列」的基礎抽象,適合一條直線走到底的流程;而 LangGraph 則更進一步,能支援「具狀態」而且「帶有循環」的運算,也就是可以記住東西、還可以繞回頭重做。我們要示範的,是一條只有兩步的提示鏈:第一步,解析一段雜亂的、非結構化的文字,把資訊擷取出來;第二步,把擷取出來的結果,轉換成結構化的格式。

那這段程式在做什麼呢?我幫你用聽得懂的方式描述一遍,你不需要去記任何符號。首先,在開始之前,要先安裝幾個函式庫,並且設定好你選用的模型供應商的 API 憑證。憑證你可以想成是一把鑰匙,沒有它就沒辦法呼叫模型。範例用的是 OpenAI,但你也可以換成 Google Gemini 或 Anthropic,只要換掉對應的套件就好。

接著程式做了幾件事。第一,它初始化一個語言模型,並且把一個叫做「溫度」的參數設成零。溫度控制的是模型回答的隨機程度,設成零代表我們希望它的輸出盡量穩定、可重現,每次都差不多,這對需要精確的資料處理任務很重要。

第二,它定義了第一個提示,內容大意是:「請從以下這段文字裡,擷取出技術規格。」這個提示留了一個空位,等著把實際的輸入文字填進去。

第三,它定義了第二個提示,內容大意是:「請把以下這份規格,轉換成一個 JSON 物件,並且用三個固定的鍵,分別是 cpu、memory、storage,也就是處理器、記憶體跟儲存空間。」這個提示也留了一個空位,等著接收上一步擷取出來的規格。

第四,也是最精彩的地方,它把這些零件串起來。範例用的是一種叫做 LCEL 的寫法,全名是 LangChain Expression Language,LangChain 表達式語言。它用一個直立的線條符號,把提示、模型、還有一個輸出解析器,像水管接頭一樣一節一節接起來。這裡那個輸出解析器的作用,是把模型回傳的訊息物件,轉換成單純好用的純文字字串。於是第一條鏈,負責擷取規格;然後完整的鏈,會自動把第一條鏈擷取出來的結果,餵進第二個轉換提示的那個空位裡,再交給模型轉成 JSON。整條水管就這樣打通了。

最後,程式餵進去一段輸入文字,內容是:「這款新筆電搭載三點五 GHz 的八核心處理器、十六 GB 記憶體,以及一 TB 的 NVMe 固態硬碟。」這是一段人類自然書寫的描述。它經過剛剛那兩步的處理,第一步先把規格從這句話裡認出來,第二步再把它整理成乾淨的、有 cpu、memory、storage 三個欄位的 JSON,最後印出來。你看,一段口語化的文字,就這樣被可靠地變成了機器能直接使用的結構化資料。這就是提示鏈最樸素、也最有代表性的樣貌。

講完實作,我想帶你延伸到一個更高層次的觀念,叫做「情境工程」,英文是 Context Engineering。如果說傳統的提示工程,在意的是「我這一句問題要怎麼措辭才漂亮」,那情境工程的格局就大得多。它是一門系統化的學科,核心任務是:在模型開始生成內容之前,為它設計、建構並交付一整套完整的資訊環境。

這裡有一個很反直覺、但很重要的主張:模型輸出的品質,其實比較少取決於模型本身的架構有多厲害,反而更多取決於你提供給它的「情境」有多豐富。換句話說,餵給它什麼,往往比它本身多聰明,更能決定結果好壞。

我們把情境工程跟傳統的提示工程對照一下。提示工程,英文是 Prompt Engineering,著重的是優化「當下這句查詢」的措辭。而情境工程,把範圍一口氣擴展到好幾個資訊層次。第一層是系統提示,英文是 system prompt,它像是給模型的角色設定與基本守則,比方說「你是一位技術文件撰寫者,語氣必須正式而且精確」。第二層是外部資料,像是檢索回來的文件、工具跑出來的結果。第三層,也是最關鍵的,是隱性資料,包括使用者的身分、過去的互動歷史、以及當下的環境狀態。重點是,就算是再先進的模型,如果它對自己所處的運作環境視野很差、什麼都不知道,它一樣會表現得很糟。

這等於是把整個任務,從「回答一個問題」,重新定位成「為代理建構出一幅完整的運作全貌」。我舉個生活化的例子。一個經過情境工程的代理,在幫你寫郵件之前,會先做好幾件事:它會去整合你行事曆上的空檔,這是工具的輸出;它會去理解你跟這封信收件人之間的專業關係,這是隱性資料;它還會去翻你們先前的會議筆記,這是檢索回來的文件。把這些通通納進來之後,它才動筆。這樣產生出來的郵件,才會是高度相關、又貼合你個人情況的。

那「情境工程」裡的「工程」兩個字,體現在哪裡?它涉及兩件事。一是要建立穩健的管線,在執行的當下即時地把資料擷取進來、轉換好。二是要建立回饋迴路,持續地改善情境的品質,讓它越用越準。實作上,這個過程可以用專門的系統來大規模自動化。舉個例子,Google 的 Vertex AI 提示優化器,能夠根據你給的範例輸入跟評估指標,系統化地去評估模型的回應,在不同模型之間調整提示跟系統指令的時候特別有用,讓你不必每換一個模型就大量手工重寫。

所以情境工程的精髓,是把「情境」當成第一優先的元件來看待,它真正在乎的是:這個代理「知道什麼、什麼時候知道、又該怎麼運用」。而這,正是把一個無狀態的、英文叫 stateless、也就是講完就忘、記不住前文的聊天機器人,升級成一個具備情境感知能力、真正能幹活的高能力系統的核心關鍵。

好,我們把這一整章的脈絡,用聽的方式再串一次,做個收尾。

先說「是什麼」。如果你硬把一個複雜任務,塞進單獨一個提示裡,往往會讓 LLM 不堪負荷。認知負荷一高,它忽略指令、遺失情境、產生錯誤資訊的機率就跟著上升。這種單體式的提示,很難同時管好多重限制跟循序推理,結果就是輸出不可靠。

再說「為什麼」要用提示鏈。因為它把複雜問題,拆成一連串較小、彼此相連的子任務,每一步只用一個聚焦的提示,執行一個特定操作,然後把輸出交給下一步,串成一條有邏輯的工作流程。這麼做,大幅提升了可靠性跟可控性。而且這種模組化的策略更好管理、更好除錯,還能順勢整合外部工具或結構化資料。它就是建構多步驟代理系統的地基。

最後是「什麼時候該用」,給你一個經驗法則。當你的任務複雜到單一提示已經應付不來、當任務涉及好幾個處理階段、當你需要在步驟之間跟外部工具互動,或者當你正在打造一個需要多步推理、還要維持狀態的代理系統的時候,就該把提示鏈搬出來用。

把今天的要點濃縮成幾句話帶走:第一,提示鏈用「分而治之」,把複雜任務拆成較小、聚焦的步驟,它有時候也被叫做管線模式。第二,每一步都會涉及一次 LLM 呼叫,或是一段處理邏輯,並且以前一步的輸出作為它的輸入。第三,這個模式大幅提升了複雜 LLM 互動的可靠性跟可管理性。第四,像 LangChain、LangGraph,還有 Google 的 ADK 這些框架,都提供了工具來幫你定義、管理跟執行這些多步驟的序列。

作為一個最基礎的模式,提示鏈讓我們有可能去打造出真正精密的 AI 代理,具備多步推理、工具整合,還有狀態管理的能力,而這些本事,遠遠超出單獨一句提示所能企及的範圍。這就是我們整本書的起點,後面所有更進階的模式,都是站在這條鏈的肩膀上長出來的。
