這一講,我們要談的是讓不同 AI 代理彼此對話的方法,也就是「代理間通訊」,英文叫 Inter-Agent Communication,常用的縮寫是 A2A,也就是 Agent to Agent,代理對代理的意思。在開始之前,我先幫你把這一講的路線圖講一遍。我們會先弄清楚為什麼需要這個東西,接著拆解 A2A 的幾個核心概念,包括有哪些角色、什麼是代理卡、代理怎麼被找到、訊息和任務怎麼傳遞、有哪幾種互動方式,以及安全性怎麼處理。然後我會把 A2A 跟另一個很容易混淆的協定 MCP 做個對照,再講幾個實際的應用情境,最後帶你走過一段真實的範例程式,看看一個 A2A 代理到底是怎麼被架起來的。

我們先從「問題」說起。就算一個 AI 代理本身的能力再強,當它要處理那種很複雜、有很多面向的問題時,單打獨鬥往往還是會碰到天花板。你可以把它想成一個很厲害的個人工作者,但再厲害的個人,面對一個橫跨好幾個專業領域的大專案,還是需要找別人合作。問題是,這些 AI 代理可能是不同人、用不同工具打造出來的。有的人用 LangGraph,有的人用 CrewAI,有的人用 Google 的 ADK,也就是 Agent Development Kit,代理開發套件。這些代理就像講不同母語的專家,彼此之間沒有一套共通的語言,沒辦法順順地協調、把任務交辦出去、或是交換資訊。這種各自孤立的狀態,正是阻礙我們組出更強大系統的原因。

為了解決這個問題,Google 提出了 A2A 協定。它是一項「開放標準」,英文叫 open standard,意思是規格是公開的,任何人都可以照著實作。它的目標,就是建立一套通用的通訊方式,讓不同框架做出來的代理可以有效協作。這裡有個很關鍵的詞叫「互通性」,英文是 interoperability,意思是不管你這個代理是用哪一種技術寫的、是誰做的,只要大家都講 A2A 這套語言,就能一起工作。

這套協定背後的支持陣容其實相當龐大。包括 Atlassian、Box、LangChain、MongoDB、Salesforce、SAP、ServiceNow 這些公司都站台支持。微軟也計畫把 A2A 整合進它的 Azure AI Foundry 跟 Copilot Studio 裡面,展現它對開放協定的投入。另外像 Auth0 跟 SAP,也正在把 A2A 的支援放進自己的平台跟代理裡。而且因為 A2A 是「開源」的,英文是 open source,所以它歡迎社群一起貢獻,讓這套標準持續演進、被更多人採用。你只要記得一個重點：A2A 不是某一家公司關起門來自己玩的東西,它正在變成業界一個共同的基礎建設。

接下來,我們進入這一講的核心,把 A2A 的幾根基礎支柱一根一根拆開來講。這幾根支柱分別是：核心參與者、代理卡、代理探索、通訊與任務、互動機制,以及安全性。如果你之後要開發跟 A2A 相容的系統,這幾個概念是一定要徹底搞懂的。我們一個一個來。

第一根支柱,是「核心參與者」,英文叫 Core Actors。在 A2A 的世界裡,主要有三個角色。第一個是「使用者」,也就是 User,就是那個發起請求、想要代理幫忙做事的人。第二個是「A2A 用戶端」,英文是 A2A Client,我們也可以叫它「用戶端代理」,Client Agent。它是一個應用程式或是一個 AI 代理,負責「代表使用者」去向別人請求動作或資訊。你可以把它想成幫你跑腿、幫你發問的那個代理。第三個是「A2A 伺服器」,英文是 A2A Server,也叫「遠端代理」,Remote Agent。它是真正提供服務的那一方,對外開了一個 HTTP 的端點,負責接收用戶端的請求、處理完之後把結果回傳。這裡有一個很重要的設計觀念：遠端代理對用戶端來說,是一個「不透明」的系統,英文叫 opaque。不透明的意思是,用戶端根本不需要知道這個遠端代理內部是怎麼運作的、用什麼模型、跑什麼邏輯,它只要知道「我問它問題、它會給我答案」就好。這就像你打電話叫外送,你不需要知道餐廳廚房裡是怎麼炒菜的,你只要會點餐、會收餐就行。

第二根支柱,是「代理卡」,英文叫 Agent Card。這是整個 A2A 裡面非常核心的東西,你可以把它理解成一張代理的「數位名片」或「數位身分證」。它通常是一個 JSON 格式的檔案,裡面寫清楚了別人要跟這個代理互動、或是要自動發現這個代理時,需要知道的所有關鍵資訊。具體來說,代理卡會包含代理的身分、它的端點網址、版本號;會說明它支援哪些能力,比如說支不支援串流、支不支援推播通知;會列出它有哪些具體的技能;會講它預設接受什麼輸入、產出什麼輸出;還會寫明它的驗證需求,也就是你要怎麼證明你有權限用它。

我舉一個書裡的例子讓你想像一下。假設有一個叫做 WeatherBot 的天氣機器人,它的代理卡裡面會寫:名字叫 WeatherBot,描述是「提供準確的天氣預報跟歷史資料」,然後有一個網址當作它的服務位置,版本是一點零。在能力這一塊,它標明支援串流、不支援推播通知、但會保留狀態轉換的歷史紀錄。在驗證這一塊,它說要用 API 金鑰。預設輸入跟輸出都是純文字。最關鍵的是它列出來的技能,這個 WeatherBot 有兩個技能。第一個技能叫「取得目前天氣」,描述是檢索任何地點的即時天氣,還附了兩個範例問句,像是「巴黎現在的天氣如何?」或是「東京目前的天氣狀況」,並且打上了天氣、即時這類標籤。第二個技能叫「取得預報」,描述是取得未來五天的天氣預測,範例問句像是「紐約未來五天的預報」或是「這個週末倫敦會下雨嗎?」。你聽到這裡應該就懂了,代理卡的價值在於:別的代理光是讀這張卡,就能知道「這個天氣機器人會做什麼、怎麼問它、怎麼跟它驗證」,完全不需要事先跟它的開發者開會溝通。

第三根支柱,是「代理探索」,英文叫 Agent Discovery。光有名片還不夠,你得有辦法「找到」這張名片,這就是代理探索要解決的事。它讓用戶端能夠找到那張代理卡,進而知道某個 A2A 伺服器有哪些能力。書裡講了三種策略。第一種叫「眾所周知的 URI」,英文是 Well-Known URI。它的做法是,代理把自己的代理卡放在一個標準化、大家都約定好的路徑上,比方說網域底下一個固定的位置。這種做法的好處是存取很廣泛、通常可以自動化,適合公開的、或是特定領域通用的用途。你可以把它想成「大家都知道公司的官網首頁在哪,直接去那裡就找得到資料」。第二種叫「策展註冊中心」,英文是 Curated Registries。它提供一個集中式的目錄,代理卡會發布在這裡,而且可以依照特定條件去查詢。這比較像企業內部的「員工通訊錄」或「黃頁」,適合那種需要集中管理、需要存取控制的企業環境。第三種叫「直接設定」,英文是 Direct Configuration。就是把代理卡的資訊直接嵌進去、或是私下分享給對方。這適合那種彼此關係很緊密、或是私有的系統,在這種情況下根本不需要動態去探索,大家本來就互相認識。

不管你選哪一種探索方式,書裡特別提醒,要確保代理卡端點的「安全」。因為代理卡裡面雖然不一定是機密,但常常含有一些敏感資訊。要保護它,可以用存取控制、用相互 TLS,也就是 mutual TLS,縮寫 mTLS,我等一下講安全性時會再展開,或是用網路層面的限制。

第四根支柱,是「通訊與任務」,英文是 Communications and Tasks。這部分講的是代理之間實際上怎麼傳東西。在 A2A 裡面,通訊是圍繞著「非同步任務」來組織的,英文是 asynchronous tasks。非同步的意思是,你發出請求之後不需要乾等在那裡,可以先去做別的事。這些任務代表的是一個「長時間執行流程」的基本工作單位。每一個任務都會被指派一個獨一無二的識別碼,然後它會經歷一連串的狀態變化,比方說從「已提交」submitted、到「處理中」working、再到「已完成」completed。這種用狀態來追蹤的設計,讓系統可以同時平行處理很多複雜的操作。

那代理之間具體傳的是什麼?是一個「訊息」,英文是 Message。一則訊息裡面包含兩種東西。一種叫「屬性」,英文是 attributes,它是用來描述這則訊息的鍵值對中介資料,比方說這則訊息的優先順序、或它的建立時間。另一種,也是更重要的,叫「部分」,英文是 parts,它負責承載真正要傳遞的內容,可能是一段純文字、一個檔案,或是一段結構化的 JSON 資料。一則訊息可以有一個或多個部分。那代理在執行任務的過程中產生出來的具體成果,我們叫它「產物」,英文是 artifacts。產物跟訊息一樣,也是由一個或多個部分組成,而且很棒的一點是,產物可以在結果一點一點生出來的時候,就以「增量」的方式串流傳出去,不用等全部做完才一次給你。

那這些東西底層是怎麼傳的?A2A 框架裡所有的通訊,都是透過 HTTP 或加密的 HTTPS,並且使用一個叫做 JSON-RPC 2.0 的協定來包裝傳遞的內容,也就是所謂的「酬載」,英文叫 payload。JSON-RPC 你可以簡單理解成一種用 JSON 格式來做「遠端程序呼叫」的規矩,也就是我這邊寫好一個請求、指定要呼叫哪個方法、帶上哪些參數,對方照著做、再回我結果。最後還有一個小但重要的設計:為了在多次來回互動之間維持連續性,系統會用一個由伺服器產生的識別碼,叫 contextId,把相關的任務分在同一組,這樣情境就能被保存下來。打個比方,contextId 就像客服系統裡的「同一個案件編號」,不管你來回問了幾次,只要案件編號一樣,客服就知道這些對話是連在一起的。

第五根支柱,是「互動機制」,英文是 Interaction Mechanisms。前面說過 A2A 是非同步任務導向的,但實際上它提供了好幾種不同的互動方式,來滿足各種不同的 AI 應用需求。我幫你整理成四種,你聽的時候可以想像它們各自適合什麼場景。

第一種是「同步請求與回應」,英文是 Synchronous Request Response。同步的意思就是你問了、就在那邊等,等到伺服器處理完、在同一次交換裡把完整答案給你。這適合那種很快、即時就能完成的操作。就像你問人「現在幾點?」,對方馬上回你,你站在那等個一兩秒就好。

第二種是「非同步輪詢」,英文是 Asynchronous Polling。這適合需要比較久才能處理完的任務。流程是這樣:用戶端送出請求,伺服器不會馬上給答案,而是立刻回你一個「處理中」的狀態,外加一個任務編號,等於跟你說「收到了,我在做了,這是你的單號」。然後用戶端就可以先去忙別的,之後再三不五時拿這個單號去問伺服器「好了沒?」,這個動作就叫輪詢,英文是 poll。一直問到任務變成「已完成」或「失敗」為止。這就像你去餐廳拿了個取餐號碼牌,然後每隔一陣子抬頭看一下螢幕上叫到幾號了。

第三種是「串流更新」,英文是 Streaming Updates,它用到的技術叫「伺服器推送事件」,英文全名是 Server-Sent Events,縮寫 SSE。這種方式適合需要即時、一點一點收到結果的場景。它會建立一條從伺服器到用戶端的「持久性、單向」的連線。持久性的意思是這條線會一直開著,單向的意思是資料只從伺服器這頭往用戶端那頭流。這樣一來,遠端代理就可以持續地把更新推過去,比方說狀態變了、或是產生了一部分結果,而用戶端完全不用一次又一次地發問。這很像你看體育賽事的文字直播,你不用一直按重新整理,畫面自己會一條一條跳出新進度。

第四種是「推播通知」,英文是 Push Notifications,它用到的技術叫 Webhook。這種方式是專門為那種非常耗時、非常吃資源的任務設計的。在這種任務裡,你要嘛一直開著一條連線、要嘛一直去輪詢,都很沒效率。所以做法變成:用戶端事先註冊一個 webhook 的網址,等於留一個「回撥地址」給伺服器。之後當任務狀態出現重大變化,比方說做完了,伺服器才主動送一個非同步通知,「推」一次到那個網址。這就像你網購之後不用一直盯著物流頁面,東西到了系統自己發簡訊通知你。

關於互動機制,有兩個重點要補充。第一,一個代理到底支不支援串流、支不支援推播通知,都會明明白白寫在它的代理卡裡面,所以你看卡就知道。第二,A2A 是「與模態無關」的,英文叫 modality-agnostic。模態指的是資料的型態。這句話的意思是,A2A 不只能拿來傳文字,它同樣能用在音訊、視訊這些其他類型的資料上,所以你可以用它來打造很豐富的「多模態」AI 應用,英文是 multimodal,也就是同時處理多種型態資料的應用。

這裡我用兩個真實的請求範例,幫你把同步跟串流的差別講得更具體一點,我不會去唸那些括號引號,只講它在做什麼。先看同步的例子。它是一個 JSON-RPC 的請求,協定版本標明是 2.0,有一個請求編號,然後它呼叫的方法叫 sendTask,也就是「送出任務」。參數裡面帶了一個任務編號、一個工作階段編號,還有一則來自使用者的訊息,訊息的內容是一段文字,問的是「美元兌歐元的匯率是多少?」,並且指定它接受純文字的輸出格式,還說保留五筆歷史紀錄。重點是,用同步的 sendTask 這個方法,代表用戶端要的是「對這個問題給我一個單一、完整的答覆」,問一次、答一次,乾淨俐落。

再看串流的例子。它的結構很像,一樣是 JSON-RPC 2.0、有編號、有任務編號跟工作階段編號,訊息內容換成了「今天日圓兌英鎊的匯率是多少?」。但關鍵差別在於,它呼叫的方法不是 sendTask,而是 sendTaskSubscribe,也就是「送出任務並訂閱」。用這個方法,等於建立了一條持久性連線,讓代理可以隨著時間,陸續回傳好幾筆增量的更新或部分結果。所以你只要記住這組對照:要一次性的完整答案,用 sendTask;要持續、分批的更新,用 sendTaskSubscribe。一個是問完就結束,一個是訂閱之後源源不絕。

第六根支柱,也是最後一根,是「安全性」,英文是 Security。在多代理系統裡,代理之間要安全又順暢地交換資料,安全性是架構裡至關重要的一環。A2A 內建了好幾種機制來確保它的穩健跟完整。我講四個。第一個是「相互傳輸層安全性」,英文全名是 Mutual Transport Layer Security,縮寫就是前面提過的 mTLS。一般我們上網用的 TLS,是用戶端去驗證伺服器是不是真的;而「相互」的意思是,雙方互相驗證對方的身分,並且把連線加密。這樣可以防止未經授權的存取,也防止資料在傳輸過程中被攔截。第二個是「全面的稽核日誌」,英文是 Comprehensive Audit Logs。意思是,代理之間所有的通訊都會被詳實記錄下來,包括資訊往哪裡流、牽涉到哪些代理、做了哪些動作。這份完整的軌跡,對於事後追究責任、排查問題、做安全分析,都非常關鍵。第三個是「代理卡宣告」,英文是 Agent Card Declaration。前面講代理卡的時候提過,驗證需求會直接寫在代理卡裡面。這麼做的好處是把安全政策集中起來、簡化了驗證的管理,你看卡就知道要怎麼通過驗證。第四個是「憑證處理」,英文是 Credential Handling。代理通常會用安全的憑證來驗證身分,比方說 OAuth 2.0 的權杖、或是 API 金鑰。重點在於,這些憑證是透過 HTTP 的標頭,也就是 header 來傳遞的,而不是塞在網址裡或訊息本體裡。這個設計很重要,因為網址常常會被記錄在各種日誌中,把憑證放在網址裡很容易外洩;放在標頭裡就安全多了。

講完六根支柱,我們來處理一個很多人會搞混的問題:A2A 跟 MCP 到底差在哪?MCP 的全名是 Model Context Protocol,也就是「模型情境協定」,是 Anthropic 提出的。書裡的定位很清楚:這兩個協定是「互補」的,不是互相取代。差別在於各自管的事不一樣。MCP 著重的是「為單一代理建構情境」,也就是讓一個代理去跟外部的資料、外部的工具打交道,把需要的材料餵給模型。而 A2A 著重的是「代理跟代理之間」的協調與通訊,讓任務委派跟協作變得可能。我給你一個好記的比喻:MCP 比較像是給一個工人配備工具箱跟原料,讓他能接觸到外面的資源把事情做好;而 A2A 比較像是讓好幾個工人之間能彼此講話、分工、把任務交辦來交辦去。一個是「對工具與資料」,一個是「對其他代理」。A2A 的目標,是提升效率、降低整合成本,並且在複雜的多代理 AI 系統開發裡,促進創新跟互通。所以說,把 A2A 的核心元件跟運作方式徹底搞懂,對於設計、實作這種協作型、可互通的代理系統來說,是不可或缺的。

理論講完了,我們來看 A2A 在實務上能用在哪。書裡點出三個典型的使用案例。第一個,也是最首要的,叫「多框架協作」,英文是 Multi-Framework Collaboration。這正是 A2A 最核心的價值:讓各自獨立的 AI 代理,不管底層是 ADK、LangChain 還是 CrewAI,都能彼此通訊、協作。當你要建構一個複雜的多代理系統,裡面不同的代理各自專精在問題的不同面向時,這一點就特別關鍵。第二個叫「自動化工作流程編排」,英文是 Automated Workflow Orchestration。在企業環境裡,A2A 可以讓代理彼此委派、協調任務,串出一個複雜的工作流程。書裡給的例子很具體:可能有一個代理負責一開始的資料蒐集,做完之後把工作委派給第二個代理去做分析,分析完再交給第三個代理去生成報告,而這一連串的接力,全都靠 A2A 協定來通訊。第三個叫「動態資訊檢索」,英文是 Dynamic Information Retrieval。代理可以透過通訊,即時去檢索、交換最新的資訊。比方說一個主要代理,可能會去向一個專門的「資料擷取代理」要即時的市場資料,而那個資料擷取代理就會去呼叫外部的 API,把資訊抓回來再回傳。你會發現,這三個案例的共同精神都是「模組化、可擴展、更聰明」:把大問題拆成小塊,讓專精的代理各做各的,再透過 A2A 把它們縫起來。

接下來進入動手實作的部分。我會帶你走過一段真實的範例程式,但我不會逐字唸程式碼,我會講它在做什麼、輸入是什麼、輸出是什麼、為什麼這樣設計。先給你背景:Google 有一個公開的範例儲存庫,裡面提供了 Java、Go 跟 Python 的範例,示範像 LangGraph、CrewAI、Azure AI Foundry、AG2 這些不同框架,怎麼用 A2A 來通訊,而且這些程式碼都是用 Apache 2.0 授權釋出的。我們要看的這個例子,是用 ADK 為基礎、搭配 Google 驗證工具,去架設一個 A2A 伺服器,具體的角色是一個「行事曆代理」。

第一段程式,是一個負責「建構代理」的非同步函式,可以把它想成一個工廠函式,你給它用戶端的憑證,它幫你組裝出一個代理回來。它做的事情有幾步。第一步,先用你給的用戶端編號跟密鑰,初始化一個「行事曆工具組」,英文是 CalendarToolset,這個工具組讓代理之後能去存取 Google 行事曆的 API。第二步,建立一個 ADK 的大型語言模型代理,英文是 LlmAgent,並且幫它設定好幾樣東西:指定它用哪一個 Gemini 模型、給它一個有描述性的名字、再給它一段「指令」。這段指令其實就是寫給代理看的工作說明書,內容大意是告訴它:你是一個能幫使用者管理行事曆的代理,使用者會來問行事曆的狀態、或要求你改動行事曆,你要用提供的工具去跟行事曆 API 互動;如果使用者沒特別指定是哪一本行事曆,就預設用「主要」那一本;呼叫行事曆 API 的時候,時間戳記要用格式正確的 RFC3339 標準格式。這段指令還有一個小巧思:它會動態地把「今天的日期」塞進去,讓代理隨時知道「今天是幾號」,這樣它在處理「明天」「這個週末」這種相對時間時才不會出錯。最後,這個函式把行事曆工具組裡的工具都掛到代理身上,讓它真的有能力去操作行事曆,然後把這個組裝好的代理回傳出去。

第二段程式,是真正的「主程式」,負責把這個代理當成一個網路服務架起來,讓外界可以透過 HTTP 來用它。它做的事情我幫你拆成幾個重點。一開始,它會先檢查驗證設定,確認有沒有設好 API 金鑰、或者有沒有設定要走 Vertex AI;如果兩個都沒有,它就直接報錯停下來,免得後面才出問題。接著,它定義了一項「技能」,這項技能叫「檢查空檔」,英文是 check availability,描述是用使用者的 Google 行事曆去檢查某個時段有沒有空,還附了一個範例問句「我明天早上十點到十一點有空嗎?」。定義完技能之後,它把這項技能包進一張「代理卡」裡,這就呼應到我們前面講的代理卡概念了。這張卡裡寫了代理的名字叫 Calendar Agent、描述、它的網路位址、版本、預設輸入輸出都是文字、能力標明支援串流,然後把剛剛那項「檢查空檔」的技能列進去。你看,前面講的代理卡,在這裡就被實際做出來了。

再往下,主程式呼叫我們第一段講的那個工廠函式,把真正的 ADK 代理建出來,並且從環境變數裡讀取用戶端編號跟密鑰傳進去。然後它設定一個「執行器」,英文是 Runner,負責實際跑這個代理,並且配上幾個「記憶體內」的服務,英文是 in-memory,意思是資料暫時放在記憶體裡,用來管理產物、工作階段也就是 session、以及記憶。接著它包了一個專門處理 A2A 協定的執行器,還準備了一個驗證的回呼函式,英文是 callback,這個回呼是用來處理使用者授權 Google 帳號之後跳回來的那個步驟,授權成功就回一句「驗證成功」。最後,它用一個叫 Starlette 的網頁框架,把 A2A 協定的處理路由跟那個驗證的路由通通組起來,再用一個叫 Uvicorn 的伺服器把整個應用程式跑起來,綁定在指定的主機跟連接埠上。到這裡,這個行事曆代理就正式以一個 HTTP 服務的形式對外公開了,別的代理就能透過 A2A 來找它、用它。

把這兩段程式串起來看,它示範的就是「打造一個 A2A 相容代理」的完整過程:從定義它有哪些能力跟技能、寫好代理卡,到把它真正當成一個網頁服務跑起來。透過運用代理卡跟 ADK,開發者就能做出可互通的 AI 代理,讓它能跟 Google 行事曆這類工具整合,進而組出一個多代理的生態系。

好,我們來把整章的脈絡收一收,我用「是什麼、為什麼、什麼時候用」這三個角度幫你做總結。先講「是什麼」。問題的本質是:單一 AI 代理,尤其是建在不同框架上的那些,常常沒辦法靠自己處理複雜、多面向的大問題,而最核心的痛點,就是它們之間缺乏一種共通的語言或協定。這種彼此孤立的狀態,讓我們很難把多個專精代理組合起來去解決更大的任務;如果沒有一套標準化的做法,要整合這些差異很大的代理,會非常花錢、非常花時間。

再講「為什麼」用 A2A。A2A 就是針對這個問題提出的一套開放、標準化的解法。它是一項以 HTTP 為基礎的協定,促成了互通性,讓不同的 AI 代理可以無縫地協調、委派任務、共享資訊,而且不被它們底層的技術綁住。其中一個核心元件就是代理卡,這份數位身分檔案描述了代理的能力、技能跟通訊端點,讓探索跟互動都變得容易。而且 A2A 定義了多種互動機制,同步的、非同步的都有,能撐起各式各樣的使用案例。透過為代理協作建立一套通用標準,A2A 培育出一個模組化、可擴展的生態系,讓我們能拿來建構複雜的多代理系統。

最後講「什麼時候該用」,也就是經驗法則。當你需要編排兩個或更多 AI 代理之間的協作時,就用 A2A 這個模式,尤其是當這些代理是用不同框架做的,比方說 Google ADK、LangGraph、CrewAI 混在一起的時候。它特別適合那種複雜、模組化的應用:讓專精的代理各自負責工作流程的一部分,例如把資料分析交給某個代理、把報告生成交給另一個代理。還有一種情況也很需要它:當一個代理需要動態地去探索、並運用其他代理的能力來完成任務時,A2A 就不可或缺。

在結束之前,我把這一講最該記住的幾個要點再幫你串一次,當作收尾。第一,Google A2A 協定是一項開放的、以 HTTP 為基礎的標準,讓不同框架建構的 AI 代理可以通訊跟協作。第二,代理卡是代理的數位識別碼,讓別的代理能自動探索它、理解它的能力。第三,A2A 同時提供同步的請求回應互動,以及串流更新,來滿足不同的通訊需求;你可以記得那組對照,一個要完整單一答案、一個要持續分批更新。第四,這個協定支援多輪對話,它有一個叫「需要輸入」input-required 的狀態,讓代理可以在互動過程中回頭跟你要更多資訊,同時還能維持住前面的情境。第五,A2A 鼓勵一種模組化的架構,讓專精的代理可以跑在不同的連接埠、也就是 port 上,各自獨立運作,進而讓整個系統能擴展、能分散。第六,像 Trickle AI 這類工具,可以幫你把 A2A 的通訊視覺化、加以追蹤,讓開發者更容易監控、除錯、最佳化多代理系統。第七,要記得 A2A 跟 MCP 的分工:A2A 是管不同代理之間任務跟工作流程的「高層級」協定,而 MCP 則是給大型語言模型一個跟外部資源對接的標準化介面。

總結來說,A2A 協定確立了一項關鍵的開放標準,用來克服單一 AI 代理天生的孤立性。它提供一套共通的、以 HTTP 為基礎的框架,確保不同平台上做出來的代理,比方說 Google ADK、LangGraph 或 CrewAI,都能無縫協作、互通。它的核心元件代理卡,扮演數位身分,清楚定義了代理的能力,也讓別的代理能動態探索。它的彈性支援同步請求、非同步輪詢、即時串流等各種互動模式,滿足很廣的應用需求。這讓我們能建立模組化、可擴展的架構,把專精的代理組起來,編排出複雜的自動化工作流程。安全性是它的根本,內建了像 mTLS、明確驗證需求這些機制來保護通訊。在跟 MCP 這類標準互補的同時,A2A 獨特的焦點在於代理之間「高層級」的協調與任務委派。再加上各大科技公司的力挺、以及實際實作的可得性,都凸顯了它越來越重要。歸根究柢,A2A 就是培育一個創新、可互通的協作 AI 生態系的基礎支柱。當你未來要把好幾個代理組成一個團隊一起工作時,A2A 就是那套讓它們能彼此對話的共同語言。
