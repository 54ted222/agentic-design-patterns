這一章我們要談的,是讓 AI 代理真正能夠走出腦袋、伸手去碰到真實世界的關鍵能力,也就是「工具使用」這個設計模式。

我先幫你把這一集的路線圖講一遍,讓你心裡有個底。我們會從一個最根本的問題開始:為什麼一個語言模型,光會講話還不夠?接著我會帶你看「工具使用」到底是怎麼運作的,它背後有一套大約六個步驟的流程,我會一步一步講清楚。然後我們會把眼界放寬,看看「函式呼叫」跟更廣義的「工具呼叫」差在哪裡。再來,我會給你六種非常實際的應用場景,從查天氣、查庫存,到寄電子郵件、控制家裡的燈,讓你對「工具」這個詞有具體的想像。之後,我們會走過四個動手實作的範例,分別用三套不同的框架來示範,但放心,我不會逐行唸程式碼,而是用講的讓你聽懂每段程式在做什麼。最後,我會用「是什麼、為什麼、什麼時候該用」這三個角度幫你收斂整章重點。

好,我們先從那個根本問題說起。

在前面幾章,我們討論的代理模式,大多是在「編排語言模型彼此之間的互動」,還有「管理代理內部工作流程裡的資訊怎麼流動」。我這裡先幫你快速回顧一下,那些模式包括:把多個步驟串起來的「串接」,英文叫 Chaining;根據情況選擇不同路徑的「路由」,英文叫 Routing;讓多件事同時進行的「平行化」,英文叫 Parallelization;還有讓代理回頭檢視自己輸出的「反思」,英文叫 Reflection。你會發現,這些模式有一個共通點:它們全部都在代理的「內部」打轉,處理的是模型跟模型之間、或者資訊在流程裡怎麼搬動的問題。

可是,如果一個代理永遠只在自己腦袋裡轉,它就沒辦法真正派上用場。要讓代理能跟真實世界、或是外部系統互動,它就必須具備一種能力,那就是「使用工具」的能力。這裡的「工具」,英文叫 Tool,你可以先把它想成代理手上可以拿來用的外掛功能。

那這個「工具使用」模式,英文叫 Tool Use,它通常是透過一種機制來實作的,這個機制叫「函式呼叫」,英文是 Function Calling。函式呼叫讓代理能夠跟外部的 API、資料庫、各種服務互動,甚至能執行程式碼。我這裡順帶把 API 這個縮寫講清楚,它的全名是 Application Programming Interface,中文叫應用程式介面,你可以把它想成兩個軟體之間溝通用的一個標準窗口。而位在代理核心的那個大型語言模型,英文縮寫是 LLM,全名是 Large Language Model,它可以根據使用者的請求、或是當下任務的狀態,自己決定「要不要用某個外部函式」,以及「該怎麼用」。

接下來,我要把這個流程的步驟一個一個講給你聽。這整套流程大概有六個步驟,你可以想像成一個代理拿到任務之後,從頭到尾會經歷的旅程。

第一步,叫「工具定義」。意思是,你要先把某個外部函式、或某項能力定義出來,然後用語言模型聽得懂的方式去描述它。這份描述要包含幾件事:這個函式是做什麼用的、它叫什麼名字、它接受哪些參數、這些參數又是什麼型別、各自代表什麼意思。你可以把這一步想成,你要先寫一張說明書,告訴模型「我這裡有一個工具,它長這樣,你需要的時候可以叫它」。

第二步,叫「LLM 決策」。模型這時候會同時收到兩樣東西:一個是使用者的請求,另一個是它手上有哪些可用工具的定義。模型會根據它對請求和工具的理解,自己判斷:我需不需要呼叫工具?需要叫一個,還是叫好幾個,才能滿足這個請求?這個判斷完全是模型自己做的,沒有人在旁邊用寫死的規則告訴它。

第三步,叫「函式呼叫生成」。假設模型決定要用某個工具了,它不會直接去執行,而是會生成一段結構化的輸出,通常是一個 JSON 物件。我這邊解釋一下 JSON,它的全名是 JavaScript Object Notation,你可以把它想成一種有固定格式、像填表格一樣的資料寫法,讓機器很容易讀。這段結構化輸出會明確寫出兩件事:要呼叫哪一個工具的名字,以及要傳進去什麼引數。而這些引數,是模型從使用者的請求裡面擷取出來的。舉個例子,使用者問「倫敦的天氣怎麼樣」,模型就會把「倫敦」這個地點抓出來,當成引數填進去。

第四步,叫「工具執行」。這一步的主角不是模型了,而是代理框架、或者說編排層。它會攔截剛剛那段結構化輸出,看懂模型想叫哪個工具,然後拿著模型給的引數,真正去執行那個外部函式。也就是說,真正動手做事的,是這個框架,不是語言模型本身。

第五步,叫「觀察」,或者說「結果」,英文是 Observation。工具執行完之後產生的輸出或結果,會被回傳給代理。你可以把這想成,代理派出去的工具辦完事,回來向它報告結果。

第六步,叫「LLM 處理」。這一步標註為「選用,但很常見」。模型會把工具回傳的結果當成新的情境資訊,拿來擬定要給使用者的最終回應,或者決定流程的下一步該怎麼走。而所謂下一步,可能是再去呼叫另一個工具、可能是回頭做一次反思、也可能就是直接給出最終答案。

把這六步串起來,你就會看到一個完整的循環:定義工具、模型決定要不要用、模型生成呼叫請求、框架真正執行、結果回傳、模型再消化結果產生回應。

那為什麼這個模式這麼基礎、這麼重要?關鍵在於,它打破了語言模型「訓練資料」的侷限。一個語言模型,它的知識其實是凍結在某個時間點的,就是它被訓練的那批資料。透過工具使用,它就能取得最新的資訊、能執行它腦袋裡算不出來的計算、能去碰使用者專屬的資料,甚至能在真實世界裡觸發某個行動。我常常打一個比方:語言模型像一個很博學、但被關在房間裡、而且手機停在去年某一天的人;函式呼叫,就是那座技術橋樑,銜接起模型的「推理能力」跟外面「五花八門的可用功能」之間的鴻溝。

講到這裡,我要幫你把一個觀念再拓寬一點。雖然「函式呼叫」這個詞,很貼切地描述了「去呼叫一個特定、預先寫好的程式碼函式」這件事,但其實思考一個更寬廣的概念會更有意思,那就是「工具呼叫」,英文叫 Tool Calling。為什麼要用更廣的詞?因為代理的能力,可以遠遠超過單純執行一個函式。一個「工具」當然可以是傳統的函式,但它也可以是一個複雜的 API 端點、可以是一個對資料庫發出的查詢請求,甚至可以是「下達給另一個專門代理的一道指令」。

你體會一下這個觀點打開的想像空間:一個主要代理,可能把一個複雜的資料分析任務,委派給一個專門的「分析師代理」去做;或者透過 API 去查詢一個外部的知識庫。當你用「工具呼叫」的角度去想,你才更能完整地看到,代理其實是一個「編排者」,英文叫 Orchestrator。它的潛力在於,可以橫跨一整個由各種數位資源、還有其他智慧實體所組成的生態系,去進行協作。換句話說,工具不一定是一段冷冰冰的程式,它也可以是另一個會思考的夥伴。

那在實作上,有哪些工具可以幫我們做這件事?目前有幾個很主流的框架,像是 LangChain、LangGraph,還有 Google 的「代理開發套件」,英文全名是 Agent Developer Kit,縮寫成 ADK。這些框架,都為「怎麼定義工具」以及「怎麼把工具整合進代理的工作流程」提供了很穩健的支援,而且通常還會直接運用現代語言模型原生就內建的函式呼叫能力,例如 Google 的 Gemini 系列、或是 OpenAI 的模型。在這些框架提供的工作環境裡,你可以先定義好工具,再去設定一個代理,讓這個代理知道有這些工具,並且能夠使用它們。

所以總結這一段,你只要記得一句話:工具使用,是建構「強大、能互動、而且對外部世界有感知能力」的代理時,最根本的那塊基石。

接下來,我要帶你看六種實務上的應用場景。我之所以要一個一個講,是因為「工具」這個詞講起來很抽象,但只要套進真實情境,你馬上就懂了。這六種場景背後有一個共同點:只要代理需要「超越單純生成文字」、進一步去執行某個行動、或檢索某些會動態變化的資訊,工具使用就派得上用場。

第一種,是「從外部來源檢索資訊」。簡單說,就是去拿那些不在語言模型訓練資料裡的即時資訊。我舉一個天氣代理的例子。它的工具,是一個天氣 API,你給它一個地點,它就回傳那個地方當前的天氣狀況。流程是這樣:使用者問「倫敦的天氣如何」,模型一聽,辨識出這需要用到天氣工具,於是它用「倫敦」這個參數去呼叫工具,工具回傳天氣資料,模型再把這些資料整理成一句對使用者友善的回答。

第二種,是「跟資料庫和 API 互動」。也就是對結構化的資料,做查詢、更新、或其他操作。例子是一個電子商務代理。它的工具,是用來檢查商品庫存、查詢訂單狀態、或處理付款的各種 API 呼叫。流程上,使用者問「商品 X 還有沒有庫存」,模型就去呼叫庫存 API,工具回傳庫存數量,模型再告訴使用者現在的庫存狀況。

第三種,是「執行計算與資料分析」。也就是借用外部的計算機、資料分析函式庫、或統計工具。例子是一個金融代理。它的工具,可能包含一個計算機函式、一個股市資料 API、還有一個試算表工具。我用一個比較複雜的問題來示範:使用者問「蘋果公司目前的股價是多少?如果我用每股 150 美元買進 100 股,潛在的獲利會是多少?」這時候模型會分兩步走,先呼叫股票 API 拿到當前股價,再呼叫計算機工具算出獲利,最後整理成一段回應。你注意到了嗎,這裡模型一口氣用到了兩個工具,而且是有先後順序的。

第四種,是「發送通訊」。也就是寄電子郵件、發訊息,或對某個外部通訊服務發出 API 呼叫。例子是一個個人助理代理。它的工具,是一個能寄電子郵件的 API。流程是,使用者說「寄一封關於明天會議的電子郵件給 John」,模型就會從這句話裡面,把收件人、主旨、跟內文這幾個資訊擷取出來,然後拿去呼叫電子郵件工具。

第五種,是「執行程式碼」。也就是在一個安全的環境裡,執行某段程式碼片段來完成任務。例子是一個程式設計助理代理。它的工具,是一個「程式碼直譯器」,英文叫 Code Interpreter。流程是,使用者貼上一段 Python 程式碼,問「這段程式碼是做什麼的」,模型就用這個直譯器工具,實際把程式跑起來,再分析它的輸出。

第六種,是「控制其他系統或裝置」。也就是去跟智慧家庭裝置、物聯網平台、或其他連網系統互動。我這邊解釋一下物聯網這個詞,它的英文縮寫是 IoT,全名是 Internet of Things,指的是把各種日常裝置都連上網路、讓它們能被遠端控制的概念。例子是一個智慧家庭代理。它的工具,是一個能控制智慧燈具的 API。流程很直覺,使用者說「關掉客廳的燈」,模型就用「關燈」這個指令,加上「客廳」這個目標裝置,去呼叫智慧家庭工具。

把這六種場景放在一起看,你就會懂這一章真正想說的那句話:工具使用,正是把一個語言模型,從一個「文字生成器」,轉變成一個「能在數位或實體世界裡感知、推理、並且採取行動」的代理的關鍵。原書在這裡放了一張圖,畫的是代理使用各種工具的範例,而那張圖想傳達的訊息,其實就是我剛剛講的這六種場景:代理一隻手連著天氣、股市、資料庫,另一隻手連著郵件、程式直譯器、跟家裡的燈。

好,觀念講完了,接下來我們進入動手實作的部分。這一章一共給了四個範例,用到三套框架。我要再強調一次,我不會逐行唸語法,我會用講的,讓你在腦中想像每段程式在做什麼、輸入是什麼、輸出是什麼、為什麼要這樣設計。

我們先看第一個範例,用的是 LangChain 框架。

在 LangChain 裡實作工具使用,基本上是一個兩階段的過程。第一階段,先定義一個或多個工具,通常做法是把一個既有的 Python 函式包裝起來。第二階段,把這些工具「綁定」到一個語言模型上。一旦綁定之後,模型就獲得一種新能力:當它判斷「我需要呼叫外部函式才能回答這個問題」時,它就會生成一個結構化的工具使用請求。

這個範例做的事情是這樣:它先定義了一個簡單的函式,名字叫「搜尋資訊」,用來模擬一個資訊檢索工具。我說「模擬」,是因為它其實沒有真的連上網路,而是內部準備了一個小小的對照表。這張表裡面寫死了幾組問答,比方說,你問「倫敦的天氣」,它就回答「倫敦現在多雲、氣溫攝氏 15 度」;你問「法國的首都」,它就回答「巴黎」;你問「地球人口」,它就回答「大約八十億人」;你問「最高的山」,它就回答「聖母峰」。如果你問的東西不在這張表裡,它就回傳一個預設的回應,大意是「沒找到特定資訊,不過這個主題看起來挺有意思」。這裡有個很關鍵的設計細節:在這個函式的描述文字裡,寫清楚了「這個工具是用來提供特定主題的事實性資訊」,還舉了像「法國的首都」、「倫敦的天氣」這種例子。這段描述非常重要,因為這就是模型用來判斷「該不該叫這個工具」的依據。換句話說,工具寫得再好,如果描述含糊,模型也不知道什麼時候該用它。

定義好工具之後,程式接著初始化了一個語言模型,用的是 Google 的 Gemini,而且特別確認這個模型具備工具呼叫的能力。然後它設計了一個提示範本,用來引導代理怎麼互動。這個範本裡有一個特別的位置,叫做「代理草稿區」,英文是 agent scratchpad。你可以把它想成代理的「計算紙」,代理在思考過程中產生的那些內部步驟,就暫存在這裡。接著,程式用一個叫「建立工具呼叫代理」的函式,把三樣東西組合在一起:語言模型、工具、還有提示。組好之後,又設定了一個「代理執行器」,英文是 AgentExecutor,它是真正在執行階段負責「呼叫代理、並執行代理所選工具」的那個運作元件。

最後,這個範例很有意思的一點是,它一次丟了三個問題,而且是讓它們平行執行的。這三個問題分別是:「法國的首都在哪裡」、「倫敦的天氣如何」、還有「跟我說一些關於狗的事」。前兩個,會打中工具裡寫死的答案;第三個關於狗的問題,因為表裡面沒有,就會觸發那個預設回應。你看,這三個問題的設計是有用意的,它同時測試了「工具命中特定答案」跟「工具回傳預設答案」這兩種情況。整段程式跑起來,你會在畫面上看到工具被呼叫的紀錄、工具回傳的結果、以及代理最終整理出來的回答。喔對了,這個範例在真正開始設定跟執行之前,還很貼心地做了一個檢查,確認語言模型有沒有成功初始化,沒成功的話就不會硬跑下去。

接下來是第二個範例,換成 CrewAI 這個框架。

CrewAI 的味道跟 LangChain 不太一樣,它更強調「角色」跟「團隊協作」的概念。這個範例設定了一個簡單情境:有一個代理,被配備了一個查資訊的工具,然後我們用它去抓一檔模擬的股價。

它定義的工具叫「股價查詢工具」,英文是 get stock price。你給它一個股票代碼,它就回傳那檔股票最新的模擬股價。一樣是模擬的,內部有一張小表,裡面寫著:蘋果,也就是 AAPL,股價是 178.15;Google,也就是 GOOGL,是 1750.30;微軟,也就是 MSFT,是 425.50。這個工具的設計有一個我特別想跟你強調的巧思:如果你查的代碼有效,它就回傳一個乾淨的浮點數,也就是一個帶小數點的數字;但如果你查的代碼不存在,它不會回傳一個字串說「找不到」,而是直接拋出一個標準的 Python 錯誤,叫 ValueError,也就是「數值錯誤」。

為什麼這樣設計比較好?原書在這裡其實是在示範一個最佳實務。讓工具回傳「原始、乾淨的資料」,或者在出錯時「拋出一個標準錯誤」,這會讓工具更容易被重複使用,而且會「逼著」代理去妥善處理各種結果。你想想看,如果工具出錯時只回一句模糊的字串,代理可能會把那句話當成正常答案唸出來;但如果它拋出一個明確的錯誤,代理就知道「這是出狀況了」,然後決定接下來該怎麼辦。

定義好工具之後,程式建立了一個 CrewAI 代理,給了它一個角色,叫「資深金融分析師」。它還幫這個代理設定了「目標」跟「背景故事」。目標是「用提供的工具分析股票資料,並回報關鍵價格」;背景故事則是「你是一位經驗豐富的金融分析師,擅長運用各種資料來源找股票資訊,而且你會給出清楚、直接的答案」。這種寫法是 CrewAI 的特色,它讓你用很像在設定一個角色人設的方式,去塑造代理的行為。當然,這個代理也被裝上了剛剛那個股價查詢工具。

接著程式定義了一個任務,叫「分析蘋果任務」。任務描述寫得很具體:「蘋果公司,代碼 AAPL,目前的模擬股價是多少?請用股價查詢工具去查。如果找不到這個代碼,你必須回報你沒辦法取得股價。」你注意到了嗎,任務描述裡同時交代了「成功該怎麼做」跟「失敗該怎麼回應」這兩種情況,這就呼應了我們剛剛說的,工具會拋錯,所以任務也要教代理怎麼面對錯誤。任務還定義了一個「期望輸出」,要求代理用一句清楚的話陳述股價,例如「AAPL 的模擬股價是 178.15 美元」,而如果查不到,就要清楚說明這一點。

再來,程式把代理跟任務組成一個團隊,在 CrewAI 裡這個團隊就叫 Crew。然後在標準的 Python 主程式區塊裡,用一個叫 kickoff 的方法去啟動執行。啟動之前,它會先檢查 OpenAI 的 API 金鑰這個環境變數有沒有設好,沒設好就先提醒你,避免跑到一半才出錯。整段程式還開了詳細日誌,英文叫 verbose,這樣你就能在執行過程中,清楚看到代理的每一步動作跟工具呼叫。它也提醒,正式環境裡建議把詳細日誌關掉,而且金鑰的管理要用更安全的方式,例如密鑰管理工具。簡單講,這個範例的核心,就是示範「怎麼定義工具、定義代理、定義任務」,然後在 CrewAI 裡組出一個協作式的工作流程。

接下來是第三組範例,主角是 Google 的代理開發套件,也就是 ADK。ADK 有一個很大的特色,是它內建了一整套「原生整合」的工具,你可以直接把它們納進代理的能力裡,不用自己從頭包。這一段會給你看三個 ADK 內建工具的例子。

第一個內建工具,是「Google 搜尋」。這個工具,就是 Google 搜尋引擎的一個直接介面,它讓代理具備上網搜尋、檢索外部資訊的能力。範例做的事情是:建立一個代理,名字叫「基本搜尋代理」,給它一段描述說「這是一個用 Google 搜尋來回答問題的代理」,再給它一段指示說「我可以透過搜尋網路來回答你的問題,儘管問吧」,然後把 Google 搜尋這個預建工具裝上去。接著它設定了工作階段服務跟執行器,我這邊解釋一下,「工作階段」英文叫 Session,你可以把它想成一次對話的紀錄與上下文,而範例用的是一個「記憶體內」的工作階段服務,意思是這些紀錄就暫存在記憶體裡。範例還定義了一個輔助函式,專門用來「把問題丟給代理、再處理它的回應」。它的做法是,把使用者的問題包裝成一則角色是「使用者」的訊息,送進執行器去跑,執行器會回傳一連串的事件,代表代理一路上的動作跟回應。程式會走過這些事件,找出哪一個是「最終回應」,把它的文字抓出來印到畫面上。最後,它實際用一個問題「最新的 AI 新聞有哪些」去呼叫這個代理,展示它真的會上網搜尋再回答。

第二個內建工具,是「程式碼執行」。ADK 內建了一個能「動態執行程式碼」的環境,這個工具會給代理一個「沙箱化」的 Python 直譯器。我解釋一下「沙箱」這個詞,英文是 sandboxed,它的意思是,程式是在一個被隔離、受保護的環境裡跑的,就算程式碼有問題,也不會去破壞到外面的系統,就像小孩在沙坑裡玩,弄得再亂也只是在沙坑裡。有了這個工具,模型就能自己「寫程式、再執行程式」,去做運算、操作資料、跑一些有步驟的腳本。為什麼這很重要?因為有些問題需要的是「確定性的邏輯」跟「精確的計算」,而這正是單靠語言模型那種「機率性地生成文字」做不到的。講白一點,你不會想讓一個靠猜下一個字的模型去算十的階乘,你會想讓它寫一段程式、實際算出來。

這個範例就定義了一個語言模型代理,叫「計算機代理」,並且特別指示它:當收到一個數學運算式時,要「寫並執行 Python 程式碼」來算結果,而且最後只回傳純文字的數值,不要包成程式碼區塊。它配備的就是那個內建的程式碼執行工具。範例的主要邏輯,是一個非同步函式,它把使用者的問題送給代理的執行器,然後處理跑出來的一連串事件。在處理過程中,它會很細心地把幾種東西區分開來:一種是「代理生成的那段 Python 程式碼」,一種是「程式碼執行之後的結果」,還有一種是「最終那個含有數值答案的事件」。它會把生成的程式碼跟執行結果都印出來,方便你除錯,然後再抓出最終答案。最後,主函式用兩道題去考這個代理:一題是算「五加七,再乘以三」,另一題是問「十的階乘是多少」,藉此展示它真的有能力執行計算。

第三個內建工具,是「企業搜尋」。這個範例用的是一個叫 VSearchAgent 的代理,它的設計,是去搜尋一個指定的「Vertex AI Search 資料儲存庫」來回答問題。我先把名詞講清楚:Vertex AI 是 Google Cloud 上的一套 AI 平台;而「資料儲存庫」,英文叫 datastore,你可以把它想成一個專門放你自己企業資料的倉庫,比方說公司內部的文件。範例建立了一個代理,叫「第二季策略搜尋代理」,給它描述、指定模型、還有那個資料儲存庫的識別碼。這個識別碼是從環境變數讀進來的。然後一樣設定執行器跟記憶體內的工作階段服務來管理對話歷史。

這個企業搜尋範例,有兩個值得特別講的地方。第一,它會「串流」回應,英文叫 stream。串流的意思是,代理的回答不是等全部生成完才一次吐出來,而是一個字、一個字,隨著它陸續產生就即時送到你畫面上,就像你看到對方正在打字那種感覺。第二,它在最終回應裡,會印出「來源歸屬」,英文叫 source attribution。這代表代理告訴你:我這個答案,是根據資料儲存庫裡哪些來源得出來的。這在企業場景裡非常重要,因為你需要知道答案的依據是什麼,而不是讓代理憑空講。範例還做了很完整的錯誤處理,如果資料儲存庫的識別碼錯了、或是權限不夠,它會給你一段有用的提示。它用的兩個範例問題分別是:「摘要說明第二季策略文件的主要重點」,還有「文件裡針對實驗室 X 提到了哪些安全程序」。整體來說,這段程式提供了一個基本框架,讓你能打造一個對話式 AI 應用,核心目的就是「從一個特定的資料儲存庫裡,檢索並綜整資訊,來回答使用者的問題」。

講完三個內建工具,這一段最後還補充了一個重要的觀念,叫「Vertex 擴充功能」,英文是 Vertex Extensions。它是一個結構化的 API 包裝器,讓模型能連接外部 API,去做即時的資料處理跟行動執行。它主打的是企業級的安全性、資料隱私、跟效能保證,可以用來生成並執行程式碼、查詢網站、分析私有資料儲存庫裡的資訊。Google 為常見的場景準備了預建的擴充功能,例如程式碼直譯器跟 Vertex AI Search,你也可以自己建客製的。

這裡我要幫你劃一個重點,因為它很容易混淆:擴充功能跟函式呼叫,最關鍵的差別,在於「誰來執行」。Vertex AI 的擴充功能,是由平台「自動」幫你執行的;而函式呼叫,是需要由使用者、或者說客戶端「手動」去執行的。還記得我們最前面講的那六個步驟嗎?在函式呼叫的世界裡,模型只負責「生成那個呼叫請求」,真正動手執行的是你自己的編排層。但擴充功能不一樣,平台會直接替你把那一步做掉。這個「自動執行 vs 手動執行」的差異,就是這兩者的分水嶺。

好,所有範例都走過一遍了,接下來我用三個角度,幫你把整章收斂起來。

第一個角度,是「這是什麼」。大型語言模型,本質上是很強的文字生成器,但它跟外部世界其實是「脫節」的。它的知識是靜態的,只限於它被訓練的那批資料;而且它本身沒有能力去執行行動、或檢索即時資訊。這種天生的侷限,讓它沒辦法獨力完成那些需要跟外部 API、資料庫、服務互動的任務。少了通往這些外部系統的橋樑,它在解決真實世界問題時,效用會被嚴重限制。

第二個角度,是「為什麼要用工具使用模式」。它通常透過函式呼叫來實作,為前面那個侷限提供了一套標準化的解法。它的運作方式,是用模型聽得懂的方式,向它描述有哪些可用的外部函式,也就是工具。然後,代理式的模型就能根據使用者的請求,判斷需不需要某個工具,並生成一個結構化的資料物件,像 JSON,寫明要呼叫哪個函式、要傳什麼引數。接著由一個編排層去執行這個呼叫、拿回結果、再餵回給模型。這樣一來,模型就能把最新的外部資訊、或某個行動的結果,整合進它最終的回答裡,等於有效地賦予了它「行動」的能力。

第三個角度,是「什麼時候該用」,也就是經驗法則。一句話:每當一個代理需要跳脫模型自己的內部知識、去跟外部世界互動時,就用工具使用模式。具體來說,如果任務需要即時資料,例如查天氣、查股價;或需要取用私有、專屬的資訊,例如查公司的資料庫;或需要做精確計算、執行程式碼;又或者需要在別的系統裡觸發行動,例如寄郵件、控制智慧裝置,那麼這個模式就是不可或缺的。原書在這裡也放了一張圖,叫「工具使用設計模式」,那張圖想傳達的,其實就是我們最開頭講的那個循環:使用者的請求進來,模型決策、生成呼叫,工具被執行,結果回流,模型再產生最終回應。

最後,我把這一章最該記住的幾顆釘子幫你敲一遍。工具使用,也就是函式呼叫,讓代理能跟外部系統互動、取用會變動的動態資訊。它的做法,是定義工具,並且替工具附上「模型能理解的清楚描述跟參數」,這一點我前面特別強調過,描述寫不好,模型就不知道何時該用。決定「什麼時候用工具、並生成結構化呼叫」的,是語言模型;而真正「執行工具、把結果回傳給模型」的,是代理框架。對於要建構「能執行真實世界行動、能提供最新資訊」的代理來說,這個模式不可或缺。在框架層面,LangChain 用一個叫 tool 的「裝飾器」來簡化工具定義,並提供「建立工具呼叫代理」跟「代理執行器」這兩樣東西來組裝代理;這裡的「裝飾器」英文叫 decorator,你可以把它想成一個貼在函式上方的小標籤,貼上去之後,這個函式就被標記成「這是一個工具」。而 Google ADK 則內建了很多現成又好用的工具,包括 Google 搜尋、程式碼執行、跟 Vertex AI Search。

所以,如果要我用一句話收束整章,我會這樣說:工具使用模式,是一項關鍵的架構原則,它把大型語言模型的能力範圍,從「與生俱來的文字生成」往外擴展。透過賦予模型「跟外部軟體與資料來源對接」的能力,代理就能執行行動、進行運算、從別的系統檢索資訊。而像 LangChain、Google ADK、CrewAI 這些框架,提供了好用的抽象跟元件,幫我們管理兩件最麻煩的事:一件是「怎麼把工具的規格揭露給模型」,另一件是「怎麼解析模型回頭發出的工具使用請求」。把這兩件事交給框架,我們就能把心思放在更重要的地方,去打造那些真正能跟外部世界互動、並在其中採取行動的精密代理系統。這,就是工具使用為什麼是代理設計裡那塊不可或缺的基石。
