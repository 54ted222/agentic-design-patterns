這一章我們要談的,是一個聽起來很工程、但其實非常貼近現實的主題,叫做資源感知最佳化,英文是 Resource-Aware Optimization。簡單說,它要解決的問題是:當你的智慧代理在做事的時候,要怎麼一邊把事情做好,一邊精打細算地花用手上的運算力、時間,還有錢。這一段路我會這樣帶你走:先說清楚這個概念到底是什麼、跟規劃有什麼不一樣;接著看幾個生活化的例子和實務上的使用案例;然後我們會走過三組動手實作,分別用 Google 的工具、用 OpenAI、再用一個叫 OpenRouter 的服務;最後,我會把這個模式延伸出去,介紹一整個光譜的進階技巧,並且把整章的重點再幫你串一次。

我們先從最核心的定義說起。所謂智慧代理,英文是 intelligent agent,你可以把它想成一個能自己感知、決策、然後動手做事的程式。資源感知最佳化,就是讓這個代理在運作的過程中,能夠動態地監控和管理三種東西:運算資源、時間,還有財務成本。這裡有個關鍵,要把它跟「規劃」這件事分清楚。規劃,英文是 planning,它關心的主要是動作的「順序」,也就是先做什麼、再做什麼、最後做什麼。但資源感知最佳化關心的不是順序,而是「執行的方式」。它要代理針對每個動作怎麼執行做決策,目標是要嘛在指定的資源預算之內把任務完成,要嘛盡可能把效率拉到最高。

講得更具體一點,這牽涉到一連串的取捨。比方說,要選一個比較準、但比較貴的模型,還是選一個比較快、比較便宜的模型?又比方說,要不要多花一點運算力,去換一個更精煉的回答,還是乾脆回傳一個快一點、但沒那麼詳盡的答案就好?這些都是資源感知最佳化要當場拿捏的判斷。

我給你一個很好懂的例子。設想有一個代理,被派去幫一位財務分析師分析一個很大的資料集。如果這位分析師現在馬上就要一份初步報告,那代理可能就會挑一個比較快、比較平價的模型,趕快把關鍵趨勢摘要出來給他。可是,如果情況反過來,這位分析師是要為一項很關鍵的投資決策做高度準確的預測,而且他手上有比較充裕的預算、也有比較多的時間,那這時候代理就會配置更多資源,改用一個更強大、雖然比較慢、但更精準的預測模型。你看,同樣一個任務,因為當下的需求、預算、時間不一樣,代理選的執行方式就完全不同,這就是「資源感知」的精神。

在這個類別裡,還有一個很重要的策略,叫做備援機制,英文是 fallback mechanism。它的角色,是一道安全防線。想像一下,你最想用的那個模型,因為流量太大被塞爆了,也就是所謂的過載,英文叫 overloaded,或者因為你呼叫得太頻繁被服務方限流了,英文叫 throttled,總之它暫時用不了。這時候,系統不會就這樣整個垮掉、直接報錯,而是會自動切換到一個預設的、或者比較平價的模型,先把服務維持住。這種「就算降級也要繼續服務、而不是徹底失敗」的做法,有個專有名詞叫優雅降級,英文是 graceful degradation。你可以把它想成,電梯壞了還有樓梯,雖然慢一點累一點,但至少你還能上樓,而不是被困在原地。

理解了核心概念,我們來看看實務上這個模式會出現在哪些地方,大概有六種常見的使用案例。第一種是「成本最佳化的 LLM 使用」。LLM 指的是大型語言模型,英文是 Large Language Model。代理會看著預算限制來決定:這個任務很複雜,就動用龐大、昂貴的模型;那個查詢很簡單,就用較小、較平價的模型應付就好。第二種是「對延遲敏感的操作」。在那種講求即時的系統裡,代理會刻意選一條比較快、但可能比較不完整的推理路徑,先求及時回應。第三種是「能源效率」。如果代理是部署在邊緣裝置上,邊緣裝置英文叫 edge device,指的就是那些放在使用者身邊、靠近資料源頭的小型設備,像是手機、感測器這類電力受限的環境,那就要最佳化處理流程來省電池。第四種是「服務可靠性的備援」,這就是我們剛剛說的,主要選擇掛了就自動切到備用模型,確保服務連續、優雅降級。第五種是「資料用量管理」,代理會選擇去拿一份摘要過的資料,而不是把整個完整資料集都下載下來,藉此節省頻寬和儲存空間。第六種是「自適應的任務分配」。在多代理系統裡,也就是有好幾個代理一起合作的系統,英文叫 multi-agent,每個代理會看自己現在的運算負載多重、還有多少時間可用,然後自己決定要不要承接任務。

接下來,我們進入第一組動手實作,看看這套想法怎麼落地。

最基本的概念是這樣:一個負責回答使用者問題的智慧系統,可以先評估每一個問題的難度。碰到簡單的查詢,它就用一個具成本效益的語言模型,書裡舉的例子是 Gemini Flash,你可以把 Flash 理解成「輕量、快、便宜」的那一款。碰到複雜的詢問,它才考慮動用更強大、但也更昂貴的模型,例如 Gemini Pro,Pro 就是「重量級、聰明、但貴」的那一款。而要不要出動 Pro,除了看難度,也要看資源夠不夠,特別是預算和時間的限制。整個系統就是這樣動態地、看情況地去選最適合的模型。

書裡用了一個很好懂的比喻來說明,就是一個旅遊規劃器。這個規劃器是用「階層式代理」蓋起來的,階層式代理英文叫 hierarchical agent,意思是代理之間有高低層級之分,像一個有主管、有部屬的小團隊。在這個團隊裡,高層級的規劃工作——也就是要去理解使用者那種很複雜的請求、把它拆解成一個多步驟的行程、還要做出合乎邏輯的決策——這些燒腦的事,就交給一個精密、強大的模型,例如 Gemini Pro 來處理。這個角色,書裡叫它「規劃器」代理,英文是 planner,因為它需要對情境有很深的理解,還要有推理的能力,就像旅行團裡那個經驗豐富、負責安排整個行程骨架的領隊。

但是,一旦這個行程骨架定下來了,計畫裡頭那些一個一個的小任務,像是去查機票價格、去確認飯店有沒有空房、去找餐廳評論,這些事情本質上就是簡單、重複的網路查詢。這類動作在技術上叫做工具函式呼叫,英文是 tool function call,意思是代理去呼叫一個外部工具來幫它辦事。這些瑣碎的小事,完全可以交給一個比較快、比較平價的模型,例如 Gemini Flash 來執行。你想想看就很合理:那些直截了當的網路搜尋,用便宜的模型就夠了;而錯綜複雜的規劃階段,才需要動用更先進模型的更高智慧,這樣才能確保整個旅遊計畫是連貫、合理的。這就是用對的人做對的事,把貴的智慧留給真正需要的地方。

那這種做法在工具上怎麼支援呢?這裡提到 Google 的 ADK。ADK 是 Agent Development Kit 的縮寫,中文可以叫代理開發套件,它是 Google 提供的一套蓋代理的工具。ADK 透過它的多代理架構,讓你的應用程式可以做到模組化、又好擴展,意思就是不同的代理可以各管各的專門任務,彼此分工。它還有一個特點叫模型彈性,英文是 model flexibility,讓你可以直接用各種 Gemini 模型,包括剛剛說的 Pro 跟 Flash,或者透過一個叫 LiteLLM 的整合層去接其他家的模型。LiteLLM 你可以理解成一個轉接頭,讓你用同一套寫法去呼叫不同廠牌的模型。除此之外,ADK 還有協調能力,英文是 orchestration,它支援那種動態的、由 LLM 來驅動的路由,英文是 routing,路由的意思就是「把請求導向到該去的地方」,靠這個就能做出自適應的行為。最後,ADK 還內建了評估功能,讓你可以很有系統地去量測代理表現得好不好,再拿這些結果回頭來精煉系統。

接著,書裡定義了兩個代理。這兩個代理的設定幾乎一模一樣,差別只在於用的模型不同、成本也不同。第一個叫 Gemini Pro 代理,它掛的是比較貴的 Gemini Pro 模型,它的自我介紹寫的是「一個能力高強、可以處理複雜查詢的代理」,給它的指令是「你是一位擅長解決複雜問題的專家助理」。第二個叫 Gemini Flash 代理,它掛的是比較便宜的 Gemini Flash 模型,自我介紹寫的是「一個快速、又高效率、可以處理簡單查詢的代理」,指令則是「你是一位處理直截了當問題的快速助理」。你聽得出來,這兩個代理本身的「骨架」是相同的,只是一個被設定成走精密路線、一個走快速路線,後面就靠別人來決定該找誰。

那「別人」是誰呢?就是路由器代理,英文是 Router Agent。它的工作,就是決定一個進來的查詢,該交給 Pro 還是 Flash。最陽春的做法,是看一個很簡單的指標,例如查詢的長度:比較短的查詢,就丟給比較便宜的模型;比較長的查詢,就丟給能力比較強的模型。但這只是入門款。一個更精密的路由器代理,可以動用一個 LLM 或者機器學習模型,機器學習英文是 machine learning,縮寫 ML,讓它去分析查詢的細微差異跟複雜度。這種「用 LLM 來當路由器」的設計,能判斷出到底哪一個下游的語言模型最適合接這個案子。舉個例子,如果一個查詢只是要做事實性回憶,也就是問一個有標準答案的事實,英文叫 factual recall,那就路由到 Flash 模型;但如果是一個需要深度分析的複雜查詢,那就路由到 Pro 模型。

書裡也給了一段路由器的程式碼,我用聽得懂的方式幫你描述它在做什麼。這個路由器代理叫 QueryRouter,它收到使用者的查詢之後,會先做一件很簡單的事:把這句話的字數數出來。然後它設了一個門檻,書裡用的例子是二十個字。如果這個查詢的字數少於二十,它就判定這是個短查詢,把它交給 Gemini Flash 代理去處理,並且印出一行訊息說「因為查詢很短,所以路由到 Flash」。反過來,如果字數達到或超過二十,它就判定這是個長查詢、比較複雜,於是交給 Gemini Pro 代理,並印出「路由到 Pro」。處理完之後,它會把對應代理產出的回應包裝起來、回傳出去。所以整段程式的設計意圖很單純:用「字數」這個最便宜、最快的指標,當作複雜度的粗略代理指標,藉此把簡單的問題分流給便宜模型,省下成本。

當然,字數這種判斷是很粗糙的,所以還有兩個最佳化的技巧可以讓 LLM 路由器變得更聰明。第一個叫提示調校,英文是 prompt tuning,意思是你去精心設計、打磨給路由器那個 LLM 看的提示詞,引導它做出更好的路由判斷。第二個叫微調,英文是 fine-tuning,做法是準備一份資料集,裡頭裝的是「各種查詢,以及它們各自最該配哪個模型」的對照範例,拿這份資料去訓練、調整路由器 LLM,讓它的準確度跟效率都往上提。透過這種動態路由的能力,我們就能在「回應品質」跟「成本效益」之間,找到一個平衡點。

光是把問題分流出去還不夠,系統最好還能知道自己分得對不對、答得好不好,這就要靠下一個角色:評論者代理,英文是 Critique Agent。它的工作,是去評估語言模型給出來的回應,然後提供回饋,而且這個回饋是多功能的。第一個功能是自我修正,英文是 self-correction:評論者會去揪出回答裡的錯誤或前後矛盾的地方,逼那個負責回答的代理去精煉自己的輸出,把品質提上來。第二個功能是效能監控,英文是 performance monitoring:它會很有系統地評估回應,追蹤像是準確度、相關性這些指標,拿來做最佳化。第三個功能,是它的回饋可以變成訊號,餵給強化學習,英文是 reinforcement learning,或拿去做微調。舉例來說,如果它一直觀察到 Flash 模型在某類問題上回得很爛,那就可以回頭去修正路由器代理的邏輯,以後這類問題別再丟給 Flash。

這裡有個很細、但很重要的觀念。評論者代理它本身並不直接管預算,它不會去算錢。可是它透過揪出那些「次佳的路由選擇」,間接地幫上了預算管理的忙。什麼叫次佳的路由選擇?就是分錯了:比方說把一個簡單查詢硬是導去了昂貴的 Pro 模型,白白浪費錢;或者把一個複雜查詢丟給了 Flash,結果答得很差。評論者把這些分錯的情況辨識出來,這些資訊就能引導我們去調整,改善資源配置、把錢省下來。所以它是用「品質把關」的方式,間接做到了「成本把關」。

另外補充一點,評論者代理可以有兩種設定方式。一種是它只審閱回答代理生出來的那段文字。另一種更全面,是它同時看「原始的查詢」跟「生成的文字」,這樣它才能判斷這個回答到底有沒有真正切中使用者最一開始問的問題。

那評論者代理是靠什麼運作的呢?靠一段預先寫好的系統提示,英文是 system prompt,你可以把它想成是給這個代理的「工作守則」。書裡給了一個很完整的評論者系統提示範例,我幫你講它的精神。這段守則告訴評論者:你是品質保證的角色,你的主要職責是仔細地審查、甚至挑戰研究員代理給出來的資訊,確保它準確、完整、而且不偏頗。它的任務包括:評估研究發現在事實正確性、完整性、還有有沒有潛在偏向這幾方面的表現;找出有沒有漏掉的資料、或者推理裡前後不一致的地方;提出一些關鍵問題,去精煉或拓展目前的理解;為了強化內容或探索不同角度,提供建設性的建議;最後確認產出是全面又平衡的。而這段守則裡反覆強調的一句話是:所有的批評都必須是建設性的,你的目標是強化這份研究,而不是把它否定掉。所以一個好的評論者提示,要清楚地確立它「評估者」的身分,指明哪些地方需要批判性地聚焦,強調要給建設性回饋而不是單純唱反調,還要鼓勵它同時指出優點跟缺點,並且引導它怎麼把回饋有條理地整理、呈現出來。

好,我們換一個技術棧,進入第二組動手實作,這次是用 OpenAI 的程式碼範例。這個系統一樣是用資源感知最佳化的策略,來有效率地處理使用者查詢,但它的分類方式更細,把每個查詢分成三種類別。為什麼要先分類?就是為了找出最適當、最划算的處理路徑,避免在簡單請求上浪費運算資源,同時又確保複雜查詢能得到該有的關注。這三種類別是這樣:第一種叫 simple,適用於可以直接回答、不需要複雜推理、也不需要外部資料的直截了當問題。第二種叫 reasoning,也就是推理類,適用於那種需要邏輯推演、或者多步驟思考的查詢,這種會被路由到更強大的模型。第三種叫 internet search,也就是網路搜尋類,適用於需要當前最新資訊的問題,碰到這種,系統會自動去觸發一次 Google 搜尋,好提供最新的答案。

我用講的,帶你走一遍這段 OpenAI 程式碼到底在做什麼,你不用看到程式碼也能在腦中想像它的流程。首先,它會從一個環境設定檔裡,把需要的金鑰載進來,包括 OpenAI 的金鑰、還有 Google 自訂搜尋的金鑰跟識別碼;如果哪一把金鑰沒設好,它會直接報錯提醒你。接著是第一步,分類。它有一個專門的分類函式,會去呼叫一個 OpenAI 模型,在這個例子裡用的是 GPT-4o,並且給它一段很嚴格的指令,要求它分析使用者的提示,然後「只能」回傳剛剛那三種類別之一,而且要用一個固定格式的 JSON 結構回覆,裡頭就一個欄位寫著分類結果。第二步,是 Google 搜尋函式,它會帶著查詢去打 Google 自訂搜尋的服務,把回來的結果整理成標題、摘要、連結這幾項;萬一連線出錯,它會回傳一個錯誤訊息而不是讓程式崩掉。第三步,是生成回應的函式,這一步是整個資源感知的核心,因為它會「根據分類結果去選不同的模型」:如果是 simple,就用最便宜的 GPT-4o-mini;如果是 reasoning,就用一個擅長推理的 o4-mini;如果是 internet search,就用 GPT-4o,而且還會把剛剛搜尋到的結果當作背景資料,塞進提示裡一起給模型參考。第四步,是一個把前面全部串起來的總指揮函式,叫 handle_prompt,它的流程是:先呼叫分類、如果分類結果是網路搜尋就去跑搜尋、最後再呼叫生成回應,然後把分類結果、實際用了哪個模型、還有生成的答案,一起回傳出來。書裡示範的測試問題是「澳洲的首都是哪裡」,這種就是會被歸到 simple、用便宜模型回答的典型例子;另外註解掉的範例還有「量子運算對密碼學的影響」這種偏推理的,跟「2026 年澳洲網球公開賽什麼時候開始」這種需要查當前資訊的。整段程式的設計意圖,就是把不同性質的查詢,各自導向最划算的那條路。

接著是第三組動手實作,主角是 OpenRouter。OpenRouter 是一個很有意思的服務,它透過單一一個 API 端點,端點英文是 endpoint,就是一個你去呼叫的網址入口,幫你統一接上數百種 AI 模型。它的賣點就是內建了自動化的故障轉移、跟成本最佳化功能,而且可以很輕鬆地用你慣用的 SDK 或框架整合進去。故障轉移英文是 failover,意思跟我們前面講的備援很像,就是這個壞了自動換下一個。

書裡的程式碼很短,我幫你描述一下:它就是用一個發送網路請求的函式庫,對 OpenRouter 的聊天補全端點,聊天補全英文是 chat completion,送出一個請求,請求裡帶著使用者的訊息,例子裡問的是「生命的意義是什麼」,然後在請求的標頭裡放進 API 金鑰做授權,另外還可以選填一些網站資訊。它的目標,就是從你指定的那個語言模型,這裡指定的是 OpenAI 的 GPT-4o,拿回一個回應。

OpenRouter 真正值得學的,是它提供了兩種不同的路由方法論,來決定一個請求該用哪個運算模型來處理。第一種叫自動化模型選擇,英文是 Automated Model Selection。你只要在請求裡把模型指定成一個特殊的值,意思就是「你幫我自動挑」,它就會從一組精選的可用模型裡,根據你這個提示的具體內容,挑一個最佳化的模型來處理。而最後到底是哪個模型接手的,它會寫在回應的中介資料裡告訴你,中介資料英文是 metadata,就是「關於這次回應的附帶說明資訊」。第二種叫循序模型備援,英文是 Sequential Model Fallback,這個就是把備援機制玩得很徹底。它的做法是,你給它一份「有先後順序的模型清單」。系統會先拿清單裡的第一個、也就是你最想用的主要模型來試。如果這個主要模型因為任何狀況回不了話——可能是服務掛了、可能是被限流、也可能是觸發了內容過濾——系統就會自動把請求重新導到清單裡的下一個模型。這個過程會一直往下試,直到清單裡有某個模型成功把請求處理掉,或者整份清單都用完為止。而且很重要的是,這次操作最後算你多少錢、回應裡告訴你用了哪個模型,都會對應到那個「真正成功完成運算」的模型,而不是那些失敗的。這就確保了,就算你最想要的模型暫時不可用,服務還是能撐下去,這正是優雅降級的精神。OpenRouter 還提供一份很詳盡的排行榜,它是依據各個 AI 模型累計的 token 產量來排名的,token 你可以理解成模型處理文字的計價單位;同時它也把來自不同供應商的最新模型都收進來,像是 ChatGPT、Gemini、Claude 這些都有。

走完三組實作,我想帶你把視野再拉高一點。動態切換模型,其實只是資源感知最佳化的其中一招而已。這個模式對於開發那種「要在真實世界的種種限制下,又有效率又有效果地運作」的代理系統,是非常關鍵的。底下我幫你介紹一整個光譜的進階技巧,你會發現它們的共通精神,都是「在限制中聰明地花用資源」。

第一個,就是我們一路在講的動態模型切換,英文是 Dynamic Model Switching。它的核心,是根據任務的複雜程度跟手上可用的運算資源,策略性地去挑語言模型:簡單查詢就派輕量、便宜的;複雜、多面向的問題,才動用更精密、更耗資源的。

第二個叫自適應工具使用與選擇,英文是 Adaptive Tool Use and Selection。這是說,代理面對一整套工具的時候,要能明智地為每個子任務挑出最適當、最有效率的那一個,而且挑的時候要把 API 的使用成本、延遲、執行時間這些因素都仔細考慮進去。透過最佳化外部 API 跟服務的使用,整個系統的效率就上來了。

第三個叫情境修剪與摘要,英文是 Contextual Pruning and Summarization。這招管的是「代理要處理的資訊量」。它的做法是,聰明地把過去的互動歷史摘要起來,只選擇性地保留最相關的那部分,藉此把提示裡的 token 數量壓到最低。token 越少,推論成本越低,也就避免了不必要的運算開銷。你可以想成,跟人聊天時你不會把過去三小時每一句話都重述一遍,你只記重點,模型也該這樣。

第四個叫主動式資源預測,英文是 Proactive Resource Prediction。這是說,代理會去預測未來的工作負載跟系統需求,提早預判資源會不會不夠,然後主動地配置跟管理資源,確保系統反應得過來、不會卡住形成瓶頸。就像餐廳在週末來客高峰前先多排班、多備料一樣。

第五個叫成本敏感的探索,英文是 Cost-Sensitive Exploration。這個專門用在多代理系統裡。它的重點是,最佳化的時候,不能只看傳統的「運算成本」,還要把「通訊成本」一起算進去——也就是代理彼此之間溝通、傳資料也是要花資源的。這會影響到代理們該用什麼策略去協作、去分享資訊,目標是把整體的資源支出壓到最低。

第六個叫節能部署,英文是 Energy-Efficient Deployment。這是專門為那些資源限制非常嚴苛的環境設計的,目標就是把整個代理系統的能源足跡降到最低,好延長運作時間、也降低整體的運行成本。前面講的邊緣裝置就是典型場景。

第七個叫平行化與分散式運算意識,英文是 Parallelization and Distributed Computing Awareness。這是善用分散式的資源,把運算工作負載分散到很多台機器、或很多個處理器上一起跑,藉此提升代理的處理能力跟吞吐量,讓任務完成得更快、效率更高。就像一個大工程,與其一個人慢慢做,不如拆給很多人同時做。

第八個叫習得的資源配置策略,英文是 Learned Resource Allocation Policies。這個引入了一種學習機制,讓代理可以根據回饋跟效能指標,隨著時間慢慢去調適、最佳化它分配資源的策略,透過持續精煉來變得越來越有效率。重點在「習得」這兩個字,它不是寫死的規則,而是會越用越聰明。

第九個,就是我們很熟的優雅降級與備援機制,英文是 Graceful Degradation and Fallback Mechanisms。它確保的是,就算資源限制變得很嚴峻,系統還是能繼續運作,也許是在能力縮減的狀態下,優雅地把效能降下來、退回到替代策略,以維持運作、提供必要的功能,而不是直接罷工。

把這些講完,我們來做整章的收斂,我會用三個角度幫你把脈絡再串一次:這個模式是什麼、為什麼要用它、還有什麼時候該用它。

先說「是什麼」。資源感知最佳化,要應對的就是「在智慧系統裡,管理運算、時間、財務這三種資源的消耗」這個挑戰。為什麼這是個挑戰?因為以 LLM 為基礎的應用程式,常常又貴又慢,而你如果不分青紅皂白,每個任務都硬要挑那個最強的模型或工具,效率反而很差。這裡其實藏著一個根本性的取捨:一邊是「輸出的品質」,一邊是「產生這個輸出要花的資源」,兩者很難兼得。如果你沒有一套動態的管理策略,系統就沒辦法因應變動的任務複雜度,也沒辦法把自己控制在預算跟效能的限制之內。

再說「為什麼」,也就是標準的解法長什麼樣。標準解法,是去建構一個能根據任務聰明地監控、配置資源的代理系統。這套系統通常會有一個路由器代理,負責先幫進來的請求做複雜度分類,然後把請求轉送到最適合的模型或工具:簡單查詢交給快又便宜的,複雜推理交給更強大的。再進一步,還可以加一個評論者代理,透過評估回應品質、提供回饋,讓路由的邏輯隨時間越來越準。這種動態的、多代理的做法,就能確保系統運作得有效率,並且在回應品質跟成本效益之間取得平衡。

最後是「經驗法則」,也就是什麼時候你該想到用這個模式。第一,當你對 API 呼叫或運算能力有很嚴格的財務預算的時候。第二,當你在做一個「快速回應至關重要、對延遲很敏感」的應用程式的時候。第三,當你要把代理部署在電池壽命有限的邊緣裝置這種資源受限的硬體上的時候。第四,當你想用程式化的方式,在回應品質跟營運成本之間取得平衡的時候。第五,當你在管理那種「不同任務有不同資源需求」的複雜、多步驟工作流程的時候。只要碰到這幾種情況,資源感知最佳化就值得你拿出來用。

我把整章最關鍵的幾條重點,再幫你濃縮一遍。第一,資源感知最佳化是不可或缺的:它讓智慧代理能動態地管理運算、時間跟財務資源,而關於用哪個模型、走哪條執行路徑的決策,都是依據即時的限制跟目標來下的。第二,多代理架構帶來可擴展性:像 Google 的 ADK 這種多代理框架,讓你能做模組化設計,把回答、路由、評論這些工作交給不同的代理各司其職。第三,動態的、由 LLM 驅動的路由是核心:路由器代理會根據查詢的複雜度跟預算,把查詢導向不同模型,簡單的用 Flash、複雜的用 Pro,藉此同時最佳化成本跟效能。第四,評論者代理很有價值:它為自我修正、效能監控、還有精煉路由邏輯提供回饋,讓整個系統的成效往上走。第五,系統靠回饋跟彈性來最佳化:有了評估能力跟模型整合的彈性,系統就能展現出自適應、能自我改善的行為。第六,別忘了還有一整個光譜的進階方法,包括自適應工具選擇、情境修剪與摘要、主動式資源預測、成本敏感探索、節能部署、平行化、習得的資源配置策略、優雅降級與備援,以及對關鍵任務的優先排序。

總結來說,資源感知最佳化之所以對智慧代理的開發這麼重要,是因為它讓代理能在真實世界的種種限制底下,還能高效運作。透過好好管理運算、時間跟財務資源,代理才能同時達到好的效能跟好的成本效益。動態模型切換、自適應工具使用、情境修剪這些技巧,是達成效率的基本功;而像習得的資源配置策略、優雅降級這些更進階的策略,則進一步提升了代理在各種狀況下的調適力跟韌性。說到底,把這些最佳化的原則,從一開始就整合進你的代理設計裡,正是你能不能蓋出一個可擴展、夠穩健、又能長久永續運作的 AI 系統的根本所在。這,就是這一章想留給你的核心觀念。
