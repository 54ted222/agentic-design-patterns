歡迎來到這一章,我們要談的主題叫做「提示鏈」,英文是 Prompt Chaining。如果你曾經試著叫一個 AI 模型一口氣幫你完成一件很複雜的工作,結果它做得零零落落、漏東漏西,那這一章就是為你準備的。我會先帶你弄懂提示鏈到底是什麼、為什麼需要它,接著一路講過它的七種實務用法,再用一個很具體的程式範例讓你在腦中想像它怎麼運作,最後我們會延伸到一個更進階的觀念,叫做情境工程。整段路線就是這樣,我們一步一步來。

先說一個前提,我們這一章一直會提到的「大型語言模型」,英文叫 Large Language Model,縮寫是 LLM,你可以把它想成像 ChatGPT 背後那種會讀文字、會生成文字的 AI 模型。後面我講到 LLM 的時候,指的就是這種模型。

那我們先從提示鏈這個概念的核心理念說起。提示鏈,有時候也被叫做「管線模式」,英文是 Pipeline pattern。它的精神其實只有四個字:分而治之。意思是,與其期待模型在單一一個、龐大又複雜的步驟裡,把一個困難的問題整個解決掉,我們反過來,把這個又大又棘手的問題,拆成一連串比較小、比較好掌控的子問題。每一個子問題,都用一個專門設計過的提示去個別處理。這裡有個關鍵的動作:某一個提示產生出來的輸出,會被刻意地拿去當作下一個提示的輸入。一棒接一棒,前一棒的成果交給後一棒繼續做。

你可以想像一條工廠的生產線。原料從第一站進去,第一站做完一道工序,把半成品送到第二站,第二站再加工,送到第三站,最後組裝成成品。沒有哪一站要負責從頭到尾,每一站只專心做好自己那一道。這就是提示鏈的樣子。

這樣做有什麼好處?最直接的好處,是它替我們跟模型的互動帶來了模組化跟清晰度。因為任務被拆開了,你要理解每一個獨立步驟、或是要除錯找出哪一步出了問題,都變得容易很多。整條流程因此更穩健,也更能解釋清楚到底發生了什麼。而且,每一步都可以被單獨設計、單獨最佳化,只專注在大問題的某一個面向上,這樣產出的東西自然更精準、更聚焦。

我剛剛特別強調「某一步的輸出,當作下一步的輸入」,這一點為什麼這麼重要?因為這種資訊的傳遞,建立起了一條所謂的「依賴鏈」,英文是 dependency chain。「鏈」這個名字,就是從這裡來的。前面操作得到的情境跟結果,會引導後面的處理。換句話說,模型可以站在前一步成果的肩膀上繼續往前走,一步步把自己的理解磨得更精細,慢慢逼近我們想要的那個答案。

提示鏈還不只是把問題拆開而已。它同時也讓「整合外部知識跟工具」變成可能。在鏈的每一步裡,我們都可以指示模型去跟外部的系統、外部的應用程式介面、也就是 API,或是資料庫互動。這樣一來,模型的知識跟能力,就不再被困在它當初訓練資料的範圍裡了。這個能力非常重要,因為它讓 LLM 從一個孤立的、只會憑記憶說話的模型,變成一個更大、更聰明的系統裡頭不可或缺的零件。

也正因為這樣,提示鏈的意義遠遠超過單純的「把問題解掉」。它其實是建構精密 AI 代理的基礎技術。這裡講的「AI 代理」,英文是 AI agent,你可以理解成一種能自己規劃、自己推理、自己採取行動的 AI 系統,而不是被動等你一句一句下指令。這些代理就是靠提示鏈,在一個會變動的環境裡,自主地規劃、推理跟行動。透過把提示一個接一個有策略地排好順序,代理就能去處理那些需要多步推理、需要規劃、需要做決策的任務。這種工作流程,會更貼近人類真正在思考時的歷程,所以面對複雜的領域,它的互動就更自然、更有效。

講完了好處,我們得反過來問一個問題:為什麼不乾脆寫一個超級厲害、超級詳細的單一提示就好?這就要談到「單一提示的侷限」。對於那種有很多面向的任務,如果你硬要用一個又單一、又複雜的提示去驅動模型,效率其實往往很差。模型會很難同時顧好那麼多限制跟指令,結果就容易出現幾種典型的毛病,我幫你一個一個點出來。

第一種叫「指令忽略」,英文是 instruction neglect,意思是你提示裡寫的某些要求,模型根本沒理會、直接跳過了。第二種叫「情境漂移」,英文是 contextual drift,意思是模型講著講著,就慢慢偏離了你原本設定的情境,愈走愈歪。第三種叫「錯誤傳播」,英文是 error propagation,意思是前面一個小錯誤,後面被不斷放大,滾成大錯。第四種跟「情境視窗」有關,英文是 context window,你可以把它理解成模型一次能讀進去、能記住的資訊容量是有限的;當提示需要塞進去的東西太多,模型拿到的資訊反而不夠完整,沒辦法好好回應。第五種叫「幻覺」,英文是 hallucination,意思是當模型的認知負擔變重,它就更容易一本正經地編出錯誤的資訊。

我舉一個很具體的例子讓你體會這件事。假設你給模型一個查詢,要它「分析一份市場研究報告、摘要裡面的發現、找出有數據支撐的趨勢,還要再幫你草擬一封電子郵件」。這四件事塞在一個提示裡,結果很可能就是失敗。模型也許摘要做得還不錯,可是它沒辦法好好地把數據擷取出來,郵件也草擬得七零八落。一次要它做太多,它就什麼都做不好。

那提示鏈怎麼解決這個問題?答案就是把這個複雜任務,拆成聚焦的、循序的工作流程,藉此大幅提升可靠性跟可控性。我們就拿剛剛那個市場研究報告的例子,把它改寫成一條鏈,你聽聽看差別在哪。

第一個提示,專門做摘要。我們只跟模型說:請摘要以下這份市場研究報告的關鍵發現,然後把報告文字給它。這一步,模型唯一要操心的就是摘要這一件事,所以這個起點的準確度就被拉高了。第二個提示,專門做趨勢辨識。我們說:根據上一步得到的這份摘要,找出前三大新興趨勢,而且要把支撐每一項趨勢的具體數據也擷取出來。注意,這個提示現在被限制得更明確了,而且它是直接建立在一個已經驗證過的輸出之上,也就是第一步的摘要。第三個提示,專門寫郵件。我們說:請寫一封簡潔的電子郵件給行銷團隊,把上一步整理出來的那些趨勢跟它們的支撐數據概述一下。

你發現了嗎?同樣三件事,拆開來做,每一步都更簡單、更不模稜兩可,模型的認知負擔降低了,最後的成果反而更準、更可靠。這種模組化,就很像一條運算管線,每個函式各自做一個特定的操作,再把結果傳給下一個。

這裡還有一個很實用的小技巧。為了讓每一步都得到準確的回應,我們可以在每個階段給模型指定一個明確的角色。比方說剛剛那個流程,第一個提示我們可以叫它扮演「市場分析師」,第二個提示叫它扮演「交易分析師」,第三個提示叫它扮演「專家文件撰寫者」。給它一個身分,它的輸出就會更貼合那個任務的口吻跟專業度。

接下來要講一個很容易被忽略、但其實是成敗關鍵的東西,叫做「結構化輸出的角色」。提示鏈的可靠性,高度依賴各步驟之間傳遞的資料有多完整、多乾淨。你想想看,如果某一步的輸出含糊不清、或是格式亂七八糟,那下一步拿到這個爛輸入,當然就容易跟著失敗。為了避免這種骨牌效應,我們很建議讓模型用一種「結構化的輸出格式」來回話,常見的像是 JSON 或是 XML。JSON 你可以理解成一種用「鍵」跟「值」一組一組對應起來、機器很好讀的資料寫法。

舉個例子,剛剛那個趨勢辨識的步驟,它的輸出就可以格式化成一個 JSON 物件。我用講的描述給你聽:這個物件裡頭有一個叫「趨勢清單」的欄位,清單裡每一筆都包含兩個東西,一個是趨勢的名稱,一個是支撐這個趨勢的數據。比方說第一筆,趨勢名稱是「AI 驅動的個人化」,它的支撐數據是:有百分之七十三的消費者,偏好跟那些會運用個人資訊、讓購物體驗更貼近自己需求的品牌往來。第二筆,趨勢名稱是「永續且符合道德的品牌」,支撐數據是:過去五年裡,帶有 ESG 相關訴求的產品,銷售成長了百分之二十八,而沒有這類訴求的產品只成長了百分之二十。這裡的 ESG,指的是環境、社會跟公司治理這三個面向的責任訴求。

用這種結構化的格式,好處是資料變成機器可讀的,可以被精準地解析、然後乾淨地塞進下一個提示裡,不會產生模稜兩可的歧義。這個做法,把「因為要去詮釋自然語言而可能產生的錯誤」降到了最低,是打造穩健的多步驟 LLM 系統的一個關鍵零件。

好,觀念的部分我們講到這裡。接下來進入第二大段,我要帶你看提示鏈在真實世界裡到底用在哪些地方。我會講七種使用案例,你不用死記,重點是去感受它們共同的那個味道:把複雜問題拆成循序、可掌控的步驟。

第一種,是資訊處理的工作流程。很多任務,本質上就是把原始資訊經過好幾次轉換來處理。比方說,先摘要一份文件、再擷取裡面的關鍵實體、然後拿這些實體去查資料庫、最後生成報告。一條鏈可能長這樣:第一步,從指定的網址或文件裡把文字內容抓出來;第二步,把清理乾淨的文字做摘要;第三步,從摘要或原文裡擷取特定的實體,像是人名、日期、地點;第四步,拿這些實體去搜尋公司內部的知識庫;第五步,把摘要、實體跟搜尋結果整合起來,生成最終報告。這套做法,常被用在自動化的內容分析、AI 研究助理,還有複雜報告的生成上。

第二種,是複雜問題的回答。有些問題,沒辦法靠單一一個資料點回答,它需要好幾步推理、或是好幾次資訊檢索。我舉一個歷史問題給你聽:「一九二九年股市崩盤的主要成因是什麼?政府的政策又是怎麼回應的?」這條鏈可以這樣設計:第一步,先辨識使用者這個查詢裡的核心子問題,也就是拆出「崩盤的成因」跟「政府的回應」這兩塊;第二步,專門去研究或檢索一九二九年崩盤的成因;第三步,專門去研究政府對這次崩盤的政策回應;第四步,把第二步跟第三步的資訊綜合成一個連貫的答案,回應一開始那個問題。

這裡我想多講一個很重要的觀念,關於「平行」跟「循序」的搭配。想像一個自動化的研究代理,要針對某個主題生成一份完整報告。一開始,它會去檢索一大堆相關文章。從每一篇文章裡擷取關鍵資訊這件事,其實彼此是獨立的,你處理第一篇不需要等第三篇,所以這個階段非常適合「平行處理」,也就是讓這些互不相干的子任務同時進行,把效率拉到最高。可是,一旦每一篇的擷取都做完了,流程的性質就變了,它轉成了本質上必須「循序」進行。為什麼?因為接下來系統得先把擷取到的資料彙整起來,再綜整成一份連貫的初稿,最後還要審閱、精煉這份初稿,才生出最終報告。這後面幾步,邏輯上一步緊扣一步,後面那步一定要等前面那步成功完成才能做。而這,正是提示鏈派上用場的地方:彙整好的資料當作綜整提示的輸入,綜整出來的文字又當作最終審閱提示的輸入。所以你會看到,很多複雜的操作,其實是把「用來獨立蒐集資料的平行處理」跟「用來綜整、精煉這些相依步驟的提示鏈」結合在一起用的。

第三種,是資料的擷取與轉換。把一段沒有結構的文字,轉成有結構的格式,常常需要一個「迭代」的過程,也就是反覆地循序修正,一次比一次更準、更完整。我舉一個處理發票的流程:第一步,先嘗試從一份發票文件裡擷取特定欄位,像是姓名、地址、金額;接著做一個檢查,看看是不是所有必填欄位都抓到了、格式對不對;如果有欄位漏掉或格式錯了,就進入一個「條件式」的第二步,專門再擬一個新提示,要模型針對那些缺漏或出錯的資訊去補抓,而且可以把上一次失敗的情況也告訴它當參考;然後再驗證一次,不行就再重複;直到通過,才輸出已經擷取好、驗證好的結構化資料。

這種做法特別適合從表單、發票、電子郵件這種非結構化來源去擷取跟分析資料。我再給你一個更具體的場景,光學字元辨識,英文叫 OCR,簡單說就是讓電腦從圖片裡認出文字。假設要處理一份 PDF 表單,用拆解式的多步驟方法會有效得多。一開始,先用 LLM 對文件影像做主要的文字擷取;接著,模型處理這個原始輸出、把資料正規化,比方說它可能把「一千零五十」這幾個中文字,轉成等值的數字一千零五十。可是這裡有個重點:對 LLM 來說,要它精確地算數學,其實是它的大弱項。所以聰明的做法是,後面的步驟把任何需要算的算術,委派給外部的計算機工具去做。模型負責「辨識出這裡需要算什麼」,把正規化後的數字餵給計算機,再把計算機算出來的精確結果整合回來。這一連串「擷取文字、正規化資料、使用外部工具」的鏈式流程,能達成單靠一次 LLM 查詢往往做不穩的最終準確結果。你看,這又呼應了我們前面說的,提示鏈讓模型可以借用外部工具的力量。

第四種,是內容生成的工作流程。寫一篇複雜的內容,其實是個有程序的任務,通常會被拆成好幾個不同階段:先發想、再列大綱、再寫初稿、最後修訂。這條鏈可以是:第一步,根據使用者的整體興趣,生成五個主題構想;接著讓使用者挑一個,或是讓系統自動選最好的;第二步,根據選定的主題,生成一份詳細的大綱;第三步,根據大綱的第一個重點,寫一段初稿;第四步,根據大綱的第二個重點寫初稿,而且把前一段一起給它當情境,確保語氣連貫,大綱裡每個重點都這樣依序做下去;第五步,審閱並精煉整份初稿的連貫性、語氣跟文法。這套方法被廣泛用在創意敘事、技術文件,還有各種結構化文字的自動撰寫上。

第五種,是具狀態的對話式代理。這裡的「具狀態」,英文是 stateful,意思是它記得之前聊過什麼。雖然真正完整的狀態管理架構,會比單純的循序串接更複雜,但提示鏈仍然提供了維持對話連續性的基礎機制。它的做法是:把每一輪對話都建構成一個新的提示,而這個新提示裡,會有系統地把之前互動得到的資訊、或擷取到的實體納進來,藉此維持情境。流程是:第一步,處理使用者講的第一句話,辨識出他的意圖跟關鍵實體;接著用這個意圖跟實體去更新「對話狀態」;第二步,根據當前的狀態,生成回應,同時辨識出下一個還需要問的資訊;後面每一輪都這樣重複,使用者每講一句新的話,就啟動一條新的鏈,並且運用那個一直在累積的對話歷史。這個原則,正是開發對話式代理的根本,讓它們能在綿延、多輪的對話裡,維持住情境跟連貫性。因為保留了對話歷史,系統才能理解那些依賴於先前交談內容的使用者輸入,並做出適當回應。

第六種,是程式碼的生成與精煉。生成能跑的程式碼,通常也是個多階段的過程,要把一個問題拆成一連串各自獨立的邏輯操作,逐步完成。這條鏈可以是:第一步,先理解使用者對某個程式函式的需求,生成虛擬碼或大綱;第二步,根據大綱寫出第一版程式碼;第三步,找出程式碼裡潛在的錯誤或可以改進的地方,這一步甚至可以動用靜態分析工具,或是再呼叫一次 LLM 來幫忙挑毛病;第四步,根據找出的問題,重寫或精煉程式碼;第五步,補上文件說明跟測試案例。在 AI 輔助軟體開發這類應用裡,提示鏈的價值就在於它能把複雜的編碼任務拆成一個個好掌控的子問題,降低模型在每一步的複雜度。這裡還有一個關鍵點值得你記住:這種做法允許我們在「模型的兩次呼叫之間」插入確定性的邏輯。「確定性」,英文是 deterministic,意思是這部分不是交給模型去猜,而是用固定、可預期的程式邏輯去跑。這樣我們就能在工作流程中間做資料處理、做輸出驗證、做條件分支。換句話說,一個原本可能給出不可靠結果的單一龐雜請求,就被轉換成一系列由底層執行框架管理的結構化操作。

第七種,是多模態與多步推理。「模態」,英文是 modality,你可以理解成資訊的不同型態,比方說文字是一種模態,影像是另一種模態。分析那種同時混有好幾種模態的資料集,就需要把問題拆成更小的、以提示為基礎的任務。書裡舉的例子是,要解讀一張圖片,而這張圖片裡頭有一張帶著嵌入文字的圖、有標示特定文字片段的標籤、還有一個解釋各個標籤的表格。這時候鏈可以這樣設計:第一步,從使用者的影像請求裡,把文字擷取出來並理解;第二步,把擷取出來的影像文字,跟它對應的標籤連結起來;第三步,運用那個表格去詮釋蒐集到的資訊,判定出最後需要的輸出。

七種使用案例講完了,你應該已經抓到那個共通的精神了。接下來我們進到第三大段,動手實作的範例,讓你在腦中真正想像一條鏈跑起來是什麼樣子。

實作提示鏈的方式,範圍其實很廣。小,可以小到你在一個腳本裡頭,直接一個函式接一個函式循序地呼叫;大,可以大到使用專門設計來管理控制流程、狀態跟元件整合的框架。常見的框架有 LangChain、LangGraph、Crew AI,還有 Google 的 Agent Development Kit,縮寫是 ADK,中文就是代理開發套件。這些框架提供了結構化的環境,來建構跟執行這些多步驟流程,對於複雜的架構特別有幫助。

書裡選了 LangChain 跟 LangGraph 來示範,理由是它們的核心介面,本來就是針對「把操作組合成鏈跟圖」而設計的。這裡簡單幫你分一下:LangChain 提供的是「線性序列」的基礎抽象,也就是一棒接一棒的直線型流程;而 LangGraph 則更進一步,支援具狀態的、還能有循環的運算,也就是流程可以繞回去、可以重複,這些是實作更精密代理行為所必需的。我們這個範例,聚焦在最基本的線性序列。

接下來我用講的,把這段程式碼在做什麼描述給你聽,你不用去想任何括號跟符號,只要在腦中想像這條兩步驟的管線就好。

這條鏈總共兩個階段。第一個階段,負責解析一段沒有結構的文字,把裡面特定的資訊擷取出來。第二個階段,接手第一階段擷取到的結果,把它轉換成有結構的資料格式。

要跑這個程式,得先安裝幾個函式庫,主要就是 LangChain 相關的套件。書裡也提醒,如果你用的不是 OpenAI、而是別家的模型供應商,只要把對應的套件換掉就好。然後,執行環境裡要先設定好你選的那家語言模型供應商所需要的 API 憑證,也就是那把讓你有權呼叫模型的金鑰,常見的供應商有 OpenAI、Google 的 Gemini,或是 Anthropic。

程式一開始,先初始化一個語言模型。這裡有一個小細節:它把模型的一個叫「溫度」的參數設成零。溫度你可以理解成模型回答的「隨機程度」,設成零,就是要它盡量穩定、每次給出一致的答案,這在我們要做精確擷取的時候很重要。

接著定義第一個提示。這個提示的意思,翻成中文就是:「從以下這段文字裡,把技術規格擷取出來。」後面接的就是要餵進去的那段文字。然後定義第二個提示,它的意思是:「請把以下這些規格,轉換成一個 JSON 物件,並且用 cpu、memory、storage,也就是處理器、記憶體、儲存空間這三個當作鍵。」後面接的,就是第一步擷取出來的規格內容。

再來是把這兩步串起來。程式用了一個叫 LCEL 的東西,全名是 LangChain 表達式語言,英文是 LangChain Expression Language,它讓你可以很優雅地把提示跟語言模型一個接一個串接起來。它先組出第一條鏈,叫擷取鏈,這條鏈的工作就是:把文字餵給模型,讓模型擷取規格,再透過一個叫 StrOutputParser 的東西,把模型回傳的訊息轉成乾淨可用的純字串。這個 StrOutputParser,你就理解成一個「把模型輸出整理成單純文字」的小工具。然後它組出完整的鏈,這條完整的鏈會做一件很關鍵的事:它把擷取鏈的輸出,自動接到第二個轉換提示的那個「規格」變數上頭。這正是我們整章一直在講的核心動作,前一步的輸出,變成後一步的輸入。

最後是執行。程式給了一段輸入文字,內容大概是:「這款新筆電,搭載一顆三點五 GHz 的八核心處理器、十六 GB 的記憶體,還有一顆一 TB 的 NVMe 固態硬碟。」程式用這段文字去呼叫完整的鏈,讓它先後跑過這兩個步驟。第一步,模型從這段話裡把技術規格擷取出來;第二步,模型把這些規格轉成一個 JSON 字串。最後,程式把這個整理好、格式化好的 JSON 結果印出來。

你聽到這裡應該能想像那個畫面了:一段人類隨手寫的、自然語言的筆電描述,進到管線的這頭;經過擷取、再經過格式化兩道工序;從那頭出來的,是一筆乾淨的、機器能讀的結構化資料。這就是一條最基本的提示鏈,實際跑起來的樣子。

好,核心模式跟實作我們都講完了。接下來進到最後一大段,我要帶你延伸到一個更高層次的觀念,叫做「情境工程」,英文是 Context Engineering。這個觀念,會讓你重新理解「提示」這件事的格局。

情境工程是一門系統化的學科,它的目標是:在模型開始生成任何字之前,先為它設計、建構並交付一個完整的資訊環境。它背後有一個核心主張,我覺得很值得你記住:模型輸出的品質,比較少取決於模型架構本身,而更多取決於你提供給它的情境有多豐富。換句話說,與其一直追求換更強的模型,不如把心力花在「餵給它的東西夠不夠完整」這件事上。

我們可以拿它跟傳統的「提示工程」做對比。提示工程,英文是 Prompt Engineering,它主要著重在「把使用者當下這一句查詢的措辭調得更好」。而情境工程,是一次重大的演進,它把範圍擴大,涵蓋了好幾個不同層次的資訊。我幫你列出來。第一層,是系統提示,英文是 system prompt,這是一組定義 AI 運作參數的基礎指令,比方說「你是一位技術文件撰寫者,你的語氣必須正式而且精確」。第二層,是用外部資料去豐富情境,這裡頭又包含兩種:一種是「檢索到的文件」,也就是 AI 主動從知識庫裡撈出資訊來輔助回應,比方說為了某個專案調出它的技術規格;另一種是「工具輸出」,也就是 AI 用外部 API 取得即時資料的結果,比方說查一下行事曆,看看使用者有沒有空。第三層,是把這些看得見的顯性資料,跟一些關鍵的隱性資料結合,隱性資料像是使用者的身分、過去的互動歷史,還有當前的環境狀態。這裡的核心原則是:就算是再先進的模型,如果你只給它一個對運作環境很有限、或很拙劣的視野,它一樣會表現得很差。

所以情境工程做的事,是把任務從「單純回答一個問題」,重新定位成「為代理建構一幅完整的運作全貌」。我舉一個很生活化的例子。一個經過情境工程處理的代理,它不會只是死板地回應你的查詢。假設你要它幫忙安排一場會議,它會先去整合好幾樣東西:它會看你的行事曆有哪些空檔,這是工具輸出;它會考慮你跟某封郵件收件人之間是什麼樣的專業關係,這是隱性資料;它還會翻出你們先前那場會議的筆記,這是檢索到的文件。把這些通通整合起來之後,它才產生回應。這樣出來的輸出,就高度相關、個人化,而且在實務上真的有用。那「工程」這兩個字到底工程在哪?它指的是,你要建立一套穩健的管線,在執行的當下去擷取並轉換這些資料,而且還要建立回饋迴路,持續改善情境的品質。

那回饋迴路、改善品質,要怎麼大規模地自動化做到?書裡提到可以運用專門的調校系統。它舉的例子是 Google 的 Vertex AI prompt optimizer,中文是 Vertex AI 提示最佳化工具。這個工具能根據一組範例輸入、加上你預先定義好的評估指標,有系統地去評估模型的回應好不好,藉此提升模型的表現。它特別適合一個情境:當你要把同一套提示跟系統指令,從一個模型搬到另一個模型上沿用,卻又不想花大量人力一句一句重寫的時候。你只要提供它範例提示、系統指令跟一個範本,它就能用程式化的方式去精煉這些情境輸入,等於替你那個情境工程所需要的回饋迴路,提供了一套有結構的方法。

說到底,這種結構化的做法,正是區分「簡陋 AI 工具」跟「精密、具情境感知系統」的關鍵分水嶺。它把情境本身當成最重要的元件來看待,非常在意三件事:代理「知道什麼」、它「在什麼時候知道」、以及它「怎麼運用這些資訊」。這麼做,確保了模型對使用者的意圖、歷史跟當前環境,有一個全面的理解。歸根究柢一句話:情境工程,是把一個沒有記憶、沒有狀態的聊天機器人,推進成一個高能力、能感知情境的系統的關鍵方法。這裡的「無狀態」,英文是 stateless,意思就是它每次對話都像第一次見面,什麼都不記得。

我們快講完了,讓我把整章的脈絡再幫你串一次,當作收尾。

先回答「是什麼」這個問題。當你把一個複雜任務硬塞進單一提示去處理,常常會把模型壓垮,引發明顯的效能問題。模型認知負擔一重,就容易忽略指令、遺失情境、編出錯誤資訊。一個龐大的單體式提示,很難有效管理那麼多限制跟一步步的推理,結果就是不可靠、不準確的輸出。

再回答「為什麼」要用提示鏈。提示鏈提供了一套標準化的解法,把複雜問題拆成一連串較小、彼此相連的子任務。鏈裡每一步,都用一個聚焦的提示去執行一個特定操作,大幅提升可靠性跟可控性。前一個提示的輸出,當作後一個提示的輸入,串成一條邏輯工作流程,一步步逼近最終解答。這種模組化、分而治之的策略,讓整個流程更好管理、更好除錯,還允許我們在步驟之間整合外部工具、或塞進結構化的資料格式。它就是開發那種能規劃、能推理、能執行複雜工作流程的精密多步驟代理系統的基礎。

最後給你一條「經驗法則」,幫你判斷什麼時候該掏出這個模式。當一個任務複雜到單一提示扛不住、當它牽涉到好幾個不同的處理階段、當它需要在步驟之間跟外部工具互動,或是當你正在建構一個需要多步推理又要維持狀態的代理系統時,就用提示鏈。

我幫你把最核心的幾句話再濃縮一遍。提示鏈,把複雜任務拆成一連串較小、聚焦的步驟,有時候也叫管線模式。鏈裡的每一步,都是一次 LLM 呼叫、或是一段處理邏輯,而且都以前一步的輸出當輸入。這個模式,提升了我們跟語言模型進行複雜互動時的可靠性跟可管理性。而像 LangChain、LangGraph,還有 Google ADK 這些框架,提供了穩健的工具,讓你去定義、管理並執行這些多步驟序列。

總結來說,提示鏈做的事,就是把一個複雜問題,解構成一連串比較簡單、比較好掌控的子任務,替我們引導大型語言模型提供了一套穩健的框架。這種分而治之的策略,因為讓模型一次只專注在一個特定操作上,所以大幅提升了輸出的可靠性跟可控性。作為一種基礎模式,它讓我們有可能開發出具備多步推理、工具整合、還有狀態管理能力的精密 AI 代理。歸根究柢,真正精通提示鏈,就是建構那種能執行複雜工作流程、穩健又能感知情境的系統的關鍵,而它的能耐,遠遠超過任何單一提示所能企及的範圍。這一章就到這裡。
